## Summary

T-0.9 realizes the one-shot (`-p`) CLI: parse the invocation, resolve config fail-fast (preserving exit 2 before any model call, ADR-0009), build the agent loop, run the prompt, and map the terminal `LoopOutcome` — or any failure thrown along the way — to an agent-process `ExitCode` honouring the exit-code contract's precedence (`cli-exit-codes.md` § 2: `130` > `2`-startup > `4` > `5` > `3` > `1` > `0`). The work splits into a **testable run-and-map seam** (`OneShotRunner`, 100% line) and a **production-only composition root** (`AgentLoopFactory`, JaCoCo-excluded like `Main`). New classes: `OneShotRunner` (run + exit-code mapping, with an injectable `OneShotLoop` seam so it is driven by a real `AgentLoop` over a scripted Bedrock double — no live AWS), `CliArguments` (narrow `-p`/`--prompt`/`--help`/`--version` parser; unknown flag / missing value / positional → `UsageException` → exit 2), `NonInteractiveApprover` (the one-shot approval seam: a required prompt is a blocking denial → `UserAbortedException` → exit 3, AC-10.2), `AgentLoopFactory` (wires credentials → Bedrock client → tools/gate/log → `AgentLoop`), and three exit-code carrier exceptions (`UsageException` → 2, `UserAbortedException` → 3, `InterruptedRunException` → 130). `Main.run` is extended to dispatch by parsed shape: usage error → exit 2 naming the argument; `--help`/`--version` → print + exit 0; config error → exit 2 naming the key (CT-EX-1 preserved); one-shot → build the loop and delegate to `OneShotRunner`; interactive (no `-p`) → exit 0 (REPL is T-1.1). The exit-code mapping: `completed()` (`end_turn`) → 0 (CT-EX-6); `CredentialResolutionException` / `ModelBackendException` → 4 (CT-EX-2 preserved, § 3.2); surfaced `model_context_window_exceeded` (no compaction at M0) → 5; surfaced guardrail/content_filtered/malformed → 1 (catch-all, per the dep-API mapping note); `UserAbortedException` → 3 (CT-EX-3); `InterruptedRunException` → 130 (CT-EX-4, caught first so SIGINT always wins per § 2); `PersistenceException` → 1 (AC-13.4); any other `RuntimeException` → 1 (§ 3.2 catch-all). Every non-zero exit prints one stderr line naming the cause (G2); the final answer goes to stdout via an injected stream (the library/CLI output split, NFR-LOG / 04-apis § 1.6 — library modules write to injected streams, only the `Main` bootstrap touches `System.out`/`System.err`). `mvn clean verify` is green: **421 unit tests** (41 new across 5 CLI test classes), 0 failures/errors/skipped; `OneShotRunner` 100% line, the three exceptions + `NonInteractiveApprover` + `ExitCode` 100%, `CliArguments` 96%; JaCoCo "All coverage checks have been met" (0.80 floor). One Minor (the production composition root `AgentLoopFactory` + `Main.runOneShot` are untested-by-design — they construct a live Bedrock client; their run-and-map logic and credential-failure→exit-4 mapping ARE tested at the `OneShotRunner` layer; `AgentLoopFactory` is JaCoCo-excluded with the same documented rationale as the `Main` bootstrap). One Discussion (the SIGINT→130 mapping is a logic-only seam at M0 — `InterruptedRunException` — with no production code yet throwing it; this matches the task's explicit instruction to assert the mapping, not OS-signal delivery, and the actual signal handler / stream-cancellation is T-1.1). No Blockers, no Majors.

### Minors

**[m1] [Testing] `cli/AgentLoopFactory.create` and `cli/Main.runOneShot` are untested — production-only composition roots that construct a live Bedrock client**
- Why: the rubric's pinned tenet "tests are not optional" flags new logic without a test. `AgentLoopFactory.create` resolves SigV4 credentials and calls `BedrockClientFactory.create(...)`, which builds a real `BedrockRuntimeClient`; `Main.runOneShot` opens a session log and calls it. Both are pure wiring with no branching business logic, and exercising either in a unit test would require a live AWS call (forbidden by the task) or mocking the SUT's collaborators in a way that adds no spec-traceable assertion. The *logic* they delegate to — the run-and-map exit-code dispatch (`OneShotRunner`, 100% line) and the credential-failure → exit-4 mapping — IS tested via `OneShotRunnerTest.credentialResolutionFailureMapsToFour`. This is a Minor (not a Blocker) because it mirrors the project's existing, documented treatment of the `Main` bootstrap shell: `AgentLoopFactory` is added to the JaCoCo `<excludes>` with the same rationale, so the coverage gate honestly reflects "untestable composition root," and the bundle remains at 91.9% line (well above the 0.80 floor).
- Suggestion: keep the JaCoCo exclude; when the REPL (T-1.1) or an integration-test harness with a `@Tag("integration")` live-Bedrock smoke test lands, an end-to-end test can narrow the exclude. No code change needed at T-0.9.

### Discussion

**[D1] [Discussion] cli-exit-codes `130` + 02-architecture § 4: the SIGINT→130 mapping is a logic-only seam at M0 — `InterruptedRunException` is defined and mapped, but no production code yet throws it**
- Why: § 4 says Ctrl-C interrupts the current step (cancel the in-flight Converse stream, kill the in-flight subprocess, flush the event log) and exits 130. The task note explicitly scopes M0 to "a clean, testable SIGINT→130 mapping ... assert the mapping logic, not OS signal delivery," and defers the full stream/subprocess-cancellation richness to T-1.1. Accordingly, `InterruptedRunException` is the seam: `OneShotRunner.run` catches it first (so 130 wins precedence per § 2) and re-asserts the thread's interrupt status; `OneShotRunnerTest` drives the mapping + precedence + interrupt-status assertions deterministically without a real signal. No production path throws `InterruptedRunException` yet — the signal handler / `Thread.interrupt()`-driven path is T-1.1's job (where the in-flight Converse stream and subprocess become cancellable). Surfaced only so a future reader does not mistake the unthrown seam for dead code: it is the documented extension point the contract test (CT-EX-4) exercises now, and the signal source attaches later. The event log already flushes per event (T-0.4), so resumability (CT-EX-4's "session remains resumable") needs no special M0 teardown — exactly as the task note states.
- Suggested amendment kind: none (no spec change). § 4 and cli-exit-codes already pin the follow-on (T-1.1 owns the signal handler + stream/subprocess cancellation). The seam (`InterruptedRunException` + the precedence-first catch + interrupt-status re-assertion) is in place. No code change needed at T-0.9.

### Praise

**[P1]** The one-shot path is split along the exact seam that makes it testable without an exception to the "no live AWS" rule: all exit-code-mapping logic lives in `OneShotRunner` (driven by an injected `OneShotLoop` functional seam — a real `AgentLoop` over a scripted Bedrock double for the happy/surfaced paths, and a throwing lambda for each exception branch), so the system under test (the run orchestration + the contract's precedence ladder) is never mocked, only Bedrock is. The runner reuses rather than re-declares: it maps the **existing** typed exceptions (`CredentialResolutionException` T-0.3, `ModelBackendException` T-0.5, `PersistenceException` T-0.4) and the **existing** `LoopOutcome`/`StopReason` (T-0.8) to the **existing** `ExitCode` (T-0.2) — no new error taxonomy, no parallel stop-reason enum. The exit-code precedence is enforced *structurally* by catch order (130 caught first → always wins § 2; 4 before 3 before 1) rather than by a fragile priority comparison. `NonInteractiveApprover` realizes exit 3 cleanly: it plugs into the *existing* `Approver` seam (T-0.7) so a one-shot run's blocking denial flows through the gate's real authorization path (verified end-to-end in `blockingDenialExitsThree`, which proves the denied tool's handler never ran — INV-8), and it correctly fires only when the gate genuinely must prompt (Class-R reads / auto-approvals never reach it — verified by `readOnlyToolCompletesExitsZero`). The library/CLI output split is honoured: `OneShotRunner` writes to injected streams, only `Main` (the bootstrap, JaCoCo-excluded) touches `System.out`/`System.err`.
