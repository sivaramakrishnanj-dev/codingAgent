## Summary

T-4.2 delivers the C1→C4 multimodal-attachment pipeline (§ 2.3 multimodal input). New
`ContentBlock.Image` / `ContentBlock.Document` variants extend the sealed hierarchy (schema-conformant:
`bytesRef` carried by reference, format enums, INV-18 name pattern enforced in the `Document` compact
constructor). `ConverseWireMapper` gains the input-only `Image → image{format,source{bytes}}` /
`Document → document{name,format,source{bytes}}` wire mapping, reading the referenced bytes at send time
(the SDK base64-encodes). `ModelCapabilityProfile` gains `supportsImageInput`/`supportsDocumentInput`
(true for Claude, false for the conservative/unknown profile — § 2.6), with a backward-compatible
window-only constructor so existing callers/tests are unchanged. A new C1 `AttachmentResolver` (+ sealed
`Attachment` result) infers the format from the extension, sanitizes a document name (INV-18), and applies
the INV-19 capability gate (decline-with-message, not sent). `--attach <path>` parses in `CliArguments`
and `/attach <path>` dispatches in `ReplRunner`'s slash-command mechanism. `AgentLoop` gains a
`run(String, List<ContentBlock>)` overload that seeds the opening user turn with the prompt then the
admitted attachment blocks, so the wire mapper renders them into the first request.

### Nits

**[N1] [Readability] `ConverseWireMapper.readBytes` uses `IllegalArgumentException` for an I/O failure.**
- Why: an unreadable `bytesRef` is arguably an environmental/runtime fault rather than a "bad argument";
  a more specific typed exception (or `ModelBackendException`) could read better at the boundary. Clean
  Code / EJ Item 72 (favour standard exceptions) — but `IllegalArgumentException` is defensible here
  because the offending input *is* the block's `bytesRef` argument, and it is chained (cause preserved).
- Suggestion: leave as-is for this task (it names the reference and chains the cause); revisit if a
  dedicated attachment-IO exception is later warranted.

**[N2] [Readability] `Main.resolveOneShotAttachment` casts `(Attachment.Declined) attachment` after an
`instanceof Attached` check.** A sealed-interface `switch` (pattern) would be marginally clearer than the
`if (instanceof Attached) … else cast`. The cast is provably safe (sealed: only Attached|Declined) and in
JaCoCo-excluded wiring, so this is taste only.

### Praise

**[P1] Schema-as-oracle discipline for the negative CTs.** CT-SCH-6/7/8 validate *literal JSON* against the
committed `content-block.schema.json` (independent of the code's own constructor, which also rejects the
same inputs), so the contract-test oracle is the spec artifact, not the implementation — exactly the T-0.4
pattern the task cited.

**[P2] INV-18 sanitization tested by property, not literal.** The sanitization tests assert the produced
name matches the schema's `DocumentBlock.name` pattern and drops the disallowed source characters, rather
than pinning a brittle impl-derived output string — keeping the oracle on INV-18, not on the resolver's
exact transformation.

## Notable scope decision (documented as a stated assumption)

The REPL `/attach <path>` command resolves + capability-gates + reports the attachment (attached /
declined) in the same `handleCommand` mechanism as `/exit`/`/mode`/`/permission`, but does **not** thread
the admitted block end-to-end into the live REPL turn — that delivery would require widening the
`BrownfieldRunner`/`GreenfieldRunner` (C3) turn seams, which are outside this task's C1+C4 component scope.
The full end-to-end attachment-to-request path IS wired and tested on the one-shot `--attach` path
(`Main.oneShotBrownfield` → `AgentLoop.run(String, List)` → `ConverseWireMapper`) and at the loop seam
(`AgentLoopTest.runWithAttachmentsSeedsTurn`). This is a deliberate, in-scope boundary, not a defect.
