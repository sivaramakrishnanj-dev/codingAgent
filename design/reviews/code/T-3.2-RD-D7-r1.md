## Summary

D7 is a live-only regression of T-3.2: the greenfield `write_artifact` tool, store, and
pre-approval registry were all built and wired, yet a real-Bedrock greenfield run called
`write_artifact` zero times — the phase artifacts held only the approval stamp, no deliverable
content (recorded in `design/07-tasks-progress.md:41-56`). The root cause is in one place: the
per-phase greenfield system prompt (`GreenfieldPlaybook`) discussed each deliverable but never
directed the model that the deliverable *is* the written artifact, authored via `write_artifact`
at its path before the turn completes. The fix is a pure prompt change — a new common block plus
per-phase artifact-path + `write_artifact` directives — with no change to the tool, store, gate,
or registry. The new contract test drives a real phase-loop turn over a scripted Bedrock that
emits a `write_artifact` tool-use and asserts the artifact on disk holds the model-authored
content, which is the behaviour a "phase returns text" test would have missed.

### Minors

**[m1] [Testing] `GreenfieldArtifactPersistenceTest.java:204-234` — content-persistence test does not assert `write_artifact` was actually the tool that wrote the file.**
- Why: The test asserts the artifact on disk equals the scripted content, which is strong, but the
  on-disk content could in principle be produced by any write path rooted at the same dir. The
  intent (per the D7 contract) is specifically that the *model's `write_artifact` tool-use* drove
  it. The risk is low here because the only write seam offered in the pre-approval registry is
  `write_artifact` and nothing else in the loop writes to `design/`, so the equality assertion is in
  practice exact. Worth a one-line strengthening, not a redesign.
- Suggestion: Optionally also assert the loop dispatched the tool — e.g. the scripted model is
  exhausted only if both turns ran (the `endTurn` follow-up consumes the second script entry), or
  read the `EventLog` for a `TOOL_RESULT` for `write_artifact`. The third test
  (`preApprovalPhasePromptDirectsArtifactWrite`) already pins the prompt side, so this is belt-and-
  suspenders, not a gap.

### Nits

**[N1] [Readability] `GreenfieldPlaybook.java:162-164` — `orElseThrow()` on `forPhase(phase)` in `artifactPath`.**
- Why: `artifactPath` is only ever called for the three pre-approval phases (the `switch` arms
  REQUIREMENTS/DESIGN/TASKS), so `GreenfieldArtifact.forPhase(phase)` is always present and
  `orElseThrow()` cannot fire. The bare `orElseThrow()` throws a contextless `NoSuchElementException`
  if a future edit ever called it for IMPLEMENT. The Javadoc already states the invariant.
- Suggestion: Either leave as-is (the invariant is documented and the call sites are local) or give
  it a message: `orElseThrow(() -> new IllegalStateException(phase + " authors no artifact"))`.
  Author's call — taste, not a defect.

### Praise

**[P1] [Design] `GreenfieldPlaybook.java:60-67` — `ARTIFACT_WRITE_TOOL` kept as its own constant, distinct from `SOURCE_CHANGE_TOOLS`.**
- Modelling the design-doc write tool as a separate named constant (rather than folding it into the
  tool-name strings) is what lets the prompt name `write_artifact` in the pre-approval phases while
  the AC-1.4 invariant — no pre-approval prompt names `write_file`/`edit_file`/`run_command` — stays
  mechanically checkable by `preApprovalPhasesDoNotNameSourceChangeTools`. Clean separation of the
  one allowed Class-X write from the withheld source writes.

**[P2] [Testing] `GreenfieldArtifactPersistenceTest.java` — drives the real loop, not a "returns text" stub.**
- The test replays a scripted `write_artifact` tool-use through a real `AgentLoop` → real
  `PermissionGate` (auto-approve) → real `WriteArtifactTool` → real `GreenfieldArtifactStore` rooted
  at a `@TempDir`, then reads the artifact back through the same store. This is exactly the live path
  the D7 smoke test exercised, and is the discipline that would have caught the original defect.

### Discussion

**[D1] [Design-Gap] The fix relies on prompt instruction; it cannot *force* the model to call `write_artifact`.**
- The D7 root cause was a missing prompt directive, and the fix supplies it. But a prompt is a steer,
  not a guarantee — a future model could still skip the write. The structural backstop already exists
  one layer down: the `ArtifactApprovalGate` refuses to advance the TASKS gate when the breakdown is
  untraceable (AC-2.5), and an empty artifact is untraceable, so a skipped write cannot silently reach
  implementation. This is in scope for T-3.2-RD-D7 (the task is the prompt fix). Flagging for the G3
  smoke-test: re-run the live greenfield arc and confirm `write_artifact` now fires with substantive
  content per phase — the prompt fix is the necessary change, the live run is the proof.

## Reuse / duplication

- `ScriptedBedrockClient` in `GreenfieldArtifactPersistenceTest` mirrors the established
  Bedrock-double pattern used by `AgentLoopTest` / `LiveToolRegistryCompositionTest` (a hand-rolled
  `BedrockRuntimeClient` replaying scripted `ConverseResponse`s, capturing requests) — same pattern,
  not a mock of the SUT. A small per-test double is the repo convention; not worth extracting to a
  shared fixture given the captured-requests field differs per test.
- The new test does **not** duplicate `GreenfieldArtifactCompositionTest`. That test pins the
  *composition* contract (the pre-approval registry contains `write_artifact` and withholds
  `write_file`; a direct `write_artifact` handle lands in `design/`; the approval gate timestamps +
  enforces traceability). The new test pins the *live end-to-end* contract (a model `write_artifact`
  tool-use flows loop → gate → tool → store so the artifact ends up with model-authored content, and
  the prompt the model receives directs that write). Distinct contracts; no merge warranted.
- `artifactPath` reuses `GreenfieldArtifact.forPhase(...).relativePath()` rather than re-deriving the
  path, so the prompt and the artifact enum stay in step — correct reuse of the existing seam.
