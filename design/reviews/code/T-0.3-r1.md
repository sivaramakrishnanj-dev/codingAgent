## Summary

T-0.3 delivers the Model Client (C4) credential seam: a `com.srk.codingagent.model.credentials` package that resolves Bedrock SigV4 credentials in ADR-0011's two-tier order (named profile → default chain), explicitly ignores any ambient `AWS_BEARER_TOKEN_BEDROCK` (the load-bearing AC-8.8/INV-16 safety property), and constructs the `BedrockRuntimeClient` with the resolved SigV4 provider for T-0.5 to consume. The package is `CredentialResolver` (probe-and-fall-back SigV4 resolution + bearer-ignore-and-warn), `BedrockCredentials` (the inspectable value object carrying the chosen SigV4 provider + resolved-tier label), `CredentialResolutionException` (typed no-credentials failure mapping to exit 4 / MODEL_BACKEND, distinct from the config fault's exit 2), and `BedrockClientFactory` (the SigV4 client-construction seam, with a pure inspectable `wiring(...)` step separated from the real SDK `build()`). `pom.xml` adds the AWS SDK v2 BOM + `bedrockruntime` (version pinned to 2.46.7 — the design-time candidate 2.46.10 no longer resolves on Maven Central). `mvn clean verify` is green; 114 unit tests pass (24 new); bundle line coverage is ~95.3% with the 0.80 JaCoCo gate met and no new exclusion. One Minor (the `RuntimeException` catch breadth, justified), one Nit, two Discussions (the SDK version drift from ADR-0001's noted 2.46.10, and the deferred exit-4 CLI-dispatch wiring). No Blockers, no Majors.

### Minors

**[m1] [Error Handling] `model/credentials/CredentialResolver.java:118,132` — `catch (RuntimeException)` is broad**
- Why: both tiers catch `RuntimeException` rather than the SDK's `SdkException`. This is deliberate: the AWS SDK surfaces no-credentials conditions as `SdkClientException`/`SdkException` (both `RuntimeException` subtypes), and a profile-not-found or env-misconfiguration can surface as related unchecked types; catching the common supertype keeps the fall-back/wrap behavior robust to the SDK's exception taxonomy. Neither catch swallows: `tryProfile` logs WARN with the cause and falls back (AC-8.7 action), `resolveFromDefaultChain` wraps into the typed `CredentialResolutionException` chaining the cause (AC-8.9 action). It is narrower than the `catch(Exception)`/`catch(Throwable)` java.md flags. Recorded as a Minor only because a `catch (SdkException)` would be marginally more precise about intent.
- Suggestion: optionally narrow to `software.amazon.awssdk.core.exception.SdkException` if no non-SDK unchecked failure is expected from the provider factories. Defensible as-is; no change required this task.

### Nits

**[N1] [Readability] `model/credentials/CredentialResolver.java` — the resolved-tier label `"default chain"` lives as a constant but the profile label is built inline**
- Why: `DEFAULT_CHAIN_LABEL` is a named constant, while the profile label (`"profile '" + name + "'"`) is constructed inline in two places (`tryProfile` and `noUsableCredentialsMessage`). Minor asymmetry; the inline form is short and clear. Author's choice.

### Discussion

**[D1] [Discussion] AWS SDK version pinned to 2.46.7, not ADR-0001's noted 2.46.10**
- Spec refs: ADR-0001 ("latest `2.46.10` per Maven Central, 2026-06-15 — pin/confirm exact version at implementation time ... if 2.46.10 no longer resolves, pick the latest stable and note it"). At implementation time (2026-06-18) `2.46.10` no longer resolves on Maven Central — `mvn dependency:get` for `bedrockruntime:2.46.10` fails, while `2.46.7` resolves and is the latest stable available (also corroborated by the Maven Central search API and the local `~/.m2` cache). The pin is therefore `2.46.7`, exactly as ADR-0001 directs for this case. Flagged informationally so a future reader is not surprised by the version delta from the ADR text. Suggested amendment kind: none — ADR-0001 explicitly anticipated this and instructed "pick the latest stable and note it"; this note is the record.

**[D2] [Discussion] The no-credentials → exit-4 *CLI dispatch* is deferred; T-0.3 wires the contract carrier, not `Main` dispatch**
- Spec refs: AC-8.9, CT-EX-2 ("no usable SigV4 credentials → exit 4, names paths attempted"). `CredentialResolutionException` carries the paths-attempted message and maps to `ExitCode.MODEL_BACKEND` (= 4), and CT-EX-2 is satisfied at the type boundary (`CredentialResolutionExceptionTest.ct_ex_2_mapsToModelBackendExitFour` + the resolver's `ct_ex_2_*` message tests). `Main.run()` is intentionally NOT changed to invoke credential resolution at launch: doing so would pull forward the launch-sequence decision (does the agent eagerly resolve credentials before any model call?), which no T-0.3 spec pins and which is owned by the Model Client / agent-loop tasks (T-0.5/T-0.8, explicitly OOS here) — and it would force the existing config-only `MainConfigTest` launches to perform real credential resolution. The exit-4 *dispatch* (catching `CredentialResolutionException` in the CLI and returning `ExitCode.MODEL_BACKEND.code()`) lands with the task that owns CLI/model-call dispatch. Suggested amendment kind: none — this is a scope boundary, not a spec gap (mirrors how T-0.2 asserted the full `ExitCode` mapping as the shared value type ahead of the dispatch that exercises each code).

### Praise

**[P1] [API-Design] The inspectable SigV4 seam (`BedrockCredentials.provider()` + `BedrockClientFactory.wiring(...)`)**
- CT-INV-13 explicitly asks to "design the seam so the chosen provider is inspectable/injected ... rather than depending on real SDK env-var pickup behavior." The design realizes this cleanly: `CredentialResolver` takes injected provider factories + an injected environment lookup (so SigV4 selection, profile fall-back, and the bearer-ignore warning are all exercised hermetically — no `~/.aws`, no real env var, no network), `BedrockCredentials` exposes the exact chosen provider, and `BedrockClientFactory` separates the pure `wiring(...)` decision (fully tested) from the un-unit-testable real `build()`. The bearer-ignore safety property is verified by asserting the wired provider IS the supplied SigV4 provider — exactly the inspection CT-INV-13 prescribes.
