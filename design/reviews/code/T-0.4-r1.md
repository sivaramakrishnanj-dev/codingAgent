## Summary

T-0.4 delivers the Event Log (C14) and Session/Lineage store (C15) under a new `com.srk.codingagent.persistence` package. The event domain is a typed envelope `Event(seq, ts, payload)` whose `type` is *derived* from the payload (envelope/body cannot drift), a sealed `EventPayload` hierarchy of the eight payload kinds present in the contract fixture, a minimal sealed `ContentBlock` union (text/toolUse/toolResult), and the supporting enums (`EventType` ×14, `SessionMode`, `StopReason`, `OperationClass`, `PermissionDecisionOutcome`, `ToolResultStatus`, `EdgeType`, `SessionStatus`). `EventCodec` is the single JSONL serialize/parse seam (Jackson). `EventLog` (C14) is the append-only writer: it assigns monotonic gap-free `seq` from 0 (INV-1), flushes per event before returning (INV-2, NFR-LOG-DURABILITY), surfaces a persist failure as `PersistenceException` rather than continuing as if logged (AC-13.4), and exposes **no** update/delete API. `SessionStore` (C15) resolves paths under `~/.codingagent/projects/<repo-key>/sessions/<session-id>.jsonl` (+ `.meta.json`) (NFR-LOG-LOCATION), creates/locates session files, reads events back in seq order (AC-7.2 capability), and derives/writes the `.meta.json` summary (token totals + outcome). `seq`/`ts`/`sessionId`/`repoKey` are all boundary-supplied — no `Instant.now()`/`UUID.randomUUID()` in the writer/store (ADR-0005). `pom.xml` adds the Jackson BOM + databind (compile) and the networknt validator (test). `mvn clean verify` is green: 181 unit tests (64 new), bundle line coverage 94.28% with the 0.80 JaCoCo gate met and no new exclusion. Five contract tests (CT-SCH-1/2/3/4, CT-INV-1) are satisfied — the CT-SCH set validates against the *formal* `event.schema.json` itself. Two Minors (a needless `transient` on a serializable field; one formatting-coupled meta assertion), one Nit, two Discussions (a cross-package validation-helper idiom; the in-memory whole-file read). No Blockers, no Majors.

### Minors

**[m1] [Correctness] `persistence/UnsupportedPayloadException.java:21` — `transient` on a serializable enum field nulls it on deserialization**
- Why: the field is `private final transient EventType eventType`. `Throwable` is `Serializable` and `EventType` (a Java enum) is serializable, so the `transient` keyword is unnecessary and actively wrong — were this exception serialized/deserialized, `eventType()` would return `null`. EJ Item 17/correctness: don't drop persistable state. Impact is low (these persistence exceptions are not normally serialized across JVMs), hence Minor not Major.
- Suggestion: drop `transient` so `eventType` survives serialization. (No change applied — Phase C is review-only; non-blocking.)

**[m2] [Testing] `persistence/SessionStoreTest.java:writeMeta_persistsDerivedSummary` — assertion couples to Jackson's pretty-print spacing**
- Why: the test asserts `json.contains("\"inputTokens\" : 100")` — the value `100` traces to AC-13.2 (token totals persisted), but the `" : "` spacing traces to the impl's `INDENT_OUTPUT` choice, not a spec symbol. The assertion still verifies the spec property (the summed token total is in the meta JSON), so the oracle-traceability self-check still passes, but the representation coupling is brittle (a formatter change would break a passing-spec test).
- Suggestion: deserialize the meta JSON and assert `meta.inputTokens() == 100` (parsed value, format-independent), or assert `contains("100")` against the token field by parsing. (Non-blocking.)

### Nits

**[N1] [Readability] `persistence/SessionStore.java:36` / `config/ConfigLocations.java:25` — the `.codingagent` / `projects` store-path literals appear in two classes**
- Why: `SessionStore` re-declares `STORE_DIR_NAME = ".codingagent"` and `PROJECTS_DIR_NAME = "projects"`, which also exist (privately) in `ConfigLocations`. Both correctly point at the same ADR-0005 store; the duplication is two short literals, not behavior. Author's choice — see D1 for the broader shared-path discussion.

### Discussion

**[D1] [Discussion] A shared validation-guard / store-path utility could absorb idioms now living in both `config` and `persistence`**
- Spec refs: ADR-0005 (on-disk layout: `~/.codingagent/projects/<repo-key>/...`), ADR-0009 (config store under the same root). Two small idioms now appear in both packages: (a) the `requireNonBlank`/`requireAtLeast` argument guards (private in `ResolvedConfig`, consolidated into the package-private `persistence.Payloads` for T-0.4 — so there is **no** duplication *within* T-0.4's own surface, only the pre-existing `ResolvedConfig` copy in another package); and (b) the store-root path constants (`.codingagent`, `projects`) in both `ConfigLocations` and `SessionStore`. Merging either would require a new shared package (e.g. `com.srk.codingagent.common`) and an edit to frozen T-0.2 code outside this task's scope — larger than a 3-line guard / two path literals warrant today. Flagged as a cross-cutting observation, not a Reuse-Major: a 3-line `if (blank) throw` guard is below the "non-trivial utility" bar, and T-0.4 did not itself duplicate. Suggested amendment kind: none (an implementation refactor, not a spec change) — surface to the coordinator as a candidate cleanup when a third consumer of either idiom appears.

**[D2] [Discussion] `SessionStore.readEvents`/`countEvents` materialize the whole log in memory; resume of a very large indefinite-retention log is unbounded**
- Spec refs: NFR-LOG-RETENTION (indefinite, operator-managed), AC-7.2 (resume = read JSONL in seq order), ADR-0005 (`.meta.json` gives a fast summary "without scanning the whole log"). `readEvents` uses `Files.readAllLines` and returns a `List<Event>`; for an indefinitely-retained session this is O(file size) memory. T-0.4's scope is "provide the read-in-seq-order capability," and the replay/`messages[]` reconstruction (where streaming would matter) is explicitly T-1.2. The `.meta.json` already exists precisely so a summary read need not scan the log. Flagged so T-1.2 considers a streaming/iterator read for replay rather than inheriting the eager list. Suggested amendment kind: none (a future-task implementation note, not a spec gap).

### Praise

**[P1] [API-Design] Append-only enforced by the type's surface, and identity captured at the boundary**
- CT-INV-1 asks that "there is NO public API to update or delete an event; append-only is enforced by the type's surface, not just convention," and ADR-0005 makes boundary-captured ids/ts load-bearing for reproducibility. The design realizes both cleanly: `EventLog`'s only mutator is `append`, which *reassigns* the caller's `seq` to the next monotonic value (a caller cannot make the sequence non-monotonic), takes the caller's `ts` verbatim (no `Instant.now()`), and the structural absence of update/delete is asserted reflectively (`noPublicMutationApi_exceptAppend`). `Event.type` derives from `payload.type()` so the envelope can never claim a type its body contradicts. The CT-SCH suite validates against the *formal* schema artifact rather than the code's own deserializer, anchoring the contract to the spec.
