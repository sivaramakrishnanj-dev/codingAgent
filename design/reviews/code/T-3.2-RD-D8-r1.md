## Summary

D8 is a live-only regression of T-3.2, downstream of the D7 prompt fix. After D7, the model
IS steered to call `write_artifact` (the live trace shows the call attempted), yet
`design/00-requirements.md` still landed as a single approval-stamp line with no requirements
content. **Root cause (confirmed by the new production-wiring test, not assumed):** `write_artifact`
is Class X (`OperationClass.SIDE_EFFECTING`), so on the live REPL path `AgentLoop.gateRequestFor`
routes it through the `PermissionGate` as a plain `GateRequest.forTool(..., write_artifact,
SIDE_EFFECTING)`; in the production-default `ASK_EVERY_TIME` mode (RD-3) the gate calls
`approver.confirm(...)` — the real `InteractiveApprover`, which reads a line from the SAME
`answerSource` that `Main.runInteractive` also hands the phase-approval `InteractiveGreenfieldApproval`.
With only the one realistic developer `y` fed, the `write_artifact` gate prompt and the phase gate
contend for that single stdin line, so the deliverable-content write is denied (a non-affirmative /
EOF answer) and only the approval-gate's `appendLine` stamp lands. The hypothesis the coordinator
posed is **confirmed**: the existing D7 test (`GreenfieldArtifactPersistenceTest`) built the gate with
an always-approve `Approver`, so it never exercised the stdin-backed `InteractiveApprover` path that
broke live.

**The fix** lands in the single authorization chokepoint, `PermissionGate.evaluate` (component C8,
coverage-counted — unlike the JaCoCo-excluded `AgentLoopFactory`/`Main`): after the denylist test and
before the per-mode switch, a `write_artifact` Class-X call is auto-approved in every mode WITHOUT a
separate per-operation prompt. ADR-0012 makes the design-markdown write the one write sanctioned before
the breakdown is approved ("the agent writes only design markdown … until the breakdown is approved");
`GreenfieldArtifactStore` confines every `write_artifact` to the target repo's `design/` directory, and
the tool is offered ONLY by the pre-approval registry — so the carve-out can never reach a source file.
AC-1.4 stays structurally enforced by the withheld source-write tools; the general RD-4 source-write
Class X operations (`write_file`/`edit_file`/`run_command`, web-lookup, sub-agent spawn) keep their full
per-mode gating, so AC-9.4 is unchanged for them.

### Nits

**[N1] [Readability] `GreenfieldSharedStdinArtifactPersistenceTest.java` — two near-identical phase-loop builders (`liveSharedStdinDriver` builds a loop inline; `productionGatePhaseLoop` is the same loop).**
- Why: `liveSharedStdinDriver` (used by the two requirements tests, which go through the full
  `GreenfieldDriver`) and `productionGatePhaseLoop` (used by the design/tasks test, which drives the
  phase loop directly because `GreenfieldDriver.run` always begins at the initial REQUIREMENTS phase)
  both assemble the same composer + ASK_EVERY_TIME gate + phase loop. The split is deliberate — the
  requirements tests need the driver + phase-gate contention end-to-end, the design/tasks tests need the
  loop seam because the driver can't be started mid-machine — but the loop-assembly lines are duplicated.
- Suggestion: Optionally extract a single `phaseLoop(...)` helper both call. Taste, not a defect; the
  duplication is ~5 lines and each call site reads clearly.

### Praise

**[P1] [Design] The carve-out lives in the gate (C8), keyed on the sanctioned tool name, after the denylist test.**
- Placing the exemption in `PermissionGate.evaluate` (the single authorization chokepoint) — rather than
  special-casing `write_artifact` in the JaCoCo-excluded `AgentLoop.gateRequestFor` / `AgentLoopFactory`
  — keeps the safety decision in the one coverage-counted seam the rubric and the contract tests already
  watch, and keeps the gate the sole authorizer (INV-8). Running it *after* the denylist test means a
  destructive command can never benefit from it, and keying on the specific tool name (not the operation
  class) means only the one path-confined, pre-approval-only tool is exempt.

**[P2] [Testing] The new test uses the production approval wiring the D7 test bypassed.**
- `GreenfieldSharedStdinArtifactPersistenceTest` drives a real greenfield phase loop with the real
  `PermissionGate(ASK_EVERY_TIME)` over a real stdin-backed `InteractiveApprover`, plus the real
  `ArtifactApprovalGate`/`InteractiveGreenfieldApproval` reading the SAME `answerSource` — exactly as
  `Main.runInteractive` wires them. Verified to FAIL with the carve-out disabled (reproducing both D8
  failure modes: content-denied/no-file, and content-written-but-no-stamp depending on stdin ordering)
  and PASS with it enabled. This is the path the always-approve D7 test could not reach.

### Discussion

**[D1] [Discussion] AC-9.4's prose ("prompt before every Class X") does not name the sanctioned design-markdown-write carve-out; ADR-0012/RD-7/AC-1.2/AC-2.1 authorize it, but the AC text reads absolutely.**
- Spec refs: **AC-9.4** (`00-requirements.md:290`, "While in `ASK_EVERY_TIME`, the agent shall prompt
  before every Class X operation"); **ADR-0012** ("the agent writes only design markdown … until the
  breakdown is approved"); **RD-7** (`:180`); **AC-1.2** (`:207`); **AC-2.1** (`:216`).
- The tension the fix resolves: AC-9.4 read literally would prompt before the `write_artifact` Class X
  call, which on the shared-stdin live path starves the phase gate and prevents the very persistence
  RD-7/AC-1.2/AC-2.1 require. ADR-0012 supplies the affirmative authority — the `design/`-confined
  design-markdown write is the one sanctioned pre-approval write — and RD-4's Class X enumeration
  (file write/edit, `run_command`, web-lookup, sub-agent spawn) does not contemplate a path-confined
  artifact writer distinct from a source write. So the carve-out is implementable as code today without
  amending AC-9.4: it does not weaken AC-9.4 for any RD-4-enumerated Class X operation. This is a
  spec-PROSE alignment worth noting, not a blocking gap — AC-9.4 would read more precisely with an
  explicit clause excepting the sanctioned `write_artifact` (mirroring how AC-9.3's UNRESTRICTED carve-out
  for the denylist is spelled out). Suggested amendment kind: `ac-update` (add the carve-out clause to
  AC-9.4 and/or a one-line note in ADR-0012's gate-reuse decision). Routed as Discussion; does not affect
  the resolved verdict.
