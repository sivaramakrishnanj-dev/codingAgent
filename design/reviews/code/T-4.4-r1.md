## Summary

T-4.4 realizes the OQ-I prompt-cache placement decision (ADR-0006) at the C4 wire boundary. `ConverseWireMapper`
gains an optional `PromptCacheCaps` (threaded by construction, exactly like the DCR-2 `maxOutputTokens` budget
flows). When the resolved model reports prompt-cache support (non-null caps, ADR-0002 / § 2.6 "null =
unsupported") AND the stable-prefix estimate meets the model's per-checkpoint token minimum
(`PromptCacheCaps.minTokensPerCheckpoint()`, Opus 4.5/4.6 ≥ 4096), the mapper appends a single
`SystemContentBlock.fromCachePoint(CachePointBlock(DEFAULT))` to the END of the `system` content array. Because
the cached region is everything *before* the breakpoint and cache order is tools→system→messages (§ 6.A.1), a
breakpoint at the tail of `system` marks tools + system (the memory-index is the tail of the system content, § 7)
as the cached prefix and leaves the variable `messages` tail uncached — the "largest static region" placement
ADR-0006 specifies. Absent support OR a sub-minimum prefix ⇒ no breakpoint, the request still builds and is valid
(graceful degradation, loop unaffected). Per ADR-0006's conservatism, exactly ONE breakpoint of the default kind
is placed — no multi-checkpoint management, no TTL micro-management. The capability gate is wired in by
`AgentLoopFactory` (the JaCoCo-excluded composition root) resolving `capabilityProfile(config).promptCache()` and
threading it through `new ModelClient(bedrock, caps)` and `ModelClient.forGreenfield(bedrock, caps)`; the
placement + gating LOGIC itself lives in the coverage-counted `ConverseWireMapper` (97% line / 90% branch).

Build-time token-minimum check: Bedrock returns exact `usage` only *after* a call (§ 6.A.1), so the gate uses a
documented conservative char-based estimate (~chars/4 across the system blocks) rather than an embedded
tokenizer (out of scope for v1) — see `stated_assumptions`. The live "cache write/read tokens observed (manual)"
in the task's Verify column is a future on-Bedrock observation, NOT a unit test; the unit-testable contract here
is the placement + capability gate, inspected on the built `ConverseRequest` without any Bedrock call.

### Nits

**[N1] [Readability] `ContentBlock` class Javadoc's cachePoint forward-reference is now slightly stale.**
`persistence/ContentBlock.java:24-27` says the cachePoint schema variant "is deliberately not modelled yet — it
is realized by the task that needs it ... See `stated_assumptions` in the task handoff." T-4.4 is that task, and
it (correctly) realized cachePoint at the WIRE level (`ConverseWireMapper` over the SDK `CachePointBlock`), NOT
as a domain `ContentBlock` variant. The comment is not *wrong* (cachePoint is still not a domain ContentBlock —
it is a wire-only placement concern), so this is taste, not a defect. Left for a future touch; Phase C does not
edit Phase A files, and re-opening Phase A for a comment-only tweak in a different component is not warranted.
- Suggestion: on a future `ContentBlock` touch, append "(realized at the wire boundary in T-4.4 —
  `ConverseWireMapper` places the cachePoint; it remains a wire concern, not a domain block)."

### Praise

**[P1] Threading follows the established `maxOutputTokens` seam exactly.** The prompt-cache caps are a per-session,
per-model constant (they don't vary call-to-call), so carrying them in the mapper constructor — the same
mechanism the DCR-2 output budget already uses — keeps the `toRequest` signature untouched and every existing
caller working. New ctor/factory overloads delegate to the canonical 2-arg forms, so the change is purely
additive: 0 of the 1264 prior tests needed edits.

**[P2] Placement is layered correctly and not duplicated.** cachePoint is a wire-format breakpoint, not domain
content; placing it in the wire mapper (the one place SDK types are constructed) over the SDK's `CachePointBlock`
is the right layer — the `ContentBlock` design explicitly left cachePoint out of the domain model. No domain
`CachePoint` type was invented, and no second token-estimator exists to collide with `estimatePrefixTokens`.

**[P3] The unpinned figures stay off the oracle surface.** The tests assert the placement contract (a cachePoint
is/ isn't present, where, how many, what kind) against ADR-0006, and assert the gate against the ADR-0006-pinned
4096 minimum. The char-per-token divisor (4) and `maxCheckpoints`/`ttls` shape are never asserted as oracles —
they are impl heuristics / value-object shape, not spec-pinned values.

### Discussion

**[D1] [Discussion] The token-minimum gate uses a build-time char heuristic, not the model's tokenizer.** ADR-0006
gates on "the prefix meets the model's token minimum (Opus 4.5/4.6: ≥4096)", but exact token counts come from
Bedrock `usage` only AFTER a call (§ 6.A.1). T-4.4 implements the gate faithfully via a documented conservative
~chars/4 estimate (under-estimates token count, so it errs toward NOT caching a borderline prefix — the safe
direction: a too-small cachePoint would just be a no-op, but we avoid even that). An exact pre-call tokenizer is
out of scope for v1. The task brief explicitly sanctioned this (and noted a "gate on capability-support only,
let the backend no-op a sub-minimum prefix" reading would also be defensible); I chose to honour BOTH ADR-0006
gate conditions rather than the weaker capability-only reading, because ADR-0006 states the minimum condition
explicitly. The "(manual)" live cache-write/read-token observation in the Verify column remains the future
on-Bedrock validation step — it is not in unit-test scope and is not a blocker.
- spec_refs: ADR-0006 (OQ-I), § 6.A.1
- suggested_amendment_kind: none required (the estimate is an impl choice within ADR-0006's bound; a future task
  may swap in a real tokenizer or a config-tuned divisor if live observation shows the heuristic mis-gates).
- Non-blocking; surfaced for awareness + the future manual live-cache-token observation.

## Reuse / duplication
Reuse sweep clean. cachePoint placement existed nowhere before (`ContentBlock` deliberately left the cachePoint
schema variant unmodelled; the prior `ConverseWireMapper` Javadoc named it explicitly deferred to "the task that
needs it" — this task). No prior build-time token estimator exists — `estimatePrefixTokens` is the only one, so
no DRY collision. The constructor-threading + delegating-overload idiom mirrors the existing `maxOutputTokens` /
`forGreenfield` pattern in the same two classes (consistency, not duplication). The cachePoint is placed via the
SDK's `SystemContentBlock.fromCachePoint(...)` factory — no bespoke wrapper invented.
