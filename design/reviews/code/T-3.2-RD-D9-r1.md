## Summary

D9 is the third regression-of-T-3.2: after D7 (prompt) and D8 (gate auto-approve), a live G3
greenfield run STILL persisted no design-doc content — `GreenfieldArtifactStore.write(...)` was
never called (no "Wrote greenfield artifact" log anywhere), only `appendLine(...)` (the approval
stamp). I root-caused this with instrumentation BEFORE fixing, per the investigate-first mandate.

**Confirmed root cause (classification d — malformed/no `write_artifact` call due to a
schema↔purpose mismatch), with per-hop evidence:**

A probe test (`GreenfieldFullDriverProbeTest`, since deleted) drove the FULL production
`GreenfieldDriver` across all three pre-approval phases over a scripted model + the production
composition (`ToolRegistryComposer` pre-approval registry + per-phase prompt + real
`PermissionGate` ASK_EVERY_TIME + real `ArtifactApprovalGate`, ONE shared stdin). With a
well-formed scripted `write_artifact(path, content)`, **all three artifacts received content + the
stamp and the session reached implementation** — proving hops (a) dispatch routes
`write_artifact` → `WriteArtifactTool`, (b) the tool calls `GreenfieldArtifactStore.write()`, and
(c) the gate auto-approves it (D8 carve-out) are ALL sound. The plumbing is not the defect.

The probe also dumped the **rendered `toolConfig` the model actually sees** for each pre-approval
phase (the thing no prior test ever inspected for `write_artifact`):

```
phase REQUIREMENTS toolNames=[read_file, grep, glob, list, read_memory, write_artifact]
wire write_artifact schemaJson={"properties":{
   "content":{"type":"string","description":"The full new contents of the file."},
   "path":   {"type":"string","description":"Workspace-relative path of the file to write."}},
   "required":["path","content"]}
```

`WriteArtifactTool.inputSchema()` returned `ToolSchemas.writeFile()` — the **same schema object as
the generic source-write `write_file` tool**, whose per-field descriptions read *"Workspace-relative
path of the file to write"* / *"The full new contents of the file."* — a **generic source-file
writer**. The greenfield pre-approval prompt simultaneously and emphatically forbids the model from
writing source/files ("you cannot and must not write source code yet", "do not create or edit any
source file", "you write design documents this way, not source code"). The schema the model fills
its arguments from gave **no signal** that `path` is a `design/`-confined artifact path (e.g.
`design/00-requirements.md`); that example lived only in the prose prompt, not the machine-readable
schema. A model handed a tool whose schema reads like a generic file writer, against a prompt
forbidding file/source writes, declines to call it — matching the live evidence exactly: model
active (14 tool_use turns), but the `write_artifact` content write never fires; only the gate's
`appendLine` stamp ever touches the store.

**Why D7/D8 (and their tests) missed it:** both hand-script a perfect `write_artifact(path,
content)` tool-use and assert the content lands — proving the plumbing, but assuming away the very
decision that broke live (whether the model calls the tool at all). Neither D7 nor D8 touched the
tool's own `inputSchema` (the contract the model reads to decide), and neither inspected the
rendered `write_artifact` toolSpec.

**Fix (one place, the tool contract the model reads):** a dedicated `ToolSchemas.writeArtifact()`
schema whose field descriptions state the design-doc-artifact contract — `path` is a
`design/`-relative artifact path with the concrete `design/00-requirements.md` example and "This is
a design document, not a source file"; `content` is the full markdown deliverable — and
`WriteArtifactTool.inputSchema()` now returns it (was `ToolSchemas.writeFile()`). Field *names*
(`path`/`content`) are unchanged (the handler reads them; the prompt references them); only the
*descriptions* change so the schema agrees with the tool's purpose and the prompt (C7 invariant). No
change to the store, gate, registry, prompt, or dispatch.

### Minors

**[m1] [Testing] `GreenfieldWriteArtifactSchemaPersistenceTest.java` (`fullProductionDriverPersists…`) still passes when the fix is reverted.**
- Why: That test scripts the model to emit `write_artifact` regardless of the schema, so — like the
  D7/D8 persistence tests — it proves the dispatch/store path across all phases but does not, by
  itself, catch the D9 schema defect. That catch is delegated to the two *schema* tests
  (`renderedWriteArtifactToolSpecDescribesADesignDocArtifact`,
  `inputSchemaDescribesADesignDocArtifactNotASourceFile`), which DO fail without the fix. This is a
  deliberate division of labour (contract-the-model-reads vs. full-plumbing), documented in the
  test's Javadoc, not a gap — but a reader should not mistake the persistence test for the
  regression guard. Severity Minor; no change required.
- Suggestion: none required; the schema tests are the regression guard and the persistence test is
  the end-to-end coverage. Documented in the class Javadoc.

### Nits

**[n1] [Readability] `GreenfieldWriteArtifactSchemaPersistenceTest.java` has two `composer(...)`
overloads (one taking an `Approver`, one taking a `ModelClient`).**
- Why: Two private helpers named `composer` with different parameter lists is mildly more to read
  than one. They serve genuinely different needs (the schema tests need a never-called model; the
  persistence test needs the scripted model + shared-stdin approver), so the overload is reasonable.
- Suggestion: optional — could rename to `composerForRendering` / `composerForDriver`. Author's call.

### Praise

**[P1]** The fix is the minimal correct change: it targets the one artifact neither prior fix
touched (the tool's own `inputSchema`, the contract the model reads), keeps the field names stable
so the handler and prompt are untouched, and reuses the existing `objectSchema`/`stringProperty`
schema helpers rather than hand-rolling JSON.

### Discussion
None. The fix needs no spec amendment: RD-7/AC-1.2/AC-2.1 require the design-doc *content* to be
persisted and ADR-0012 makes the design-markdown write the one allowed pre-approval write; the C7
invariant ("a tool's schema and its handler agree") is the affirmative authority for making the
schema describe the design-doc write it is. No spec is silent or contradictory here.

## Reuse / duplication
The new `ToolSchemas.writeArtifact()` reuses the package's `objectSchema`/`stringProperty` builders
(good reuse). It deliberately does NOT reuse `ToolSchemas.writeFile()` — that reuse WAS the D9 root
cause (a design-doc tool wearing a source-file schema). The field NAMES (`path`/`content`) coincide
with `writeFile()` by necessity (the handler reads them), but the DESCRIPTIONS diverge intentionally;
this is a required divergence, not a duplicate-to-merge. `reuse_self_check: passed`.

## Design-doc alignment
RD-7 / AC-1.2 / AC-2.1 (deliverable content persisted as markdown in the target project) — now
served, because the schema the model reads agrees with the design-doc write. C7 invariant
(schema↔handler agree) — preserved and strengthened. AC-1.4 (no source write pre-approval) —
unaffected: path confinement is enforced by `GreenfieldArtifactStore` (unchanged), and the schema now
explicitly states "This is a design document, not a source file." AC-1.5 / AC-2.5 — exercised by the
full-driver persistence test (stamp + traceability admit into implementation). No `[Design-Gap]`.
