## Summary

T-2.8 is an M2 integration / live-reachability regression (of T-2.2) — the compaction analogue of
T-2.7. The `Compactor` (T-2.2) was fully built + unit-tested but **not wired into the live
`AgentLoop`**: on `BudgetGuard.Decision.COMPACT` the loop only LOGGED and returned
`LoopOutcome.surfaced(...)`, the constructor had no compaction collaborator, and
`AgentLoopFactory` never built a `Compactor`. So the G2 headline ("a long task compacts +
continues") was unreachable from the live CLI — the same green-against-mocks / broken-live gap
as T-2.7 and the M0 D1/D2 defects.

The fix introduces a `CompactionSeam` (loop package, the analogue of `BudgetGuard` — default
`NONE`), threads it into the loop's T13 budget hook so a `COMPACT` signal now invokes the seam and,
on a `CONTINUED` result, **continues driving in the derived session** (state machine A T14) rather
than surfacing-and-stopping; on a `SURFACED` result (compaction failure, T15) it surfaces
`MODEL_CONTEXT_WINDOW_EXCEEDED`, which `OneShotRunner` already maps to exit 5 (CT-SM-7, cli-exit-codes
5) — no mapper change needed. The production seam (`CompactingSeam`) is extracted into the `cli`
package as a **gate-covered** unit (the T-2.7 lesson) that wraps the real `Compactor`, threads the
live session identity + a boundary-captured derived-session-id supplier (ADR-0005), and replays the
derived session into the `messages[]` the loop continues in (reusing `SessionReplay` — the same
events→messages projection `ResumeCommand` uses). `AgentLoopFactory` (JaCoCo-excluded) narrows to
composing the seam + the shared `MemoryStore`-backed learning harvest. `mvn clean verify` is green:
**844 tests, 0 failures, JaCoCo BUNDLE LINE 0.80 gate met**; `AgentLoop` is 96/96 line-covered
(incl. the new COMPACT branch) and `CompactingSeam` is 23/24.

## Discussion (spec-level; never blocks resolution)

**[D1] [Discussion] AC-18.5 live harvest is wired but inert at v1 (no production `LearningExtractor`).**
- Spec: AC-18.5 — "Where durable learnings are identified during compaction, the agent shall propose
  them for memory (per US-21) before archiving." The harvest seam is now wired on the live path
  (`MemoryLearningHarvester` over the shared `MemoryStore` + `LearningProposer`, invoked by the Compactor
  before derive), satisfying the *ordering* AC-18.5 requires. But T-2.5 left `LearningExtractor` a seam
  with only `NONE` in production — there is no production summary-extraction heuristic — so a live
  compaction identifies (and proposes) zero learnings, and the one-shot approver is `DENY_ALL` (the safe
  no-developer-present default, AC-21.4/INV-13). The harvest is therefore reachable-but-inert at v1.
- This is the correct anti-poisoning stance (never auto-extract without approval), and inventing a
  production extractor is outside this wiring task's scope. Surfaced so the designer can confirm whether
  a production `LearningExtractor` + an interactive `LearningApprover` (REPL) is a planned later task or
  whether AC-18.5 should note "v1 wires the seam; extraction heuristic deferred."
- Suggested amendment kind: `ac-update` (a note on AC-18.5 / ADR-0007 that v1 ships the seam without an
  extraction heuristic). Not written here — Discussion only.
