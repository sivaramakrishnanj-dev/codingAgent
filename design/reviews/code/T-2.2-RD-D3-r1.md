## Summary

T-2.2-RD-D3 is a regression-of-T-2.2 fixing the third live-only Bedrock-wire defect in the
compaction path (same class as D1/D2): `Compactor.summarize()` passed `null` as the `toolConfig`
arg on its summarizer Converse call. The summarizer replays the original transcript verbatim — and
a real coding session's transcript carries `toolUse`/`toolResult` content blocks — so Bedrock
rejected the live call with `ValidationException: "The toolConfig field must be defined when using
toolUse and toolResult content blocks."` (§ 6.A.1 wire rule). It passed against the in-test
`ScriptedBedrockClient` (which never enforced the rule) and broke live.

The fix follows the user-chosen approach exactly: thread the **session's** `ToolConfiguration`
(the tools available in the session being summarized) into the `Compactor` and pass it on the
summarizer call instead of `null`, keeping the tool blocks in the transcript verbatim (best summary
fidelity) — no stripping or flattening. Concretely:

- A nullable `ToolConfiguration toolConfig` field is added to `Compactor`, set via the constructor.
  The two-constructor convenience pattern is preserved: the no-harvest constructor now takes
  `(..., recentTailTurns, toolConfig)` and delegates to the canonical
  `(..., recentTailTurns, toolConfig, harvester)` with `LearningHarvester.NONE`. `toolConfig` is
  null-tolerant **only** for the genuinely-no-tools session (the wire rule only bites when
  `messages[]` carry tool blocks); the live path always threads the real registry's config.
- `summarize(...)` now passes the threaded `toolConfig` (was `null`).
- The class-level and `summarize` Javadoc are corrected: they previously implied "no tool config";
  they now state that the request carries the session's tool definitions (citing § 6.A.1 and that
  the tool blocks are kept verbatim) while the *response* summary text still excludes reasoning
  blocks.
- The live wiring is closed at the composition root: `AgentLoopFactory.create(...)` threads
  `tools.toToolConfiguration()` (the already-built parent registry's rendered config, C7) into
  `compactionSeam(...)`, which passes it to the live `Compactor`. `AgentLoopFactory` is
  JaCoCo-excluded, so the unit coverage for the fix comes from `CompactorTest` directly.

`mvn clean verify` is green: **846 tests, 0 failures/0 errors, JaCoCo BUNDLE LINE 0.80 gate met
("All coverage checks have been met")**. `Compactor` is 94/104 line-covered (90.4%), 25/28
branch-covered. The new regression test fails against the pre-fix `null` (it asserts
`request.toolConfig() != null` on a tool-block transcript).

### Praise

**[P1] The regression test asserts the wire contract that broke live, not the impl's call shape — and it genuinely bites.**
- `sixA1_summaryCallCarriesSessionToolConfigForToolBlockTranscript` first asserts the *precondition*
  (the replayed summary transcript carries `toolUse`/`toolResult` blocks — the exact condition that
  triggered the live `ValidationException`), then asserts `request.toolConfig() != null` **and** that
  it carries the summarized session's tool definition (`read_file`, the tool whose blocks
  `seedOriginal` plants). The oracle is phrased as the § 6.A.1 wire rule / live error, not "the impl
  now passes toolConfig" — and the assertion fails against the pre-fix `null`, so it is a real
  regression guard (the D2-class lesson: the double must *verify* the toolConfig argument is present
  and correct for a tool-block transcript, not merely that `summarize` returns text).

  logic, the fixed `COMPACTION_SYSTEM_PROMPT`, and the harvest seam are untouched.

## Design-doc alignment

- **§ 6.A.1 (the verified Bedrock Converse wire rule):** a request whose `messages[]` carry tool blocks
  must define `toolConfig`. The summary call now presents the session's `toolConfig`, so a tool-block
  transcript is wire-valid — the exact rule the live `ValidationException` enforced. ✅
- **ADR-0006 (OQ-D):** the summary call remains a dedicated Converse call to the configured summarizer
  model with the fixed compaction prompt, getting the original transcript verbatim. ADR-0006 pins WHAT
  the call does and does not say "no toolConfig" — the prior `null` was an implementation assumption,
  now corrected. The fix keeps the call spec-faithful (real transcript verbatim) while satisfying the
  wire rule. ✅
- **AC-18.2 / AC-18.3 / AC-18.4:** unchanged — manual/threshold triggering, original-preserved-and-linked,
  and summary-carries-enough-context are all untouched; the fix only changes the summarizer request's
  `toolConfig` argument, not the transcript content, the derive logic, or the summary framing. ✅
- **INV-4 / INV-5 / INV-7:** preserved — derive-don't-mutate, original preserved byte-identical, and
  reasoning signatures replayed verbatim are unaffected (re-verified by the unchanged CT-INV-3/INV-4/INV-6/INV-7
  tests, all still green). The tool blocks are kept verbatim in the transcript (no stripping/flattening),
  consistent with INV-6 pairing. ✅
- **C6 / C7 / C4 component boundaries:** the Context Manager (C6, `Compactor`) consumes a
  `ToolConfiguration` rendered by the Tool Registry (C7, `toToolConfiguration()`) and passes it through
  the Model Client (C4, `converse(..., toolConfig)`) seam — no boundary crossed or widened; `ModelClient`
  and `ToolRegistry` are unchanged. ✅
