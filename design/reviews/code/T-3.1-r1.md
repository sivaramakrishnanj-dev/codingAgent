## Summary

**The two load-bearing safety contracts are realized structurally, not by prompt.** AC-1.4 ("while in

### Nits

**[N1] [Testing] `GreenfieldPlaybookTest.java:namedToolsMatchRegisteredToolset` â€” constant-equality assertion borders on impl-literal**
- Why: `assertEquals("read_file, grep, glob, and list", GreenfieldPlaybook.EXPLORE_TOOLS)` and the
  `SOURCE_CHANGE_TOOLS` equivalent assert exact prose strings that are the playbook's own constants.
  The literals do trace to RD-4 (Class R = read/grep/glob/list; Class X = write/edit/run_command), and
  the mirror `BrownfieldPlaybookTest.namedToolsMatchRegisteredToolset` does exactly this, so it is a
  deliberate convention rather than a defect â€” but it is the weakest oracle link in the suite (an exact
  prose-string match rather than a behaviour-keyword match). It is kept because the constants are the
  RD-4 tool taxonomy the registry actually offers, and the test's value is catching a drift between the
  playbook's named tools and the registry's tool names.
- Suggestion: leave as-is for parity with the brownfield test; if the wording ever churns, relax to a
  keyword/contains assertion. No action required for T-3.1.

### Praise

**[P1] [Design] Structural AC-1.4 enforcement via tool withholding, not gate-denial**
- `ToolRegistryComposer.greenfieldRegistry(phase)` makes "no source write before approval" a property
