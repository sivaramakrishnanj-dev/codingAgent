## Summary

T-2.4-RD-D5 is a recall-side live-only integration/wiring fix (regression-of-T-2.4): the two-tier
memory INDEX is built and maintained on disk (`MemoryStore.loadIndexes` exists and is unit-tested,
re-read-fresh per INV-14/AC-14.2) but was never injected into the live system prompt. The factory
handed the loop the STATIC `BrownfieldPlaybook.systemPrompt()` and nothing else, so a fresh
real-Bedrock session was blind to which curated entries exist — it had to GUESS a slug for
`read_memory` (guessed "money-representation", actual "money-values-use-long-cents") and missed,
then fell back to grepping the filesystem. Same live-only class as T-2.7 (tools composed but
unwired) and T-2.8 (compactor unwired): capability + API exist and are unit-tested, but the live
composition root does not wire them into what the model sees.

The fix follows the established T-2.7 gate-covered-seam pattern. `ToolRegistryComposer` (NOT
JaCoCo-excluded; already holds `memoryStore` + `repoKey`) gains a `parentSystemPrompt()` method
that loads both tiers' indexes via `MemoryStore.loadIndexes(repoKey)` at assembly time (session
start) and appends ONE additional system-prompt block rendering each entry's slug + one-line
description — only when the index is non-empty. The JaCoCo-excluded `AgentLoopFactory` is lifted to
hold the composer in a local and now passes `composer.parentSystemPrompt()` (instead of the static
playbook) to the loop's `system` arg; no second `MemoryStore` is constructed (the existing
`memoryStore` local is reused). `mvn clean verify` is green: 852 tests, 0 failures, JaCoCo BUNDLE
LINE gate (0.80) met (overall 92.24%). The contract — index content reaches the live system prompt
— is now pinned by `LiveSystemPromptCompositionTest`, the test that would have caught D5.

## Reuse / duplication

Reuse self-check: **passed.** `loadIndexes` is called from exactly one production site (the new
`parentSystemPrompt()`); no other source file injects the index into a prompt. The only existing
renderer of the `- [slug] description` shape is `MemoryIndex.renderLine` (package-private in
`memory`, takes a `MemoryEntry`, produces one on-disk INDEX.md line) — a different input type
(`MemoryEntry` vs. the `MemoryIndexLine` I have), a different output purpose (an on-disk index line
vs. a multi-line system-prompt block), and it is inaccessible from `cli` (package-private). The
coincidental `- [slug] desc` substring is the ADR-0007-pinned surface format, not copied logic. No
duplicate; `renderMemoryIndexBlock` is the single site that renders an index block into a prompt.

## Design-doc alignment

- **ADR-0007 "Index + selective load" (RD-9, AC-14.3):** "On session start, both indexes load into
  the system prompt (cheap awareness)... The agent pulls a full entry on demand via a
  `read_memory(slug)` tool." `parentSystemPrompt()` loads both tiers via `loadIndexes(repoKey)` at
  assembly time and renders slug + description per entry, naming the `read_memory(slug)` retrieval
  path. ✅
- **ADR-0007 "Two tiers" (ordering):** "on load, both tiers' indexes are read." `loadIndexes`
  returns GLOBAL-then-PROJECT and the block renders them in that order; `bothTierEntriesReach...`
  pins both tiers visible. ✅
- **ADR-0007 "Read-fresh + external edits (AC-14.2)" / INV-14:** "re-read from disk on each load —
  not baked into a resumed transcript." Loaded per call to `parentSystemPrompt()` (session start);
  no caching introduced (`MemoryStore` holds no masking cache). `indexIsReReadFreshPerSessionStart`
  proves a second write appears on the next build. ✅
- **AC-14.3 (index for quick review) / empty→no section:** empty index appends no block;
  `emptyIndexAddsNoMemorySection` proves the prompt equals the playbook blocks exactly. ✅
- **ADR-0012 / D5 requirement (do not replace or reorder the priming):** `parentSystemPrompt()`
  starts from `BrownfieldPlaybook.systemPrompt()` unchanged and only appends; the G1
  explore→edit→verify gate's blocks are preserved in order. `playbookBlocksPreservedAndIndexAppended`
  pins size = playbook+1 and the prefix = the playbook blocks. ✅
- **ADR-0006 (cacheable static prefix):** the index block sits in the static prefix alongside the
  playbook + tool defs, consistent with the prompt-caching note. ✅
