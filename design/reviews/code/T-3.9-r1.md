## Summary

T-3.9 wires the greenfield IMPLEMENT phase's **single end-of-phase verify** on top of the post-T-3.8
implement-every-task loop (DCR-7): once every task is implemented + marked complete, the configured
build/test command runs **once** over the whole phase via the reused T-1.4 `VerifyLoop`
(`overConfig` already carried the `verifyExecutor`/`verifyConfig` collaborators T-3.8 reserved). The
single verify outcome maps to three new `ImplementOutcome` dispositions: a passing verify →
`ALL_IMPLEMENTED` (clean success, AC-3.2); an exhausted verify → `VERIFY_FAILED` (stop-and-surface,
AC-3.4/AC-20.5, CT-GF-7), bounded by `NFR-VERIFY-MAX-ITERATIONS` with the shared `RemedyPrompt`
driving a remedy turn between attempts (AC-20.3); and **no configured test command** →
`COMPLETE_WITH_WARNING` — a **terminal complete-with-warning** success that logs ONE warning and maps
through `asLoopTurn()` → driver → runner to a genuine `COMPLETED` (exit 0), killing the D1 re-prompt
livelock (AC-3.6, CT-GF-5). The demonstrably-wrong **AC-20.6 mis-citation** on the no-test-command
behavior is corrected at the named site (`VerifyLoop.java:129` + its surrounding Javadoc).
