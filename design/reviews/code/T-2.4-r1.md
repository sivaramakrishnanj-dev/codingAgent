## Summary

T-2.4 implements the two-tier curated-memory store (component C16, `com.srk.codingagent.memory`, ADR-0007) plus the memory tools (component C12, `com.srk.codingagent.tool.memory`). `MemoryStore` persists each learning as one human-editable markdown file `<slug>.md` (no DB, AC-12.2/AC-14.1) under a `~/.codingagent` store root that mirrors `SessionStore`'s layout: GLOBAL memory under `<store>/memory/`, PROJECT memory under `<store>/projects/<repo-key>/memory/` (AC-12.3, RD-9). The root is injectable (constructor arg + `forUserHome()` factory) exactly like `SessionStore(Path)` / `SessionStore.forUserHome()`, so the store is unit-testable against a `@TempDir`. Each tier carries an `INDEX.md` of `- [slug] description` lines (AC-14.3) — the always-loaded awareness surface; `loadIndexes(repoKey)` reads BOTH tiers' indexes (ADR-0007), and `readEntry(slug, repoKey)` pulls a full entry on demand.

**Re-read-fresh, no masking cache (INV-14, AC-14.2).** The store holds no in-memory copy of any entry or index: every `loadIndexes` / `readEntry` reads from disk, so a hand-edited or hand-deleted file is honoured on the very next load and a write is immediately visible to a subsequent read (CT-INV-12). **Curated-only, no auto-extract (INV-13).** The store's sole mutating method is `write(entry, repoKey)`; nothing persists an entry as a side effect of anything else (CT-INV-11, AC-21.4).

The agent-facing tools are `read_memory` (Class R — auto-approved, never gated, ADR-0004) and `write_memory` (Class X — gated by the permission mode, ADR-0004). `write_memory` is the **explicit** "remember X" write path (US-12, AC-12.1): it classifies the tier (AC-12.3), writes the entry + index line via `MemoryStore.write`, and logs a `MEMORY_WRITE` provenance event in the session log (AC-12.4). The `created` timestamp and `originSession` are boundary-captured (injected `Supplier<String>` clock + session id, ADR-0005 — never `Instant.now()`), consistent with `AgentLoop`/`Compactor`. A new typed `MemoryWritePayload` (mirroring T-2.3's `SubAgentSpawnPayload`) is added to the `EventPayload` sealed permits and the `EventCodec` switch; `MEMORY_WRITE` now decodes to a typed payload instead of throwing `UnsupportedPayloadException` (the schema enumerates `MEMORY_WRITE` but pins its payload as a generic object, so no schema edit — same arrangement T-2.2 used for `ERROR`). `mvn clean verify` is green (779 tests, +62 new; JaCoCo gate met). No Blockers, no Majors; two documented Minors (forced cross-package helper duplication).

## Casing resolution (ADR prose vs schema — resolved toward the schema; raised as Discussion)

The formal schema (`memory-entry.schema.json`) + the validated fixture pin **`tier` uppercase** (`GLOBAL`/`PROJECT`) and **`status` lowercase** (`active`/`retired`). ADR-0007's prose example shows lowercase `global|project` for tier, and `03-data-model.md` § 2.5 names the enum `MemoryStatus = ACTIVE | RETIRED` (uppercase). I resolved toward the schema (authoritative) per the task directive: `MemoryTier` constants are uppercase and emitted verbatim; `MemoryStatus` constants are uppercase Java names but `wireValue()`/`fromWire()` bridge to the schema's lowercase tokens, so emitted front-matter validates (CT-SCH-11) and an uppercase `status` is rejected (asserted in `MemoryEntrySchemaContractTest`). No schema or fixture edited. The prose/schema mismatch is logged as a `Discussion` (doc/adr-clarification) below — it never blocks resolution.

**[m1] [Reuse] `tool/memory/MemoryToolInputs.java:30` — `requireString` duplicates `tool/ToolInputs.requireString`.**
- Why: `ToolInputs.requireString` (`tool/ToolInputs.java:29`) is byte-for-byte the same logic, but `ToolInputs` is **package-private** to `com.srk.codingagent.tool`, and the memory tools live in the sub-package `com.srk.codingagent.tool.memory` (the C12 package per 02-architecture § 6), so they cannot reference it. The duplication is forced by the cross-package visibility rule, and is documented in `MemoryToolInputs`'s Javadoc.
- Suggestion: leave as-is for T-2.4 (a fix would widen `ToolInputs` to public or move it to a shared `tool.support` package — a refactor that touches the T-0.6 component, out of this task's scope). If a third sub-package ever needs it, promote `ToolInputs` to a shared internal package then. Not verdict-changing.

**[m2] [Reuse] `tool/memory/MemoryToolSchemas.java:60` — `objectSchema`/`stringProperty` duplicate the `tool/ToolSchemas` idiom.**
- Why: same forced-duplication story — `ToolSchemas` (`tool/ToolSchemas.java`) is package-private to `com.srk.codingagent.tool`, so the memory tools' schemas are authored locally. Documented in `MemoryToolSchemas`'s Javadoc. The `Document.mapBuilder()` shape is small and fixed.
- Suggestion: leave as-is; promote to a shared `tool` schema-builder helper only if/when a third tool package appears. Not verdict-changing.

### Praise

**[P1] Re-read-fresh is asserted as an out-of-band edit AND delete, on the real filesystem.** `MemoryStoreTest.externalEditHonouredOnNextLoad` writes an entry, edits the `.md` file *behind the store's back* (replacing the body via `Files.writeString`), and asserts the next `readEntry` returns the hand-edited body; `externalDeleteHonouredOnNextLoad` deletes the file and asserts the next read reports absent; `freshWriteImmediatelyVisible` proves a write through one store instance is visible through a *second* instance over the same root — three independent angles on INV-14/CT-INV-12 with no mock that could mask a cache.

**[P2] CT-SCH-11/12 are anchored to the formal schema, not to this code's construction rules.** `MemoryEntrySchemaContractTest` loads the verbatim `memory-entry.schema.json` via the networknt validator (the same pattern `EventSchemaContractTest` uses), validates both the validated fixture's front-matter and an entry this code renders, and asserts an out-of-enum `tier`, an uppercase `status`, and a missing required field are all rejected — so the casing resolution is proven against the spec artifact, not asserted by fiat.

**[P3] The MEMORY_WRITE event is asserted at-most-once and only on success.** `WriteMemoryToolTest.logsExactlyOneEventPerWrite` counts exactly one `MEMORY_WRITE` line per write (AC-12.4 — "record memory writes as events", not "write a burst"), and `rejectsBadSlug` asserts that a bad slug produces NO file and NO event — the entry is built (and self-validates) before the store write and before the event append, so a failed write never leaves a phantom provenance marker.
