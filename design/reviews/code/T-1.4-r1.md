## Summary

T-1.4 delivers the verify loop as a standalone, injectable, fully unit-testable C2/C10-boundary unit in `com.srk.codingagent.loop`: after a unit of work's changes are applied, run the configured test command, react to the exit code, and converge on success or — bounded by `config.verifyMaxIterations()` (default 5, NFR-VERIFY-MAX-ITERATIONS) — stop and surface the failure rather than continuing silently. Four new types: `VerifyLoop` (the run→check→remedy→retry control), `VerifyOutcome` (VERIFIED / EXHAUSTED / NO_TEST_COMMAND), `CommandRunner` (the injected command-runner seam mirroring `BudgetGuard` / `OneShotRunner.OneShotLoop`), and `RemedyAttempt` (the AC-20.3 "feed failure back and attempt a remedy" seam, with a `NONE` no-op default). The model-driven remedy is modelled as an injected seam the workflow driver (T-1.6/T-3.3) supplies; T-1.4 builds the bounded control + surface-after-N and tests it directly with scripted exit-code sequences (no real `mvn` shelled out in unit tests). The bound uses the config value, never a literal 5. `mvn clean verify` is green (554 tests; coverage gate met). No Blockers, no Majors, no Minors. One Discussion item: the CT-SM-5 Element-cell mis-cite (T13/T15 are compaction, not verify) is recorded for the designer.

### Praise

**[P1] The bound is tested as the config value at multiple values, not a hardcoded 5 — the exact D2-class defense the task called for.** `VerifyLoopTest.boundIsTheConfiguredValueNotALiteral` drives the loop at bounds 1/2/3 and asserts it surfaces after exactly that many failing attempts; `forConfigUsesConfiguredBound` proves the wiring carries `config.verifyMaxIterations()` into the loop (bound 2 → surfaces after 2). A loop that hardcoded 5 fails both. Paired with `exactlyMaxIterationsFailuresThenSurface` (the scripted runner *throws* if a 6th run is attempted, structurally proving no run past the bound) and `nMinusOneFailuresThenPassVerifies` (success on attempt N still verifies), the off-by-one boundary is pinned on both the surface side and the success side — the failure mode that only bites in a real long verify cycle.

**[P2] The success predicate is asserted in both directions, including the timeout's 124.** `timedOutRunIsFailure` constructs a real `CommandResult.timedOut` (exit 124) and asserts it is *not* success and surfaces with its output preserved — CT-INV-14/INV-17 failure direction, the dep-API note's explicit "timeout's 124 counts as failure". `zeroExitFirstAttemptVerifies` covers the success direction. No test treats a non-zero exit as success.

## Reuse / duplication

- **Seam idiom mirrored, not duplicated.** `CommandRunner` and `RemedyAttempt` follow the established injected-loop-control-seam idiom (`BudgetGuard` with its `NONE` default; `OneShotRunner.OneShotLoop`) — same shape, distinct concern. They are net-new interfaces, not a reuse of `BudgetGuard` (verify is a different decision than the budget check, per the dep-API note's explicit "do not overload BudgetGuard").
- **`CommandRunner.over(...)` composes the existing config-timeout pattern, not a copy of it.** It draws the per-command timeout from `ResolvedConfig.commandTimeoutSeconds()` and delegates to the real `CommandExecutor` — the same pattern `RunCommandTool` uses, applied at the verify boundary. The executor itself is reused, not reimplemented (the loop never constructs a process).
- **Bound drawn from config, not re-pinned.** `verifyMaxIterations` is consumed only via `ResolvedConfig.verifyMaxIterations()` (sweep confirms: the only `src/main` references are the config plumbing + this loop). No literal `5` anywhere in the new code.
- **No pre-existing verify/retry loop to merge with.** Sweep for `class Verify` / `interface Verify` / `for (int attempt` / `maxIterations` across `src/main` returns only the new `VerifyLoop`. This is the first bounded-retry control loop in the codebase.

## Design-doc alignment

- **AC-20.1 / AC-3.2** (run the configured build/test command to verify): `VerifyLoop.verify()` runs the configured test command via the `CommandRunner` seam; `forConfig` wires `config.commands().test()`. ✓
- **AC-20.4 / RD-10 / INV-17 / CT-INV-14** (zero exit from the configured test command = success, both directions): `verify()` returns VERIFIED iff `result.exitCode() == 0`; any non-zero (incl. timeout 124) is a failed attempt. Asserted both directions in `VerifyLoopTest` + `VerifyOutcomeTest`. ✓
- **AC-20.3** (feed failure output back and attempt a remedy): on a non-zero exit with an attempt remaining, `verify()` invokes `remedy.attempt(failure)` with the failing `CommandResult`; the remedy is the injected model-driven-fix seam the workflow driver supplies. Asserted via `RecordingRemedy` (fed the failure, count between attempts). ✓
- **AC-3.4 / AC-20.5 / CT-SM-5** (stop after NFR-VERIFY-MAX-ITERATIONS and surface, with the relevant output): after exactly `maxIterations` non-zero attempts, `verify()` returns EXHAUSTED carrying the *last* failing result; no (N+1)-th run. Asserted (`exactlyMaxIterationsFailuresThenSurface`, `surfacedOutcomeCarriesLastFailureOutput`). ✓
- **AC-20.6** (prefer configured named commands; absence is a config state): when `config.commands().test() == null`, `forConfig` builds a loop with a `null` runner that yields NO_TEST_COMMAND — no crash, no invented ad-hoc command. ✓
- **NFR-VERIFY-MAX-ITERATIONS = 5** (the bound is the config default, not a literal): `forConfig` reads `config.verifyMaxIterations()`; tests drive the bound at 1/2/3 and 5. ✓
- **02-architecture § 3.2** (verify-exhausted is *surfaced, not necessarily fatal*): `VerifyOutcome.EXHAUSTED` is a returned value the workflow driver branches on; the loop never calls `System.exit` (contrast a compaction failure's fatal exit 5). ✓
- **Component C2/C10 boundary respected** — the verify loop is its own unit in `com.srk.codingagent.loop`, runs the configured command via the C10 executor seam, and is NOT bolted into `AgentLoop.drive` (it is the workflow-side behavior T-1.6 wraps around the agent loop). ✓
- **Scope discipline:** the model-driven remedy turn (AC-20.3's reasoning step) is left as the injected `RemedyAttempt` seam for T-1.6/T-3.3; the verify loop is NOT wired into `AgentLoopFactory` (that factory composes the one-shot `AgentLoop`, not a workflow driver — wiring verify is T-1.6's job). `forConfig` is the documented composition seam those tasks call. ✓

## Discussion

**[D1] CT-SM-5's Element cell "A: T13/T15" is a mis-cite — those are the compaction transitions, not verify.** CT-SM-5's *assertion* ("verify loop stops after NFR-VERIFY-MAX-ITERATIONS and surfaces", refs AC-3.4/AC-20.5) is exactly T-1.4's contract and is built to faithfully. But its Element cell points at state-machine A T13 and T15, which (`state-machine.md` § A, verified) are the **compaction** transitions: **T13** `S1/S0 → S6` on `usage.inputTokens ≥ 0.85×window OR /compact` (→ machine B), and **T15** `S6 → S8` on compaction-failed (→ exit 5). The verify loop is not a numbered transition in machine A; it is the workflow self-verify behavior (US-20), and the surface it produces is the **S7 Surfacing** state, whose meaning explicitly names "verify-exhausted" (`state-machine.md` § A: *"a condition is surfaced to the user (guardrail/malformed/verify-exhausted/error)"*). Built to the assertion + the S7 verify-exhausted surface; did NOT contort the verify loop into the compaction transitions to match the cell (which would conflate the surfaced-not-fatal verify-exhausted outcome with the fatal exit-5 compaction failure — 02-architecture § 3.2 keeps them distinct). Suggested amendment kind: `contract-test-update` (correct the CT-SM-5 Element cell to reference the S7 verify-exhausted surface / US-20 instead of T13/T15), optionally a `state-machine-update` if the designer wants verify-exhaustion modelled as an explicit transition into S7. Mirrors T-1.2's CT-INV-3 reconciliation. Spec-level / informational only — does not affect the verdict.
