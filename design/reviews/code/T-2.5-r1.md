## Summary

T-2.5 implements the propose-and-approve curated-write path (US-21, component C12, ADR-0007) and wires the compaction learning-harvest (AC-18.5, component C6 → C12, ADR-0006). The work was inherited as uncommitted in-flight code from a prior task-builder that died on a transient API error before returning a handoff; this invocation reconciled it (ran `mvn clean verify` to establish the green baseline), assessed every new class for coverage, and added the missing tests — specifically the end-to-end harvest path and the per-class contract coverage the in-flight `LearningProposerTest` did not reach.

**The propose-and-approve path.** `LearningProposer.propose(LearningProposal)` is the single approved-write seam: it presents the proposal to the developer via an injected `LearningApprover` (AC-21.1); on `APPROVE` it writes the entry through the same `MemoryStore.write` the explicit `write_memory` tool uses (AC-21.3) with boundary-captured `created`/`originSession` provenance (AC-12.2, ADR-0005) and logs a `MEMORY_WRITE` event (AC-12.4); on `DENY` it persists nothing — no file, no index line, no event (AC-21.2/AC-21.4/INV-13). It returns `Optional<MemoryEntry>` so a caller can tell approve from deny. `LearningProposal` is the in-memory candidate (slug/tier/why/body, self-validating, with a `presentation()` line for the approval prompt); constructing one writes nothing.

**The compaction harvest.** `MemoryLearningHarvester implements LearningHarvester`: it runs `LearningExtractor.extract(summary)` to surface durable-learning candidates from the compaction summary text, then runs **each** through `LearningProposer.propose` — propose, never auto-extract (AC-21.4, INV-13). The `Compactor` gained a 7-arg ctor taking a `LearningHarvester` (the 6-arg ctor delegates with `LearningHarvester.NONE`), and `compact(...)` invokes `harvester.harvest(summary)` **after** a usable summary is produced and **before** `derive(...)` — the "before archiving" window AC-18.5 requires. The harvest only proposes; the original is preserved byte-identical regardless (INV-5), and a denied harvest never blocks the derive.

`mvn clean verify` is green: **805 tests** (T-2.4 baseline 779 + the in-flight `LearningProposerTest` 9 + 17 new this round), 0 failures, JaCoCo BUNDLE LINE 0.80 gate met (~92% bundle line). No Blockers, no Majors; one Nit (a same-value constant pair in the E2E test).

### Nits

**[N1] [Readability] `CompactionHarvestEndToEndTest.java:88` — `MEMORY_SESSION` and `ORIGINAL` are two constants holding the same string.**
- Why: `MEMORY_SESSION` is defined equal to `ORIGINAL` (`"2026-06-22T090000Z-original"`) to express "the harvested learning's `originSession` is the session being compacted" (AC-12.2 provenance). Two named constants with an identical literal can read as accidental rather than intentional.
- Suggestion: either reference `ORIGINAL` directly in the proposer wiring with a comment ("the harvest's originSession is the compacted session"), or keep `MEMORY_SESSION` but add a comment that the equality to `ORIGINAL` is the intended provenance. Author's call; not verdict-changing.

### Praise

**[P1] AC-18.5 is proven end-to-end through real collaborators, not asserted on the seam in isolation.** `CompactionHarvestEndToEndTest` wires a real `MemoryLearningHarvester` (real extractor + real `LearningProposer` + real `MemoryStore` + real `EventLog`) into a real `Compactor` and drives an actual `compact()`. The approved-path test recalls the entry from a *fresh* `MemoryStore` after the compaction completes, so it proves the whole summarize→harvest→derive ordering and the on-disk round-trip — the harvest path is genuinely exercised, which the task directive flagged as the difference between "AC-18.5 proven" and "AC-18.5 asserted."

**[P2] The harvested entry is validated against the authoritative schema, on disk.** Rather than trusting the in-memory `MemoryEntry`, `harvestedEntryFrontMatterIsSchemaValid` reads the `.md` file the harvest wrote to the PROJECT tier and validates its front-matter against `memory-entry.schema.json` (networknt validator) with explicit `tier: PROJECT` (UPPERCASE) / `status: active` (lowercase) checks — anchored to the schema, sidestepping the ADR-0007 prose lowercase-tier mismatch that the open-questions log records.

**[P3] INV-5 is asserted on both the approve and the deny path, by raw bytes.** Both `originalPreservedWhenHarvestApproves` and `denyHarvestsNothingButCompactionSucceeds` read the original session log's bytes before compaction and `assertArrayEquals` them after — proving the harvest writes to the memory store, never the session log, regardless of approval outcome. The deny test additionally proves the harvest never blocks the derive (compaction still succeeds), which is the propose-only contract AC-18.5/AC-21.4 require.

## Reuse / duplication

- **`LearningApprover` is a justified parallel to `permission.Approver`, not a duplicate.** A sweep surfaced `com.srk.codingagent.permission.Approver.confirm(GateRequest)` — the same approve/deny shape, but over a `GateRequest` (which carries `toolUseId`/command/file path). A proposed learning is not a tool call (no `toolUseId`), so `LearningApprover.approve(LearningProposal)` is a domain-appropriate parallel seam rather than a copy; forcing a learning through `GateRequest` would be primitive-shoehorning. The new interface's Javadoc explicitly documents this choice. Both reuse the shared `PermissionDecisionOutcome` enum (no parallel decision vocabulary). Not a finding.
- **`LearningProposal.presentation()` shares a name with `GateRequest.presentation()`, not a body.** Both render a one-line confirmation prompt, but for different value objects with entirely different content (tier/slug/why vs command/file/tool). Same naming convention across two prompt-bearing value objects — there is no shared body to extract. Not a finding.
- **The persistence/store/event path is composed, not re-implemented.** `LearningProposer` reuses `MemoryStore.write` (T-2.4) + `EventLog.append` + the existing `MemoryWritePayload` (T-2.4) + the `new Event(log.nextSeq(), clock.get(), payload)` idiom `WriteMemoryTool`/`AgentLoop`/`Compactor` use. No parallel serialization.
- **`requireNonBlank`** follows the codebase-wide per-class convention (`Payloads.requireNonBlank` is package-private to `persistence`, so each value object re-declares it — as `MemoryEntry`/`GateRequest`/`CompactionRequest`/`MemoryStore` already do). `LearningProposer`/`LearningProposal` follow that convention. Not a finding.
- **`LearningHarvester.NONE`** mirrors the established seam-default idiom (`BudgetGuard.NONE` per the Compactor's own Javadoc) — an injected default no-op rather than a `null` the Compactor must guard. Consistent with the codebase, not a duplication.
