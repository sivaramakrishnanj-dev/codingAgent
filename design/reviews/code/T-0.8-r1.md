## Summary

T-0.8 delivers the Agent Loop (C2, ADR-0001) — the system's heartbeat — as a new `com.srk.codingagent.loop` package of three production classes (`AgentLoop`, `BudgetGuard`, `LoopOutcome`) plus one accessor (`ToolRegistry.operationClass(String)`), no new dependency. `AgentLoop.run(String)` drives the Converse client-side tool-use cycle of state machine A: it appends a `USER_MESSAGE` (T1), then loops — `ModelClient.converse` (S1), append `MODEL_RESPONSE` + `MODEL_USAGE` (T2/T3), then dispatch on `StopReason`. On `tool_use` (T2→S2) it logs a `TOOL_USE` digest, routes a `GateRequest` through the `PermissionGate` and logs the `PERMISSION_DECISION` (T6/T7/T8), and only on approve dispatches the tool via `ToolRegistry.dispatch` and logs the `TOOL_RESULT` (T9); a denial logs a `TOOL_RESULT(denied)` and runs no handler (T8, CT-SM-2). All tool-result blocks for the turn are batched into one user message and the loop re-calls (T10), iterating until `end_turn` (T3→S5), which returns the final assistant text. `stop_sequence` is treated as `end_turn` (§ 3.1). The edge reasons (`max_tokens`, `model_context_window_exceeded`, `guardrail_intervened`, `content_filtered`, `malformed_*`) and a `BudgetGuard.COMPACT` signal (T13) are surfaced as a `LoopOutcome.SURFACED` value carrying the terminal stop reason — the compaction body (S6/machine B), the bounded repair-retry (T5), and exit-code dispatch (S8) are deliberately NOT built here, matching the task's scope note; the loop leaves clean seams (`BudgetGuard`, `LoopOutcome.SURFACED`). Log-before-act (INV-2, CT-INV-2) holds because every event flows through `EventLog.append`, which flushes per event (T-0.4) before the loop acts on its consequence; the per-cycle event order emitted matches the contract fixture `session-tool-use-cycle.jsonl` exactly (minus `SESSION_START`/`OUTCOME`, not this task's lane). Gate-in-the-middle (INV-8) holds because the loop is the only caller of `ToolRegistry.dispatch` and calls it only after `GateDecision.approved()`; an unknown/coarse tool fails closed to a side-effecting `GateRequest`. Pairing (INV-6) holds because every `TOOL_RESULT` event and every resent result block carries the answering `toolUseId`. ADR-0005 holds: the loop never calls `Instant.now()` — every event's timestamp comes from the injected `clock` supplier. `mvn clean verify` is green: 380 unit tests (27 new — 21 `AgentLoopTest`, 4 `LoopOutcomeTest`, 2 added `ToolRegistryTest`), 0 failures/errors/skipped; the `loop` package is **100% line / 89.7% branch**; JaCoCo "All coverage checks have been met" (0.80 floor), **no new exclusion**. One Minor (an impl-shaped `\n` join literal in a spec-silent multi-text-block render test), one Nit (fully-qualified `Optional`/`OperationClass` in the new `ToolRegistry.operationClass`), one Discussion (the loop folds the spec's `max_tokens` "continue-or-surface" and the `malformed_*` "repair-retry" branches into a single surface, which the task scope explicitly defers — surfaced for traceability, not a defect). No Blockers, no Majors.

### Minors

**[m1] [Spec-Adherence] `loop/AgentLoopTest.java:endTurnJoinsMultipleTextBlocks` — the `\n` join separator is an impl-derived literal on a spec-silent formatting detail**
- Why: § 3.1 / state-machine A T3 pin "Return final text" but do not pin how multiple text blocks of one `end_turn` turn are concatenated. The test asserts `outcome.finalText()` equals `"line one\nline two"`, and that `\n` separator comes from `AgentLoop.finalText()` (`text.append('\n')`), not from a spec symbol. The test's core intent — that *both* text blocks render, not just the first — is spec-grounded (§ 3.1 "render the final text"); only the separator literal is impl-shaped. This is a Minor (not a Major) because the spec is genuinely silent on the join character and the multi-block-render behaviour the test exercises is itself a thin-spec edge; the impl-derived part is a formatting nicety, not the asserted contract.
- Suggestion: either narrow the oracle to a shape assertion tied to § 3.1 (`assertThat(finalText).contains("line one").contains("line two")`), or keep the literal and record the newline join as a stated assumption (done — see Self-discipline section). Non-blocking; the join is reversible and documented in `AgentLoop`'s Javadoc.

### Nits

### Discussion

**[D1] [Discussion] § 3.1 distinguishes `max_tokens` ("Continue generation (re-call) or surface if a hard cap") and `malformed_*` ("Repair-retry up to a small bound, then surface") from a plain surface; the loop folds all edge reasons into one `LoopOutcome.SURFACED`**
- Why: the § 3.1 table and state-machine A T4/T5 give two edge reasons richer semantics than "surface": `max_tokens` may continue (re-call) unless a hard cap is hit, and `malformed_tool_use`/`malformed_model_output` get a bounded repair-retry before surfacing. The loop's `default ->` arm surfaces all of them uniformly. This is **correct for T-0.8**: the task's scope note explicitly defers the bounded repair-retry (CT-SM-3), the compaction body (CT-SM-6/7), and exit-code dispatch (S8) to later tasks, and instructs the loop to "surface the edge reasons via `LoopOutcome.SURFACED` and leave clean seams." The `LoopOutcome.SURFACED` value carries the precise `stopReason` so the later caller (T-0.9) can branch. Surfaced only so a future reader does not mistake the uniform surface for a gap — the differentiated handling is a named follow-on, not missing work.
- Suggested amendment kind: none (no spec change). The state-machine and § 3.1 already pin the follow-on tasks (T5 repair-retry, T4 compaction). A later task that builds repair-retry will replace the `default ->` arm's malformed branch; the seam (`LoopOutcome.SURFACED` + the terminal `stopReason`) is in place. No code change needed at T-0.8.

### Praise

**[P1]** The loop is pure composition — it reuses rather than re-declares. It drives the four collaborators it is handed (`ModelClient` T-0.5, `ToolRegistry` T-0.6, `PermissionGate` T-0.7, `EventLog` T-0.4) and builds every event from existing payload types (`UserMessagePayload`, `ModelResponsePayload`, `ModelUsagePayload`, `ToolUsePayload`, `ToolResultPayload`/`ToolResultStatus`, `PermissionDecisionPayload` via `GateDecision.toPayload`). It does not re-classify tools: the new `ToolRegistry.operationClass` accessor lets the loop build the right `GateRequest` from the registry's single source of truth (the handler's own `operationClass()`), with no parallel classifier — and it fails closed (unknown/coarse tool → side-effecting `GateRequest`) so an unregistered name cannot slip past the gate, while still routing to `dispatch` for the structured-error result. Two invariants are enforced *structurally* rather than by hope: gate-in-the-middle (INV-8) is guaranteed because `tools.dispatch` is called only inside the `decision.approved()` branch, and log-before-act (INV-2) rides on `EventLog.append`'s per-event flush, which the two probe tests verify *observably* (a tool that reads the log at the instant it runs sees its `MODEL_RESPONSE` and `PERMISSION_DECISION` already on disk). The event-order test asserts against the authoritative contract fixture, and the model-facing tests assert against the captured `ConverseRequest` (real wire shape via the real `ConverseWireMapper`), not impl state.

## Reuse / duplication

Reuse self-check: **passed.** The loop composes the four collaborators named in the task (no parallel reimplementation):
1. **`ModelClient.converse`** (T-0.5) is the only path to Bedrock — the loop builds no `ConverseRequest` itself and owns no wire mapping (verified: the loop passes domain `ConverseMessage`s and reads `Turn.response()`/`Turn.usage()`).
2. **`ToolRegistry`** (T-0.6) renders the `toolConfig` (`tools.toToolConfiguration()`), dispatches (`tools.dispatch`), and classifies (`tools.operationClass`) — the loop adds no tool registry, dispatch table, or classifier of its own. The new `operationClass(String)` accessor is the single new public surface, and it reads the handler's existing `operationClass()` rather than inventing a class map.
3. **`PermissionGate.evaluate`** (T-0.7) is the sole authorizer; the loop builds `GateRequest` via the existing `forCommand`/`forWrite`/`forTool` factories and turns the `GateDecision` into a `PERMISSION_DECISION` event via the existing `GateDecision.toPayload` — no new permission logic, no parallel mode/class enum.
4. **`EventLog.append`** (T-0.4) is the sole persistence path; the loop's private `append(EventPayload)` helper is a one-line convenience over `log.append(new Event(log.nextSeq(), clock.get(), payload))` — it adds no buffering, flushing, or sequencing (those stay `EventLog`'s responsibility).

Active-grep for a pre-existing loop / stopReason-dispatch / tool-use cycle across `src/main/java` (outside `loop/`) found none: `ModelClient` Javadoc explicitly defers the loop to T-0.8, `ToolRegistry` Javadoc defers the loop to T-0.8, and `PermissionGate` Javadoc defers dispatch + event emission to T-0.8 — the three collaborators were each authored expecting this loop to compose them, and it does. No duplicate `StopReason` switch, no parallel `ContentBlock` walk, no second timestamp source (the loop reuses the injected `clock` supplier, ADR-0005). The denied-result content-block status maps the event-level `ToolResultStatus.DENIED` to the narrower content-block `error` wire status — reusing both existing enums rather than minting a third, and documented inline.

## Design-doc alignment

- **ADR-0001 (owned loop):** "build ConverseRequest → call → parse `output.message.content[]` → on `stopReason == tool_use`, route each `toolUse` through the permission gate and tool registry → append `toolResult` blocks → re-call → until `end_turn`" → realized cell-for-cell in `AgentLoop.drive` + `dispatchTools` + `handleToolUse`. `stopReason` is the loop's state selector (the `switch` on `response.stopReason()`). The default model id is injected (resolved-at-runtime per NFR-MODEL-DEFAULT), not hardcoded in the loop.
- **02-architecture.md § 2 (the two load-bearing properties):** (1) **Log-before-act** — every assistant block and tool result is appended before the next step; `EventLog.append` flushes per event, verified by `eventsAppendedInLogBeforeActOrder` + the two probe tests. (2) **Gate-in-the-middle** — the gate sits between parsing a `toolUse` and executing it; the loop calls `tools.dispatch` only inside `decision.approved()`. Read-class tools flow through the gate and auto-approve (AC-9.6, via `PermissionGate`). The § 2 sequence (append user msg → loop: budget check → Converse → append assistant blocks + usage → per toolUse gate→exec/deny→append result → batch as one user msg → re-loop; end_turn renders) matches `run`→`drive` step-for-step.
- **§ 3.1 stopReason table:** `end_turn` → return final text (T3); `tool_use` → dispatch → append → re-loop (T2→T10); `stop_sequence` → treat as end_turn; `max_tokens` / `model_context_window_exceeded` / `guardrail_intervened` / `content_filtered` / `malformed_*` → surfaced (the differentiated continue/repair-retry handling is the deferred follow-on — see [D1]).
- **state-machine A transitions:** T1 (`run` appends USER_MESSAGE, seeds transcript), T2/T3 (append MODEL_RESPONSE+MODEL_USAGE), T6 (loop over toolUse blocks), T7 (approve → dispatch + PERMISSION_DECISION + TOOL_RESULT), T8 (deny → PERMISSION_DECISION(deny) + TOOL_RESULT(denied), no handler), T9 (handler returns incl. error → TOOL_RESULT), T10 (batch results as one user msg, re-call), T13 (budget seam consulted after each turn) — all realized. T4/T5 surface via `LoopOutcome.SURFACED`.
- **INV-2 / INV-6 / INV-8:** INV-2 (every side-effecting step's event flushed before the effect) — `EventLog.append` flushes per event; verified observably. INV-6 (every TOOL_RESULT.toolUseId matches the toolUse) — both the event and the resent block carry `toolUse.toolUseId()`; verified by `toolResultIsSentBackOnRecall` / `deniedResultSentBackToModel`. INV-8 (no path reaches execute without a PERMISSION_DECISION) — `dispatch` is called only after `decision.approved()`, and the decision is always logged first; verified by `permissionDecisionLoggedBeforeHandlerRuns`.
- **Contract tests (milestone scope):** **CT-SM-1** (`toolUseCycleDrivesToEndTurn` + `multipleSequentialCycles` + `multipleToolUseBlocksInOneTurn`), **CT-SM-2** (`gateDenyLogsDeniedResultAndRunsNoHandler` + `deniedResultSentBackToModel`), **CT-INV-2** (`eventsAppendedInLogBeforeActOrder` + the two ordering probes) — all satisfied with passing test methods. CT-SM-3 (repair-retry), CT-SM-4 (SIGINT/T18), CT-SM-6/7 (compaction) are out of this task's scope per the task's contract-test note and are NOT claimed here.
- **Component C2 (02-architecture.md § 1.2):** "Drive the Converse cycle: send → parse blocks → dispatch tools → append results → repeat on stopReason; one in-flight model call per conversation; every block logged before acting" — all realized. The loop is single-threaded (one in-flight call per conversation), documented not-thread-safe. Package `com.srk.codingagent.loop` per § 6.
