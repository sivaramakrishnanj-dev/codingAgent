## Summary

T-4.1 delivers component C11 (ADR-0008): the `web_search`/`web_fetch` Class-X tools, a swappable
`WebLookupBackend` seam, and the v1 implementation `ClaudeCliWebLookupBackend` that shells out to a
constrained headless Claude CLI (`<program> -p '<task>' --output-format text`) via the **reused**
`CommandExecutor` subprocess machinery (ADR-0003), rooted at a **scratch** temp dir (not the workspace,
ADR-0008) with a hard 120 s timeout (NFR-NET-WEBLOOKUP-TIMEOUT). Both tools are registered in the live
parent registry (`ToolRegistryComposer.parentRegistry()`) so they are reachable at runtime (the T-2.7
lesson), and both declare `OperationClass.SIDE_EFFECTING`, so the existing permission gate denies them in
`READ_ONLY` (AC-11.2, RD-6) with no new gate path. Failure (delegate absent on PATH / non-zero exit /
empty output / timeout) is returned as a `WebLookupResult.failure` the tools surface to the model as
text (AC-11.3 — report, never fabricate). Invocations and results are recorded as events by the existing
loop `TOOL_USE`/`TOOL_RESULT` path (AC-11.4) — no parallel logging path was added.

### Nits

**[N1] [Readability] `ToolRegistryComposer.java` — `WEB_LOOKUP_TIMEOUT` and `webScratchDir` use
fully-qualified `java.time.Duration` / `java.nio.file.Files` inline rather than imports.** The composer
already imports many types; the inline FQNs in the new web-lookup wiring are slightly noisier than an
import. Author's call — kept inline to keep the diff localized to the web-wiring block and avoid touching
the import list ordering. Not worth churn.
- Why: Clean Code (consistency) — purely stylistic, no principle is violated.
- Suggestion: optionally add `import java.time.Duration;` if/when the composer grows more time-based wiring.

### Praise

**[P1] Reuse — subprocess management is delegated, not duplicated.** `ClaudeCliWebLookupBackend` spawns
the delegate through the existing `CommandExecutor` (ADR-0003) and reads the structured `CommandResult`
(exit code / stdout / timedOut), rather than calling `ProcessBuilder` itself. This honours the task's
"reuse CommandExecutor, do not duplicate process management" directive and means the tree-kill timeout
and stream-drain safety come for free.

**[P2] Spec-Adherence — gating is structural, via the existing seam.** The web tools declare
`SIDE_EFFECTING` and route through `GateRequest.forTool(...)`, so READ_ONLY denial (AC-11.2/RD-6) is
enforced by the unchanged `PermissionGate` Class-X branch — no parallel gate path, no risk of a
web-specific gate drifting from the general one. (`PermissionGateTest` already referenced `web_search` as
a generic Class-X tool before this task; that assumption is now realized.)

**[P3] Spec-Adherence — failure is a value, the agent reports it (AC-11.3).** Absent-on-PATH (exit 127),
non-zero exit, empty output, and timeout each map to a `WebLookupResult.failure` carrying a report the
tools pass through as text — the agent never fabricates and never crashes. The backend throws only for a
true programming error (`null` request), keeping the AC-11.3 path distinct from a caller bug.

## Reuse / duplication

Active sweep across `src/main/java/com/srk/codingagent/tool/**`:
- **No duplicated process management** — `ProcessBuilder`/`Runtime.exec` appear in no new class; the
  backend uses `CommandExecutor`.
- The `singleQuote` shell-escape helper is unique to `ClaudeCliWebLookupBackend`; no prior twin exists
  (`RunCommandTool` passes the command opaque and never quotes a sub-argument, so there is nothing to
  extract or share). The web backend genuinely needs to embed the lookup task as one shell-quoted
  argument — a new, distinct need, not a near-duplicate.
- The two tools (`WebSearchTool`/`WebFetchTool`) share shape with each other but differ in tool name,
  input field (`query` vs `url`), and request kind (SEARCH vs FETCH). They mirror the established
  per-tool class convention (one `ToolHandler` per tool, like `ReadFileTool`/`WriteFileTool`); collapsing
  them into one parameterized class would fight that convention and obscure the two distinct 04-apis § 3
  registry rows. No extraction warranted.

`reuse_self_check: passed.`

## Inherited-rubric sweep (Layer A)

- Logging: SLF4J with `{}` placeholders and FQ-class loggers in the backend and both tools; INFO on the
  external-boundary delegate spawn, WARN on each failure branch (shared.md § 3). No `System.out`/`err`.
- Exception discipline: backend catches only the specific `CommandExecutionException` (not
  `Exception`/`Throwable`), logs with action, and returns a failure value; no swallowed exceptions.
  Value types validate in compact constructors (EJ Item 17 defensive invariants).
- API design: every new public class/method has Javadoc with `@param`/`@return`/`@throws`. The backend's
  test-only program-name constructor is package-private (minimised accessibility, EJ Item 15).
- No `equals`/`hashCode` issues (records), no mutable static state, no `Thread.sleep` in production, no
  raw `RuntimeException` thrown where a typed one fits, no magic numbers (127 is a named constant,
  120 s a named `Duration`).
- Testing tenet: SUT not mocked — the tool tests use a real tool over a stub *backend* (a true external
  collaborator, the swappable seam), and the backend test uses the real backend over a real
  `CommandExecutor` with only the delegate program test-controlled. No test asserts only on mocks.
