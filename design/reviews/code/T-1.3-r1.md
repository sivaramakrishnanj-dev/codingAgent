### Minors

**[m1] [Reuse] `GlobTool.java:127` / `GrepTool.java:176` — duplicated glob-matcher construction idiom**
- Why: Both tools build a `FileSystems.getDefault().getPathMatcher("glob:" + p)` inside a try/catch that wraps an invalid-glob `IllegalArgumentException` as a `ToolInvocationException` (~90% token overlap; the only difference is the field name in the message). DRY (Fowler) — a small shared helper (e.g. `ToolInputs.requireGlob(...)` or a `Globs.matcher(...)`) would remove the twin. Severity is Minor, not Major: it is a 3-line idiomatic block, not duplicated business logic, and the two sites differ in their diagnostic. The load-bearing reuse targets (`WorkspacePaths`, `ToolInputs`, `ToolSchemas`) were correctly reused; no pre-existing glob helper was duplicated (confirmed by sweep — these are the only two `getPathMatcher` sites in `src/main`).
- Suggestion: Extract a package-private `glob:`-matcher builder in `ToolInputs` (the established tool-input-mechanics home) and call it from both tools. Deferred as a Minor — does not block resolution.

### Nits
None.

### Praise

**[P1] EditFileTool unique-substring contract is the right D2-defense.** edit_file refuses a zero-match (`no match`) and a multi-match (`ambiguous`) edit rather than silently no-op'ing or guessing (AC-5.4 spirit), and the tests assert the file is left untouched on both — exactly the live-only defect class a mocked test would miss. Targeted-edit (substring splice) vs whole-file replace (`write_file`) is cleanly separated.

- **AC-4.3** (report, don't fabricate): missing/non-directory/non-file targets throw `ToolInvocationException` (error result) in all four tools. ✓
- **AC-4.4** (read/grep/glob/list non-gated): grep/glob/list declare `OperationClass.READ`; the gate's `gateRequestFor` routes a Class-R tool through `GateRequest.forTool` which auto-approves. Verified against `AgentLoop.gateRequestFor` (no gate change needed). ✓
