## Summary

T-1.6 delivers the brownfield "understand→change" workflow driver (C3, ADR-0012 brownfield side, US-4/US-5) as a standalone, fully unit-tested unit in the brand-new `com.srk.codingagent.workflow` package, and wires it into the actual `codingagent -p` / REPL run path so the live G1 explore→edit→verify→resume smoke test exercises real code. Four new production types: `BrownfieldPlaybook` (the tested system-prompt artifact that primes the model to explore-before-edit / report-not-fabricate / ask-when-ambiguous / verify-after-change), `BrownfieldDriver` (the orchestration: run the understand→change turn through the loop, then on completion invoke the configured `VerifyLoop` with a `RemedyAttempt` that drives another loop turn fed the failure — the AC-20.3 wiring T-1.4 deferred), `BrownfieldOutcome` (the terminal disposition: VERIFIED / VERIFY_EXHAUSTED / NOT_VERIFIED), and `BrownfieldRunner` (the adapter mapping a `BrownfieldOutcome` back to a `LoopOutcome` so the run path's exit-code mapping is unchanged). The two wiring edits — `AgentLoopFactory` (passes `BrownfieldPlaybook.systemPrompt()` to the loop's `system` arg instead of `null`) and `Main` (routes one-shot + REPL through `BrownfieldDriver.overConfig(...)`) — are both in the JaCoCo-excluded composition shell, with all driver logic in the tested workflow unit. `mvn clean verify` is green (619 tests, +32 new; JaCoCo gate met). No Blockers, no Majors, no Minors, no Nits. One Discussion item (the spec is silent on the agent-process exit code for a verify-exhausted brownfield run). The multi-turn-continuation question is resolved as single-run-with-tools, no `AgentLoop` change.

### Praise

**[P1] The orchestration contract is asserted, not "the driver called run()".** `BrownfieldDriverTest` proves the actual arc: a *completed* change-turn triggers verification (AC-5.3) while a *surfaced* one does not (the verifier factory would throw if built — `surfacedChangeTurnDoesNotVerify`); an EXHAUSTED verify outcome surfaces VERIFY_EXHAUSTED carrying the *last* failure's `stderr` (AC-20.5), not a clean success; NO_TEST_COMMAND is handled (AC-20.6). This is the structurally-blind-test defense the G1 gate demands.

**[P2] The deferred AC-20.3 wiring is realized as an observable second loop turn.** `remedyDrivesAnotherLoopTurnWithFailure` captures the `RemedyAttempt` the driver hands the verify loop, invokes it as the verify loop would, and asserts a *second* loop turn ran with the failure's `AssertionError` text in its prompt — the load-bearing "model reacts to the failure" wiring, proven to carry the real output. `overConfigBoundsRemedyTurnsByConfig` then drives the **real** `VerifyLoop` with an always-failing `false` command at bound 3 and asserts exactly 3 loop runs (one change-turn + two remedy turns, no 4th), proving the fix-and-retry cycle is bounded by `NFR-VERIFY-MAX-ITERATIONS`, not unbounded.

**[P3] The playbook is a tested artifact, not a string literal in the excluded factory.** The explore-before-edit / verify-after-change instructions live in `BrownfieldPlaybook` (covered), and `BrownfieldWiringTest` confirms they actually reach the model on the Converse call through a real loop — so a regression that silently dropped the playbook from the wire would fail a test, which is the thing the G1 gate's correctness rests on.

## Reuse / duplication

- **No pre-existing driver / playbook / orchestration / system-prompt-content helper to merge with.** Sweep across `src/main` for `Driver` / `Playbook` / `orchestrat` / `Workflow` / `systemPrompt` / `RemedyPrompt` / "You are a coding agent" / "brownfield mode" / "explore...before" returns only (a) the new `workflow` package, (b) the `system`-prompt *parameter* docs on `ModelClient` / `ConverseWireMapper` (the wire seam the playbook flows through — expected, not duplication), and (c) the `AgentLoopFactory` / `Main` wiring edits this task makes. The `workflow` package is brand new; this is the first workflow-driver logic in the codebase, exactly as 02-architecture § 6 places C3.
- **`BrownfieldDriver.overConfig` mirrors the established config-wired-control idiom, not duplicates it.** It follows `VerifyLoop.forConfig` / `OutputDisposer.forConfig` (config → wired control unit) and deliberately *delegates to* `VerifyLoop.forConfig` for the verify half rather than re-implementing the run→check→retry loop. Named `overConfig` (not `forConfig`) only to avoid a third identically-named static when the two compose; same idiom, distinct concern.
- **The `LoopTurn` seam mirrors `OneShotRunner.OneShotLoop` / `ReplRunner.ReplLoop`** (all three are the `LoopOutcome run(String)` shape) — an intentional, established testability seam, not copied logic; the driver consumes the same shape so `Main` can hand it `loop::run` exactly as the runners do.
- **`RemedyPrompt.forFailure` is the first failure-feedback prompt builder** — no existing prompt-construction utility to reuse (the only other prompt content, the playbook, is a static system prompt; this is a per-failure user-turn prompt). It is small and single-purpose, not a duplicate-able utility.

## Design-doc alignment

- **AC-4.1 / AC-5.1** (explore/locate before editing): the playbook instructs using the read-only tools (`read_file`/`grep`/`glob`/`list`) before editing and locating files by searching, asserted in `BrownfieldPlaybookTest.instructsExploreBeforeEdit` / `instructsLocateBySearch`; the model drives this within one `AgentLoop.run` turn (the loop cycles tool-use until end_turn). ✓
- **AC-4.3** (report a missing file, don't fabricate): the playbook forbids inventing contents and instructs reporting a non-existent file / tool error; the driver does not paper over a tool error (it surfaces a non-completed turn rather than declaring success). `instructsReportNotFabricate`. ✓
- **AC-5.2** (edit subject to the active permission mode): respected by *not* bypassing the gate — the driver never calls tools directly; the model's `edit_file`/`write_file` calls route through the loop's gate (the C2 invariant the driver inherits). No driver code touches the gate, so the gate's enforcement is intact. ✓
- **AC-5.3** (verify a completed change via the configured build/test command): `BrownfieldDriver.run` invokes the verify step only on a *completed* change-turn (`completedChangeTurnTriggersVerification`), through `VerifyLoop.forConfig` on `config.commands().test()` (`overConfigVerifiesWithPassingCommand` exercises the real verify loop). ✓
- **AC-5.4** (ask a clarifying question when ambiguous): the playbook instructs asking rather than guessing on a multi-target ambiguity (`instructsAskWhenAmbiguous`) — the loop-level clarification AC is model behaviour the prompt primes, distinct from `edit_file`'s mechanical ambiguity error. ✓
- **AC-20.3** (feed the failure back and attempt a remedy): the driver's `RemedyAttempt` drives another loop turn whose prompt carries the failure output (`remedyDrivesAnotherLoopTurnWithFailure`, `remedyPromptCarriesFailureOutput`). ✓
- **AC-20.5** (stop and surface with the relevant output): a VERIFY_EXHAUSTED disposition carries the last failing run's output, and `BrownfieldRunner` folds it into the surfaced user-facing text (`verifyExhaustedMapsToCompletedWithSurfacedOutput`). ✓
- **AC-20.6** (no test command → nothing to verify): mapped to NOT_VERIFIED carrying NO_TEST_COMMAND (`noTestCommandYieldsNotVerified`, `overConfigNoTestCommandNotVerified`). ✓
- **NFR-VERIFY-MAX-ITERATIONS** (the fix-and-retry cycle is bounded by the config value): bound is `config.verifyMaxIterations()` via `VerifyLoop.forConfig`; `overConfigBoundsRemedyTurnsByConfig` proves exactly N attempts / N-1 remedy turns at bound 3, not unbounded. ✓
- **ADR-0012** (orchestration on top of the loop, not a separate engine; brownfield is the simpler sibling — no phase machinery): the driver adds exactly one explicit orchestration step (verify-after-complete) on top of the model-driven loop; no requirements/design/tasks phases, no approval gates, no hard-coded state machine. Greenfield phase machinery is left to M3 (T-3.1/3.2/3.3). ✓
- **02-architecture § 1.2 C3 + § 6** ("Mode is fixed for a session"; C3 lives in `com.srk.codingagent.workflow`): the driver runs a brownfield session for its whole life (no mid-session mode switch); the package is the one § 6 names. ✓
- **cli-exit-codes G4** (agent-process exit codes are distinct from the build/test verification signal): a verify-exhausted change maps to a *completed* outcome (the agent did its work — exit 0) whose text surfaces the failure (AC-20.5), rather than being masked as an internal-error exit. See Discussion D1. ✓
- **Scope discipline / no live AWS** — the driver is pure orchestration over scripted/real seams; unit tests use a scripted Bedrock double + scripted/real `CommandExecutor` (a trivial `true`/`false`), never a live Bedrock call. `aws_consulted` is empty. The real-Bedrock G1 smoke test is the main agent's next step. ✓

## Discussion

**[D1] [Discussion] The exit-code contract is silent on the agent-process code for a verify-exhausted brownfield run.** `cli-exit-codes.md` § 1 pins 0/1/2/3/4/5/130; none names "the agent ran fine but the change did not pass tests." G4 explicitly separates the build/test verification signal from the agent-process exit code, and § 1 says "New categories, if ever needed, take new unused numbers." I resolved this by treating a VERIFY_EXHAUSTED run as a *completed* agent process (exit 0) that surfaces the verification failure with its output in the user-facing text (honouring G4 + AC-20.5), rather than inventing a non-contract code or mapping it to `1 INTERNAL` (which is "a bug or environment fault not otherwise classified" — a verify failure is neither). This is the spec-correct reading given the contract's silence and G4's separation, but a future amendment may want to pin a dedicated disposition for scripting use (G3: one-shot returns codes so CI can branch — a CI that runs `codingagent -p "make X pass"` cannot today distinguish "verified" from "change made, still failing" by exit code alone). **Suggested amendment kind:** `exit-code-update` (add a verify-not-passed code) or `ac-update` (pin the one-shot disposition for a verify-exhausted change). Not built — surfaced for the designer. Does not block resolution.
