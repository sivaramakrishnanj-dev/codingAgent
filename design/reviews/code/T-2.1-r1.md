## Summary

T-2.1 swaps the loop's no-op `BudgetGuard.NONE` for a real guard that fires the auto-compaction trigger (state machine A T13 / machine B LT2; the T-2.1 half of CT-SM-6) when a model turn's **measured** `usage.inputTokens` reaches `NFR-CONTEXT-COMPACT-THRESHOLD (0.85) × contextWindowTokens`. Two new production types: `ModelCapabilityProfile` (component C5, `com.srk.codingagent.model`, ADR-0002) — an intentionally **thin** v1 profile carrying only `contextWindowTokens`, resolved via a static registry keyed by model-id prefix (the live default `us.anthropic.claude-opus-4-8` and the bare/Sonnet/Haiku forms resolve to the real Claude window through the `anthropic.claude` infix; an unknown id degrades to a conservative safe-minimum window per ADR-0002); and `TokenBudgetGuard` (component C6 arithmetic, `com.srk.codingagent.loop`, ADR-0006) — the `BudgetGuard` implementation that computes the absolute trigger `ceil(0.85 × window)` once and returns `COMPACT` iff measured `inputTokens >= ` that trigger. The wiring edit in `AgentLoopFactory` (JaCoCo-excluded composition root) builds the profile for the active model and constructs `TokenBudgetGuard.forConfig(config, profile)` in place of `BudgetGuard.NONE`; the `AgentLoop`'s existing surface-on-COMPACT behaviour is left intact (the real S6→machine-B handler is T-2.2). `mvn clean verify` is green (643 tests, +24 new; JaCoCo gate met). No Blockers, no Majors. One Minor (a constant left package-private that should be `private`), one Nit (fully-qualified `java.util.Objects` where the package convention varies), two Discussion items (the Claude/safe-minimum window values are not spec-pinned; the conservative-default-window "read from config" mechanism is deferred to T-4.3).

## Scope boundary held (noted explicitly per the task directive)

T-2.1 delivers **only the LT2 trigger-condition detection** — the boolean `usage.inputTokens >= 0.85 × window`. It does **not** implement: the compaction body (summary Converse call, derived session, original-preserved invariant, learning harvest — T-2.2/T-2.5), the S6→machine-B handler (when the guard returns `COMPACT`, `AgentLoop` still *surfaces* it — that surface is correct for T-2.1 and is what T-2.2 will replace), or the full `ModelCapabilityProfile` registry + feature detection + schema validation (T-4.3, CT-SCH-15/NFR-MODEL-PROVIDER/OQ-J). No new config key was added (so `resolved-config.schema.json` / CT-SCH-13/14 are untouched); the conservative-default safe-minimum window is a compiled-in constant in the composition root, documented for T-4.3 to source from config `NFR-MODEL-CONTEXT-WINDOW` (see D2). The profile is kept forward-compatible: T-4.3 can add `providerFamily`, `supportsExtendedThinking`, `promptCache`, etc. (03-data-model § 2.6) without reworking the callers that already read the window through this seam.

### Minors

**[m1] [Readability] `ModelCapabilityProfile.java:51` — `CLAUDE_4_FAMILY_CONTEXT_WINDOW_TOKENS` is package-private but used only inside the class.**
- Why: Effective Java Item 15 (minimize accessibility). The constant was originally package-private to let the test reference it, but the test was revised to assert the spec property (`!= fallback`, `> 0`) rather than the literal window — so the broader visibility is now leftover, not load-bearing. A `grep` confirms it is referenced only at `ModelCapabilityProfile.java:94`.
- Suggestion: change `static final int` to `private static final int`. Does not block resolution (Minor); a one-keyword change for a fresh round.

### Nits

**[N1] [Style] `TokenBudgetGuard.java:88,107` — fully-qualified `java.util.Objects.requireNonNull` instead of an import.**
- Why: stylistic. The codebase is split: `ResolvedConfig` uses the FQ form (precedent), while `AgentLoop` / `OutputDisposer` import `java.util.Objects`. Within `TokenBudgetGuard` the FQ form appears 3×, where an import would read cleaner.
- Suggestion: `import java.util.Objects;` and drop the qualifier. Author's call — there is codebase precedent both ways.

### Praise

**[P1] The threshold is asserted from the spec formula, computed independently — not copied from the impl.** `TokenBudgetGuardTest` derives every boundary from `0.85 × window` (170000 at window 200000, 85000 at 100000, 34000 at 40000), and the non-integer case `0.85 × 150001 = 127500.85` proves the `>=` semantics fire at the ceiling (127501 triggers, 127500 does not) — exactly the ADR-0006 "0.85×window is exact" contract, with the arithmetic verified against a standalone computation rather than read back from `TokenBudgetGuard`.

**[P2] The "measured input, not estimate" rule of ADR-0006 is enforced, not assumed.** `largeOutputSmallInput_continues` (output 9.9M, input one-below-threshold → CONTINUE) and `cacheFieldsIgnored_inputDrivesDecision` (cache fields 500K each, sub-threshold input → CONTINUE) prove the trigger keys on `usage.inputTokens` alone — so a regression that summed output or cache tokens into the budget would fail a test.

**[P3] No model branching in the guard (ADR-0002).** `TokenBudgetGuard` never inspects the model id; the window arrives pre-resolved from the profile, and `WindowDrivenThreshold` proves the absolute trigger scales with whatever window is passed (40000 → 34000, etc.) rather than a hardcoded count — the feature-detection-over-`if (modelId.contains("claude"))` discipline ADR-0002 mandates.

## Discussion

**[D1] [Discussion] The concrete context-window figures (Claude 200K, conservative default 100K) are not pinned by any spec symbol.** 03-data-model § 2.6 types `contextWindowTokens` as `int` and ADR-0002 says the default profile uses "a safe minimum context window," but no symbol pins the Claude family window or the safe-minimum value. I chose 200000 (the published Claude 4-family input window) for the registry entry and 100000 (a deliberately low safe-minimum, so the threshold fires conservatively rather than risking overflow on an unvalidated model) for the fallback. These are forward-compatible and reversible. **Suggested amendment kind:** `nfr-update` or `data-model-update` if the team wants the window figures pinned (e.g. a per-family table in 03-data-model § 2.6 or an NFR). Not built — surfaced for the designer. Does not block resolution.

**[D2] [Discussion] ADR-0002 says the conservative-default window is "read from config `NFR-MODEL-CONTEXT-WINDOW`," but T-2.1 carries it as a compiled-in constant.** ADR-0002 / 03-data-model § 2.6 both say the unknown-id default profile's window is "read from config." Adding a `contextWindowTokens` config key would require editing `06-formal/resolved-config.schema.json` (and the CT-SCH-13/14 contract + `ConfigKeys`/`ConfigDefaults`/`ResolvedConfig`/`ConfigResolver`) — which is a design-artifact change beyond T-2.1's "minimal viable shape" mandate and overlaps T-4.3's full-registry+schema-validation scope. I therefore kept the safe-minimum as a documented compiled-in constant in the JaCoCo-excluded composition root (`AgentLoopFactory.CONSERVATIVE_DEFAULT_CONTEXT_WINDOW_TOKENS`), with a Javadoc note that T-4.3 sources it from config. The live default model resolves to the real Claude window through the registry, so the fallback path is not on the production happy path today. **Suggested amendment kind:** `schema-update` (add the `contextWindowTokens` / `NFR-MODEL-CONTEXT-WINDOW` config key) — most naturally folded into T-4.3 when the full capability registry + schema validation land. Not built — surfaced for the designer. Does not block resolution.
