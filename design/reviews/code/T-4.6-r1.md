## Summary

T-4.6 wires `NFR-BEDROCK-CALL-TIMEOUT` (DCR-4, ADR-0001) into the Bedrock client. Four things land,
each mirroring the established `commandTimeoutSeconds` config pattern exactly:

1. **Two new config keys + defaults.** `ConfigKeys` gains `BEDROCK_CALL_CONNECT_TIMEOUT_SECONDS`
   (`bedrockCallConnectTimeoutSeconds`) and `BEDROCK_CALL_RESPONSE_TIMEOUT_SECONDS`
   (`bedrockCallResponseTimeoutSeconds`), both added to `TOP_LEVEL`; `ConfigDefaults` gains
   `BEDROCK_CALL_CONNECT_TIMEOUT_SECONDS = 10` and `BEDROCK_CALL_RESPONSE_TIMEOUT_SECONDS = 300`
   (NFR-BEDROCK-CALL-TIMEOUT / AC-8.11).
2. **Two new `ResolvedConfig` components.** `int bedrockCallConnectTimeoutSeconds` and
   `int bedrockCallResponseTimeoutSeconds` appended after `commandTimeoutSeconds`, each validated
   `requireAtLeast(..., 1, ...)` in the compact constructor (schema `minimum 1`).
3. **Resolver binding.** `ConfigResolver.bindAndValidate` adds two `intInRange(...)` resolutions
   (defaults 10 / 300, range `[1, Integer.MAX_VALUE]`) and threads both into the positional
   `new ResolvedConfig(...)` call ‚Äî the JSON-Schema `default` keyword is documentary, so the resolver
   (C17, ADR-0009) is what applies the defaults when the keys are absent (CT-SCH-17).
4. **Client wiring.** `BedrockClientFactory.create(...)` and `wiring(...)` take the two timeout values;
   the `Wiring` record carries them as `Duration connectTimeout` / `Duration responseTimeout`. `create(...)`
   wires `apiCallTimeout` = response (via `overrideConfiguration`), and an Apache `httpClientBuilder`
   (`software.amazon.awssdk:apache-client`, BOM-managed in `pom.xml`) with `connectionTimeout` = connect
   and `socketTimeout` = response (ADR-0001). `AgentLoopFactory` (JaCoCo-excluded composition root) threads
   both budgets from `config` into `create(...)`.

**The CT-INV-13 SUT-not-mocked pattern is preserved.** The timeout-application decision rides on the pure,
inspectable `wiring(...)` seam; the wiring test asserts the configured connect/response `Duration`s by
inspection ‚Äî no SDK client is constructed, no live Bedrock/Converse call is made. `create(...)` reads
both durations from the wiring, so asserting the wiring durations pins the apiCall/socket = response and
connect = connect mapping.

**The load-bearing new tests.** `BedrockClientFactoryTest.nfr_bedrockCallTimeout_wiresDefaultBudget`
(connect 10 s / response 300 s default budget ‚Äî AC-8.11) + `nfr_bedrockCallTimeout_honoursConfiguredBudget`
(25 s / 600 s override binds ‚Äî AC-8.10) + two sub-1 rejections; `ConfigResolverTest$BedrockCallTimeouts`
with `ct_sch_17_bothAbsent_resolverAppliesDefaults` (CT-SCH-17), the two override-binds, and the two
out-of-range-rejected-naming-the-key cases (AC-8.5); `ResolvedConfigTest.ct_sch_16_bedrockTimeoutKeys_validate`
(CT-SCH-16 ‚Äî integers >= 1 validate at the record level) + the two min-1 rejections; the
`ConfigKeysTest`/`ConfigDefaultsTest` schema-mirror extensions.
