## Summary

T-3.8 reworks `GreenfieldImplementLoop` to the DCR-7 model: implement **every** task in breakdown
order and **mark each complete as it is implemented** (a durable on-disk `- [x] <taskId>` marker via
the reused `GreenfieldArtifactStore.appendLine`), with the **per-task verify cycle removed** — no
per-task `Verifier`/`VerifyLoop` construction, no per-task EXHAUSTED hard-stop, no per-task
`NO_TEST_COMMAND` early-return. `ImplementOutcome` collapses to two self-consistent dispositions
(`ALL_IMPLEMENTED`, `NO_TASKS`) and drops the per-task verify fields. `asLoopTurn()` still maps an
all-implemented (or no-tasks) run to a completed `LoopOutcome`, so the driver's `COMPLETED` mapping is
unregressed. The end-of-phase verify, the no-test-command terminal behaviour, the AC-20.6→AC-3.6
mis-citation fix, and intra-IMPLEMENT resume are correctly left for T-3.9/T-3.10; `VerifyLoop`/
`VerifyOutcome` are untouched. Full `mvn clean verify` is green (1107 tests, JaCoCo BUNDLE line 0.9106).

### Nits

**[N1] [Readability] `GreenfieldImplementLoop.java:64-77` — two collaborator fields are reserved for T-3.9, not read in T-3.8**
- Why: `verifyExecutor` / `verifyConfig` are stored by `overConfig` but never read in the T-3.8 loop
  body (per-task verify is dropped, DCR-7). This is deliberate seam-shaping per the task directive
  (item 5/6: "shape … so T-3.9 can plug the end-of-phase verify in … with minimal churn" and "flag
  rather than aggressively delete shared seams T-3.9 will reuse") — keeping `overConfig`'s signature
  stable means `ToolRegistryComposer` and `Main` wiring are untouched across T-3.8→T-3.9. No SpotBugs/
  Checkstyle gate exists, and the fields are assigned (covered) in the constructor. Recorded as a Nit,
  not a defect: it is intentional plumbing-for-the-next-task, clearly Javadoc'd as such.
- Suggestion: T-3.9 consumes both fields when it adds the end-of-phase verify after the per-task loop.
  If T-3.9 slips, revisit whether to drop them to avoid genuinely dead state.
