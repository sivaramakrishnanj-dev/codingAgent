## Summary

T-2.8-RD-D4 is a regression-of-T-2.8 fixing the fourth live-only Bedrock-wire defect in this class
(same family as D1/D2/D3). `AgentLoop.drive()` ordered its steps so that when `TokenBudgetGuard`
returned `COMPACT` on a `TOOL_USE` turn, it called `compaction.compact(transcript, ...)` while the
transcript still ended in a dangling `toolUse` block (the matching `toolResult` was not appended
until the `switch(stopReason)` ran *after* the COMPACT block). The summarizer replays that transcript
verbatim to Bedrock, which enforces tool_use/tool_result pairing (Â§ 6.A.1 / INV-6) and rejected the
live call with `ValidationException: "tool_use ids were found without tool_result blocks immediately
after ... Each tool_use block must have a corresponding tool_result block in the next message."` A
G2 real-Bedrock smoke test forced it deterministically (`contextCompactThreshold=0.001`); at the real
0.85 threshold the same break happens whenever measured usage crosses 85% on a tool_use turn.

The fix follows the user-chosen approach exactly: **on a `TOOL_USE` turn, complete the turn first â€”
dispatch the tool(s) and append the batched `toolResult` (via the existing `dispatchTools` helper) â€”
THEN evaluate the budget guard and invoke the compaction seam.** This guarantees the seam always
summarizes a well-formed, toolUse/toolResult-paired transcript at every stop reason. Concretely, the
only change is a reorder inside `drive()`'s `while(true)` body:

- After `append(response); append(turn.usage())` and adding the assistant turn to the transcript
  (INV-2 log-before-act, **unchanged** as the first action after `converse(...)`), a new guarded step
  runs `dispatchTools(...)` when `response.stopReason() == StopReason.TOOL_USE`, appending the
  `toolResult` and pairing the transcript before the budget seam.
- The budget seam (`budgetGuard.evaluate(...)` â†’ `compaction.compact(...)` â†’ continue/surface) is
  evaluated **after** that pairing, so it never sees a dangling `toolUse`.
- The `switch(stopReason)` is unchanged except that its `TOOL_USE` arm is now a no-op (the dispatch
  already happened above) that falls through to the re-call; `END_TURN`/`STOP_SEQUENCE` still return
  completed; edge reasons still surface.

The fix is confined to `AgentLoop.drive()`. No public API signature changed; `BudgetGuard`,
`CompactionSeam`, `Compactor`, `ModelClient`, `ToolRegistry`, and `OutputDisposer` are untouched.

`mvn clean verify` is green: **847 tests, 0 failures / 0 errors / 0 skipped; JaCoCo BUNDLE LINE 0.80
gate met ("All coverage checks have been met")**; BUNDLE LINE coverage 2960/3210 = 92.21%. The new
regression test (and the corrected NONE-wiring test) both fail against the pre-fix ordering and pass
after the fix â€” proven by a temporary revert (see Self-discipline section).
