## Summary

DEFECT D2: `ConverseWireMapper.toWireToolResult` unconditionally routed a tool result's
content into the Converse `toolResult.content[].json` member, but Converse requires the
`json` member to be a JSON object. A plain-text (String) tool result — e.g. the file
contents from `read_file` — must use the `text` member instead, otherwise Bedrock returns
`ValidationException: "The format of the value at ...toolResult.content.0.json is invalid.
Provide a json object for the field"`. The fix branches on the content's runtime shape:
a `Map` (structured object) → `json`; any other non-null content → `text`. Two regression
tests assert the text-vs-json member explicitly (the assertion the prior structurally-blind
tests omitted, which is why the bug shipped). Surgical: only `toWireToolResult` + its
class-Javadoc § 7 note and `ConverseWireMapperTest` were touched.

### Nits

**[N1] [Readability] `ConverseWireMapper.java:278` — non-object scalar/list content routed to `text` via `String.valueOf`**
- Why: The edge case (a bare `Number`/`Boolean`/`List` content) is not load-bearing for
  this regression — the spec's two sanctioned shapes are text and structured object
  (`content-block.schema.json`). Routing such content to `text` via `String.valueOf` is a
  defensible conservative choice (Converse `json` rejects non-objects, including lists),
  and it is documented in the handoff `stated_assumptions`. No action required.
- Suggestion: None; the comment at lines 271–277 already records the rationale and the D2
  evidence. Recorded here only for audit completeness.

### Praise

**[P1] [Testing] `ConverseWireMapperTest.java` — regression tests assert the exact member that shipped broken**
- The two new tests assert both the populated member and the *absent* member
  (`text()` non-null + `json()` null for a String; the inverse for a Map). Asserting the
  absent member is what catches a future re-introduction of the always-json bug — the
  prior tests only checked `toolUseId`/`status` and stayed green against the broken
  mapping. The oracle is cited to `content-block.schema.json` + § 6.A.1, not to the impl.
