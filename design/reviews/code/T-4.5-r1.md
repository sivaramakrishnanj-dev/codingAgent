## Summary

T-4.5 adds the net-new inspect/curate surface to component C1 (CLI), reusing the existing `ResumeCommand`
subcommand shape exactly. **`resume`/`sessions` were NOT rebuilt** — they pre-exist (T-1.2: `ResumeCommand` +
`Main` dispatch), confirmed present + tested (`ResumeCommandTest`, `CliArgumentsTest`), left untouched. The
net-new work is four pieces:

1. **`memory [list|show <slug>|edit <slug>|rm <slug>]`** — new `MemoryCommand` (mirrors `ResumeCommand`: a small
   command class with exit-code-returning action methods over an injected `MemoryStore`, never the production
   composition). `list` prints the two-tier index (AC-14.3); `show` prints the entry markdown (AC-14.1); `edit`
   resolves + points at the on-disk markdown path for hand-editing (AC-14.1 "also hand-editable on disk"); `rm`
   removes the entry + its index line (AC-14.1/14.3). A minimal `MemoryStore.delete(slug, repoKey)` and
   `MemoryStore.entryPath(slug, repoKey)` were added, mirroring `write` (file + index op, library logs only).
   Dispatched in `Main` before the config gate as pure on-disk persistence (no config/model call).
2. **`config [show|path]`** — new `ConfigCommand`. `show` renders the resolved `ResolvedConfig` (reusing `Main`'s
   existing `ConfigResolver`/`ConfigLocations` resolution — no duplication); `path` prints the config file
   locations (US-8). Dispatched in `Main` after config resolution so `show` reflects the layered merge.
3. **`--debug` flag** — parsed in `CliArguments` (orthogonal to `Kind`, like `--mode`/`--attach`), and the actual
   level toggle applied by a new `DebugLogging` helper that sets the slf4j-simple
   `org.slf4j.simpleLogger.defaultLogLevel` system property to `debug` (else `info`). Because slf4j-simple reads
   that property once, at first-logger bind (empirically confirmed), the property MUST be set before any logger
   is created. A new `Launcher` (the manifest `Main-Class`, holding no logger of its own) calls
   `DebugLogging.applyFrom(args)` as its first action, then delegates to `Main.run`. The pom manifest now points
   at `Launcher`; both `Launcher` and the level-decision-applied-too-late problem are documented inline.
4. **SLF4J log levels (05-operations § 3)** — the INFO/WARN/ERROR/DEBUG semantics were already in place from
   prior tasks; the deliverable is the `--debug` toggle + confirming the level is tunable (DEBUG when `--debug`,
   INFO default) and that library code logs via SLF4J (NFR-LOG, verified: no `System.out`/`System.err` in
   `memory/` or `config/`). The new command classes print via an injected `PrintStream` (they ARE the CLI layer).

`mvn clean verify` → BUILD SUCCESS, 1308 tests (0 fail / 0 error), JaCoCo BUNDLE line 0.9145 (gate 0.80).

### Praise

**[P1] The subcommand shape mirrors `ResumeCommand` exactly — no new pattern invented.** `MemoryCommand` and
`ConfigCommand` are small CLI-layer command classes with exit-code-returning action methods over an injected
collaborator (real `MemoryStore` / real `ResolvedConfig`, never the production composition), tested against a
`@TempDir` store with a captured `PrintStream` — the same seam `ResumeCommandTest` uses. `resume`/`sessions`
(T-1.2) were confirmed present and left untouched; the "sessions" half of the task was already done.

**[P2] The `--debug` toggle is placed at the only correct point and the reason is documented.** slf4j-simple reads
its default-level property once, when the provider binds at first `getLogger` — so setting it inside `Main.run`
(after `Main`'s static logger has already bound the provider) would silently no-op. The new `Launcher` (no logger
of its own) sets the property before it touches `Main`. The load-bearing ordering is spelled out in `DebugLogging`
and `Launcher` Javadoc so a future contributor doesn't "simplify" it into a `Main.run` call and quietly break
`--debug`.

**[P3] Reuse over re-implementation.** `config show` renders the `ResolvedConfig` produced by `Main`'s existing
`ConfigResolver`/`ConfigLocations` wiring — the layered merge is not duplicated. `MemoryCommand` drives the
existing `MemoryStore.readEntry`/`loadIndexes`; only the genuinely-missing `delete`/`entryPath` ops were added
(minimal, mirroring `write`). The CLI-layer entry render in `MemoryCommand` is distinct from `ReadMemoryTool`'s
raw-body tool result (different consumers: terminal display vs. model payload), so it is not a DRY collision.

### Discussion

**[D1] [Discussion] CT-INV-11's test mechanism widened to admit T-4.5's authorized curation surface; the formal
oracle is unchanged.** The pre-existing `NoAutoExtractContractTest.writeIsTheSolePersistencePath` asserted (via a
reflection allowlist `{write, readEntry, loadIndexes}`) that `write` is the only public instance method on
`MemoryStore`. Adding `delete` (the `memory rm` seam the task explicitly authorized) and `entryPath` (a pure
query) mechanically broke that allowlist. The **formal** CT-INV-11 oracle
(`06-formal/contract-tests.md` § 3.1, line 57) is: *"no MemoryEntry persists without explicit/approved write (no
auto-extract)"* — referencing INV-13, AC-21.4. That oracle is about **no auto-extraction of persisted entries**,
not a literal "the class has exactly one method." `delete` is explicit, developer-initiated curation (not
auto-persistence) and `entryPath` is a query — neither introduces an auto-extraction path, so the formal oracle is
**preserved**. I widened the test's allowlist to the now-complete non-auto-persisting surface
(`{write, readEntry, loadIndexes, entryPath, delete}`), kept the positive `write`-exists assertion, kept the two
behavioural assertions (read-empty-persists-nothing; entry-appears-only-after-explicit-write — the real teeth of
"no auto-extract"), and tightened the comment/Javadoc to state the oracle as "no auto-extract" rather than "sole
method." This is a spec-faithful test fix, not a contract weakening.
- spec_refs: CT-INV-11, INV-13, AC-21.4 (`06-formal/contract-tests.md` § 3.1 line 57)
- suggested_amendment_kind: contract-test-update (optional, documentary) — the formal CT-INV-11 row could note
  that "the store's public surface may also expose explicit query/curation methods (`entryPath`, `delete`); the
  invariant is no *auto-persistence*, not a single-method surface." No design behaviour changes; the wording would
  just align the formal note with the now-richer (and still auto-extract-free) curation surface T-4.5 adds.
- Non-blocking; surfaced for the coordinator's Discussion lane. The behavioural CT-INV-11 assertions remain
  unchanged and green.
