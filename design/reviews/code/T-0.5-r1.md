## Summary

T-0.5 delivers the Model Client (C4) wire-format boundary under a new `com.srk.codingagent.model.converse` package — the one place our domain types cross to/from the Bedrock Converse SDK types (03-data-model.md § 7, § 6.A.1; ADR-0001). The surface is: `ConverseWireMapper` (request-build `toRequest` + response-parse `toContentBlocks`/`toStopReason`/`toUsage`/`toModelResponse`), `ModelClient` (the adapter that wraps an injected `BedrockRuntimeClient`, calls `converse` once, parses the `Turn`), `DocumentConverter` (bidirectional JSON ↔ SDK `Document` for tool input/content), `StopReasonMapper` (wire→domain `StopReason` by case fold), the provider-agnostic request DTO `ConverseMessage` + `Role`, and two typed exceptions (`ToolProtocolException` for the INV-6 protocol fault, `ModelBackendException` for the exit-4 backend failure). The three block kinds named in the title — text, toolUse, toolResult — are fully mapped both directions where § 7 allows (toolResult is input-only; toolUse-from-model is response-only). The toolUse↔toolResult pairing invariant (INV-6) is enforced in `toRequest` by an order-preserving scan: a toolResult whose toolUseId has no *prior* toolUse in the transcript is rejected with `ToolProtocolException` before any request leaves the boundary (CT-INV-5, the negative test). The real `.converse()` call is the only un-unit-testable line; it is isolated behind the injected client seam, so request-build and response-parse run for real against a hand-rolled `BedrockRuntimeClient` stub — no network, no live Converse call (per the task's mocked-Bedrock directive). `mvn clean verify` is green: 229 unit tests (48 new), bundle line coverage 95.24%, converse-package line coverage 98.01%, the 0.80 JaCoCo gate met with **no new exclusion**. Non-streaming mapping only; streaming deferred (stated assumption). One Minor (a couple of defensive guards uncovered by design), one Nit (fully-qualified SDK `ContentBlock` references), one Discussion (whether `ConverseMessage`/`Role` belong here vs a future shared home). No Blockers, no Majors.

### Minors

**[m1] [Testing] `model/converse/ConverseWireMapper.java:296` / `DocumentConverter.java:102` — two defensive arms uncovered**
- Why: the "unsupported response content block type" throw (296-297) and the unreachable "unsupported document node" throw (102) have no test. Both are defensive guards: § 6.A.1 says a model turn emits only text/toolUse, and the SDK `Document` is a closed node set, so neither arm is reachable with a well-formed input the in-scope spec permits. Covering them would require constructing a malformed SDK object (e.g. a response block of an out-of-scope type), which tests the SDK more than the mapper. Minor, not Blocker, because the pinned "tests are not optional" tenet targets *business logic* — these are unreachable-by-contract guards, and the package is at 98.01% line coverage.
- Suggestion: optionally add a test that feeds an out-of-scope response block (e.g. an `image` content block) to `toContentBlocks` and asserts the `IllegalArgumentException`, documenting that out-of-scope kinds are rejected rather than silently dropped. Non-blocking; deferrable to the task that adds multimodal mapping (T-4.2). (No change applied — Phase C is review-only.)

### Nits

**[N1] [Readability] `model/converse/ConverseWireMapper.java` — fully-qualified `software.amazon.awssdk...ContentBlock` references**
- Why: our domain `ContentBlock` and the SDK `ContentBlock` collide by simple name, so the SDK type is written fully-qualified at each use (`toWireBlock`, `toContentBlocks`, `toDomainResponseBlock`). This is correct and unambiguous but verbose. An import alias is not available in Java; a private static factory wrapper would hide the FQN but add indirection. Author's choice — the FQN makes "which `ContentBlock`" unmistakable at the boundary, which has review value here.
- Suggestion: leave as-is, or extract the wire-`ContentBlock` construction calls behind small private helpers if the verbosity grows in T-0.6/T-0.8. Non-blocking.

### Discussion

**[D1] [Discussion] `ConverseMessage` + `Role` placement — Model Client request DTO vs a future shared conversation type**
- Why: `ConverseMessage(Role, List<ContentBlock>)` is the provider-agnostic request-side turn the owned loop (T-0.8) will assemble. It is *not* a duplicate of any persistence type — the event-log payloads (`UserMessagePayload`, `ModelResponsePayload`) carry role *implicitly* (by event type) and live in the event domain, whereas `ConverseMessage` carries role *explicitly* for the wire boundary. 03-data-model.md § 7 notes replay "reverses this: events → our blocks → Converse messages[]" — that events→messages translation is the loop's/replay's job (T-0.8 / AC-7.2), and it will consume `ConverseMessage`. The open question is purely *where the type lives*: it sits in `.model.converse` today (the boundary that defines it). If T-0.8 finds it wants the same role-carrying turn type upstream of the Model Client, it may warrant promotion to a shared package. No spec symbol pins the home, so this is a placement note, not a defect.
- Suggested amendment kind: none now (no spec change needed). Surfaced so the coordinator/T-0.8 is aware the type exists and where.

### Praise

**[P1]** The injected-`BedrockRuntimeClient` seam (mirroring the T-0.3 `BedrockClientFactory.wiring` pattern) keeps the real `.converse()` call the *only* un-coverable line and made a JaCoCo exclusion unnecessary — request-build and response-parse are fully exercised against a hand-rolled stub with the real mapper, so the SUT is never mocked and no live Bedrock call occurs. The INV-6 scan is an order-preserving pass that correctly rejects both the orphan-toolResult case and the subtler toolResult-before-its-toolUse case (`ct_inv5_toolResultBeforeItsToolUseRejected`).

## Design-doc alignment

- **03-data-model.md § 7 (wire-format table):** all in-scope rows mapped — TextBlock↔text, ToolUseBlock↔toolUse{toolUseId,name,input} (response→us), ToolResultBlock↔toolResult{toolUseId,content,status} (us→request, user-role msg), ResolvedConfig.modelId→request modelId, system prompt→system[], StopReason→response stopReason, Event(MODEL_USAGE)→response usage. Out-of-scope rows (ReasoningBlock, ImageBlock, DocumentBlock, CachePointBlock) deferred per scope (stated assumption).
- **§ 6.A.1 (verified Converse facts):** request shape (modelId, messages[] with role+content[], system[] separate, optional toolConfig) built; stateless transcript resent (the full `List<ConverseMessage>` is the request input); stopReason wire form (lowercase/underscore) mapped to UPPERCASE domain enum by case fold; usage (input/output + optional cacheRead/cacheWrite) mapped; toolResult appended as a USER-role message.
- **INV-6 / CT-INV-5:** enforced in `toRequest` and surfaced through `ModelClient.converse` before any backend call.
- **ADR-0001:** Converse (non-streaming) request-build + response-parse mapping delivered as the wire seam; the loop (T-0.8), toolConfig rendering (T-0.6), permission gate (T-0.7), capability fields (T-4.3), and live calls are all out of scope and untouched.
- **Component C4 (02-architecture.md § 1.2):** provider-agnostic surface (callers pass/receive only domain types + the opaque SDK `ToolConfiguration`/`Turn`); no business logic; read/invoke only (no AWS write verbs); credential chain honored via the injected `BedrockRuntimeClient` (built by the T-0.3 `BedrockClientFactory`).
