## Summary

T-1.2 delivers the three resume behaviors over the C15 session store and the C1 CLI, with no new dependency and no live model call (pure offline persistence + replay). **(1) Listing (AC-7.1 / AC-15.2):** `SessionStore.listSessions(repoKey)` enumerates `<store>/projects/<repoKey>/sessions/*.jsonl`, builds a lightweight `SessionListing` per session (id from the filename, log modification time, and the lineage edge read from the sibling `.meta.json` when present), and orders **most-recent-first** by log mtime (tie-broken by session id descending for a total, deterministic order). mtime is the defensible ordering key: the timestamp-prefixed ids of 03-data-model § 2.1 sort the same way, but the M0 one-shot lineage id `"one-shot"` is not timestamp-prefixed, so mtime is robust where reverse-lexical would mis-order. **(2) Replay (AC-7.2, INV-1):** `SessionReplay.replay(List<Event>)` reverses the wire-format boundary (03-data-model § 7: events → our blocks → Converse `messages[]`) by projecting the message-bearing events in seq order — `USER_MESSAGE → ConverseMessage.user`, `MODEL_RESPONSE → ConverseMessage.assistant` — and skipping the audit-only kinds (`SESSION_START`, `MODEL_USAGE`, `TOOL_USE`, `PERMISSION_DECISION`, `TOOL_RESULT`, `OUTCOME`, …). This reproduces exactly the transcript `AgentLoop` accumulates (the loop's batched tool-result `USER_MESSAGE` carries the `toolResult` blocks; the per-block `TOOL_RESULT` events are audit), preserving `toolUse↔toolResult` pairing (INV-6) and user/assistant ordering. **(3) Latest-continuation default (AC-7.4):** `SessionLineage.latestContinuation(id, listings)` walks `DERIVED_FROM` continuations to the tip of the derived chain (a `SPAWNED_BY` sub-agent edge is not a continuation and is not followed; a fabricated cycle is broken by a visited-set so the walk terminates per INV-3). `SessionStore.readMeta` reads back the derived `.meta.json` lineage edge the walk consumes. The CLI surface adds `CliArguments` kinds `RESUME`/`SESSIONS` (recognizing `resume [<id>]` and `sessions` as leading subcommands, 04-apis § 1.2) and a unit-testable `ResumeCommand` orchestrator that `Main` delegates to (the `resume`/`sessions` dispatch runs before the config gate — listing/resume need no config or model). `mvn clean verify` is green: **492 unit tests** (≈42 new — 8 `SessionReplayTest`, 8 `SessionLineageTest`, 6 `SessionListingTest`, 8 `ResumeCommandTest`, 5 added `SessionStoreTest`, 7 added `CliArgumentsTest`), 0 failures/errors/skipped; JaCoCo "All coverage checks have been met" (0.80 floor), **no new exclusion**; new classes 100%/98%/95% line. One Minor (a `ResumeCommand` resume-report message text the AC does not pin, asserted by `contains` in `ResumeCommandTest`), one Nit, one Discussion (the task row's Verify cell cites CT-INV-3, which pins INV-4 compaction byte-identity — M2's lane — not T-1.2's replay-fidelity contract; AC-7.2 + INV-1 are the binding contract here). No Blockers, no Majors.

**[m1] [Spec-Adherence] `cli/ResumeCommandTest.java:resume_replaysAndReportsTranscript` — the `"Resuming session '<id>'"` message text is impl-shaped on a spec-silent detail**
- Why: AC-7.2 pins that resume "reconstruct[s] its context by replaying the session's persisted events"; it does not pin the user-facing report wording. The test asserts the output `contains("Resuming session '2026-06-17T090000Z-abc'")`, which couples to `ResumeCommand.announceResume`'s exact phrasing. The load-bearing, spec-grounded part of the assertion — that the session id is named and the reconstructed turn count (`2 conversation turn(s)`) is reported (AC-7.2's reconstruction is observable) — is fine; only the verb phrase "Resuming session" is impl-shaped. The AC-7.4 sibling (`resume_defaultsToLatestContinuation`) is *not* in this category: its assertion that the report names the *continuation* rather than the *original* is the AC-7.4 behavior itself, not a wording coupling.
- Suggestion: narrow to a shape assertion tied to AC-7.2 (assert the output contains the session id and the turn count), or keep the literal and treat the report wording as a stated assumption. Non-blocking; the wording is reversible and the turn-count assertion already pins the reconstruction contract.

### Discussion

**[D1] [Discussion] The task row's Verify cell cites `CT-INV-3 (replay fidelity)`, but CT-INV-3 pins INV-4 / compaction byte-identity (US-18) — M2's lane (T-2.2), not T-1.2's replay-fidelity contract**
- Why: `06-formal/contract-tests.md` § 2 indexes `CT-INV-3 | INV-4 ⚙ | compaction creates a new session; the parent's events are byte-identical after | US-18`. That is the **compaction** byte-identity invariant, which is created by the M2 compaction tasks (T-2.1/T-2.2); compaction does not exist at M1 (the JSONL codec cannot even decode a `COMPACTION` event yet — `EventCodec` throws `UnsupportedPayloadException` for it, and no session in the store carries a `DERIVED_FROM` edge until compaction writes one). The replay-fidelity contract T-1.2 actually realizes is **AC-7.2** (reconstruct context by replaying persisted events) preserving **INV-1** (gap-free seq order), with **INV-6** pairing as the wire-validity guard. T-1.2 pins replay fidelity directly against the contract fixture and a real loop round-trip (right roles, right blocks, right order, pairing preserved) rather than pulling compaction forward to satisfy a mis-cited CT id. AC-7.4's latest-continuation **default logic** is built and unit-tested with synthetic `DERIVED_FROM` metas so it is ready when compaction lands, but no real continuation can exist until M2 — so the AC-7.4 condition ("a session that has compaction-derived continuations") is vacuously satisfied today and becomes live in M2.
- Suggestion (designer's lane, non-blocking): consider amending the T-1.2 Verify cell from `CT-INV-3 (replay fidelity)` to `AC-7.2 + INV-1 (replay fidelity); INV-6 (wire-valid)` and noting that CT-INV-3 belongs to the T-2.2 compaction task's Verify column. Suggested amendment kind: `contract-test-update` (or a `tasks` Verify-cell correction). Discussion only — does not affect resolution.

## Stated assumptions (audit trail)

1. **AC-7.1 / repo-key scope.** The lister and resume scope to `ONE_SHOT_LINEAGE = "one-shot"` — the same repo key the one-shot and REPL paths write under today. Real git-remote-else-normalized-path repo-key derivation (03-data-model § 2.1) is a deferred session task (Main's Javadoc says so); scoping to the key the rest of the system writes to is what makes `resume`/`sessions` list the sessions that actually exist on disk. Alternative considered: derive the real repo key now — rejected as out of T-1.2 scope and explicitly deferred by the milestone.
2. **AC-7.1 ordering key.** "most-recent-first" is realized by session-log modification time (newest first), tie-broken by session id descending. Alternative considered: reverse-lexical sort of the timestamp-prefixed session ids (03-data-model § 2.1) — rejected because the M0 `"one-shot"` id is not timestamp-prefixed and would mis-order under pure lexical sort; mtime is the robust observable signal and matches the timestamp-prefixed ids' order anyway.
3. **AC-7.2 / replay membership.** The replayable transcript is exactly the `USER_MESSAGE` + `MODEL_RESPONSE` events in seq order (the loop's accumulated transcript per AgentLoopTest's CT-INV-2 event order); `TOOL_USE`/`PERMISSION_DECISION`/`TOOL_RESULT`/`MODEL_USAGE`/`SESSION_START`/`OUTCOME` are audit, not turns. The batched tool-result `USER_MESSAGE` the loop emits carries the `toolResult` blocks the next call needs (INV-6).
4. **Continuation wiring (C2 seam).** T-1.2 provides replay (`messages[]` reconstruction) + listing + the latest-continuation default + the `resume`/`sessions` CLI surface; it does **not** wire the reconstructed `messages[]` into a live, continued `AgentLoop` turn, because that requires the production Bedrock composition (`AgentLoopFactory`, the coverage-excluded live-credentials seam). `ResumeCommand.resume` produces and reports the reconstructed transcript that wiring starts from; the live-continue wiring is the documented next-task seam (no C2 structural change was forced — adding a speculative `AgentLoop.run(prior, prompt)` entry that nothing tested could feed end-to-end was deliberately avoided).
