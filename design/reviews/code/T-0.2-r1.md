### Minors

**[m1] [Reliability/Correctness] `cli/Main.java:64-75` — only `ConfigException` is caught; an IO read failure on a config file escapes as an uncaught fault**
- Why: `ConfigLoader.load(Path)` wraps a genuinely-unreadable (but present) file as `UncheckedIOException`, which `Main.run()` does not catch — it would propagate out of `run()`/`main()` and terminate the JVM with exit 1, not 2. AC-6.4/AC-8.5 scope config faults to exit 2 for *missing/invalid/malformed* values; a present-but-unreadable file is an environmental fault, arguably exit 1 (INTERNAL) territory, so this is not a contract violation. But the boundary is undocumented, and a permissions-denied config file is a foreseeable operator error. Tenet: debugging must be easy.
- Suggestion: out of T-0.2's cited scope to fully wire the exit-code dispatch (that is a later CLI task), but a one-line comment at the catch site noting that IO faults intentionally fall through to the INTERNAL path would make the boundary explicit. No code change required this task.

**[m2] [Reuse] `config/ResolvedConfig.java:80-96` vs `config/ConfigResolver.java:200-247` — duplicated range/blank validation across two classes**
- Why: the resolver validates ranges/types (throwing `ConfigException` naming the key) and the record's compact constructor re-validates the same invariants (throwing `IllegalArgumentException`). This is intentional belt-and-braces (EJ Item 17 defensive invariant; the record can never hold an invalid value even if constructed directly), and the two layers throw different exception types for different contracts — so it is not a true DRY violation. Recording it as a Minor only because the second layer's branches are largely unreachable through the resolver path (the resolver rejects first), which slightly dilutes coverage signal.
- Suggestion: keep the record invariant (it is the right call for an immutable value object). Optionally add a short Javadoc note on the record that the resolver is the primary gate and the constructor is the defensive backstop — the existing Javadoc already says this, so no change needed.

### Nits

**[N1] [Readability] `config/ConfigResolver.java:148`, `config/ResolvedConfig.java:70,72` — fully-qualified `java.util.*` names inline instead of imports**
- Why: `java.util.Arrays.toString(...)` and `java.util.Objects.requireNonNull(...)` are written fully-qualified rather than imported. Harmless; a static/regular import would read marginally cleaner. Author's choice.

### Praise

**[P1] [API-Design] `ConfigLoader` / `ConfigResolver` / `ConfigLocations` testability seams**
- The loader takes `Path` *or* `Reader` (so YAML can be fed in-memory), the resolver exposes both a file-path overload and a pre-loaded-map overload (so precedence is unit-testable without disk), and `ConfigLocations` injects its store root. This made hermetic, mock-free tests straightforward — every test exercises the real SUT. The immutable `ResolvedConfig` with a validating compact constructor (ADR-0009 "one immutable resolved object") is a clean realization of the ADR.
