## Summary

T-3.6 implements approved amendment DCR-6: `GreenfieldArtifactStore.resolveArtifact` (C9, with C3) is
tightened from the bare `resolved.startsWith(<workspaceRoot>/design)` confinement to a strict allowlist
of the three known design-doc artifacts (`design/00-requirements.md`, `design/01-design.md`,
`design/02-tasks.md`). The prior check let any source path under `design/` (e.g. `design/impl/pom.xml`,
`design/impl/src/main/java/.../Calculator.java`) resolve under `<workspaceRoot>/design` and pass,
leaving the AC-1.4 pre-approval source-write hole open against the class Javadocs' "cannot reach a
source file" promise. The fix adds a self-contained `ARTIFACT_FILE_NAMES` set in the `tool` package
(the authoritative `GreenfieldArtifact` enum lives in `workflow`, which already depends on `tool`, so
importing it back would be circular — the same constraint the `APPROVAL_STAMP_MARKER` Javadoc calls
out), resolves the three allowed absolute paths in the constructor, and refuses any path not in the
allowlist. The now-dead `artifactRoot` field was removed. `WriteArtifactTool.java` is unchanged — it
delegates confinement to the store (one source of truth). A belt-and-braces drift-guard test pins the
hard-coded names against `GreenfieldArtifact.values()` `relativePath()` so the two cannot drift.

### Praise

**[P1] One source of truth + a drift guard — the same belt-and-braces the store already uses.**
The allowlist is expressed self-contained in the `tool` package (avoiding the `tool -> workflow`
circular dependency the directive and the `APPROVAL_STAMP_MARKER` Javadoc both flag), and a unit test
(`allowlistMatchesGreenfieldArtifactEnum`) pins `ARTIFACT_FILE_NAMES` resolved under `ARTIFACT_DIR`
against `GreenfieldArtifact.values()` `relativePath()`. The store's hard-coded names cannot silently
drift from the authoritative enum — exactly the pattern the existing approval-stamp marker uses.

**[P2] The tightening subsumes the prior check rather than replacing it loosely.** Any path resolving
outside `design/` is also absent from the allowlist, so the bare-source-path case (`pom.xml`,
`src/main/java/Foo.java`) and the `../`-traversal case stay rejected — no regression of the prior
confinement — while the new `design/impl/**` source-write hole is closed. The allowlist (exactly three
artifacts) is the tightest expression of the directive's "ONLY the three known design-doc artifacts",
strictly narrower than a looser "`*.md` directly under `design/`" rule.
