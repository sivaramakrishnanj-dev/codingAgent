## Summary

D10 implements the DCR-1 (Option A) amendment: greenfield phase-deliverable persistence is now
**driver-guaranteed**, not model-tool-dependent. The `GreenfieldDriver` gains a third injected seam
`PhaseArtifactWriter`; on each pre-approval phase's `END_TURN` the driver captures the model's settled
final text (`LoopOutcome.finalTextIfPresent()`) and writes it to that phase's design-doc artifact in
code (truncating `GreenfieldArtifactStore.write()`) **before** the `ArtifactApprovalGate` stamps
(AC-1.2/AC-2.1). The driver also injects each approved earlier-phase artifact's content into later
phases' prompts (DCR-1 transcript continuity). `write_artifact` stays registered/available in the
pre-approval registry but is now optional, no longer the persistence path (ADR-0012 amended, C7 note).
AC-1.4 is preserved: the in-code write is `design/`-confined by the unchanged `GreenfieldArtifactStore`,
and the source-write Class-X tools stay withheld from the pre-approval loops.

**Why this attempt is structurally different from D6–D9.** Those four fixes all passed mock-scripted
tests yet live persistence still failed because the tests scripted a `write_artifact` tool_use the
live model never emits (a clean G3 run reached `END_TURN` in each pre-approval phase without ever
calling the tool). With persistence now driver-authored from the `END_TURN` prose, the contract is
finally **deterministically, mock-stably assertable with no scripted tool_use** — the test the prior
attempts could not be.

**The load-bearing new test** (the real, mock-stable contract):
`GreenfieldArtifactAuthoringTest.driverPersistsEachPhaseEndTurnProseWithoutAToolCall` — drives the
real `GreenfieldDriver` + real `PhaseArtifactWriter` (over a real `GreenfieldArtifactStore` on a
`@TempDir`) + real `ArtifactApprovalGate`, with phase loops that **write nothing** (no scripted
tool_use) and only return a per-phase deliverable as `LoopOutcome.finalText`. It asserts each artifact
on disk **contains the phase's END_TURN prose** — proving the driver authored it. The companion
`requirementsArtifactHoldsDriverProseThenApprovalStamp` adds the AC-1.5 stamp ordering, and
`laterPhasePromptsInjectApprovedEarlierArtifacts` pins the DCR-1 transcript-continuity injection.

### Minors

**[m1] [Testing] `GreenfieldDriver.authorPhaseArtifact` has one defensively-uncovered branch (`artifact.isEmpty()`).**
- Why: `authorPhaseArtifact` guards `GreenfieldArtifact.forPhase(phase).isEmpty()` (the IMPLEMENT
  case), but it is only ever called for a non-terminal phase (the loop returns early via
  `phase.isTerminal()` before reaching it), so the empty branch is unreachable in practice — JaCoCo
  shows 11/12 branches on the class. It is a correct defensive guard (it keeps `authorPhaseArtifact`
  self-contained should the call site ever move), not dead code. Line floor (0.80) is met; the class
  is at 98% line / 91.7% branch.
- Suggestion: none required. A test could call a package-private path to hit it, but that would test
  an impossible state; leaving the guard as documented defence is the cleaner choice.

### Nits

**[n1] [Readability] The `PhaseArtifactWriter` anonymous implementation in `ToolRegistryComposer.greenfieldArtifactWriter()` references `com.srk.codingagent.workflow.GreenfieldArtifact` by FQN.**
- Why: The method body uses the fully-qualified `com.srk.codingagent.workflow.GreenfieldArtifact` in
  the two override signatures rather than an import. It is mildly more verbose; an import would read
  cleaner. The composer already imports several `workflow` types, so adding one more is harmless.
- Suggestion: optional — add `import com.srk.codingagent.workflow.GreenfieldArtifact;` and drop the
  FQN. Author's call; does not affect correctness.

### Praise

**[P1]** The third seam mirrors the driver's existing two-seam shape (`PhaseLoopFactory` +
`ApprovalGate`) exactly — same injection discipline, same gate-covered-composer assembly, same
test-substitutability — so the DCR-1 change reads as a natural extension of the established design,
not a bolt-on. The driver writes the artifact *before* the gate consults the decision, which is the
precise ordering the amended ADR-0012 requires (driver writes → gate stamps), and the untraceable-stop
test confirms the gate's AC-2.5 check reads the driver-written artifact.

**[P2]** The persistence contract is finally testable with **no scripted tool_use**
(`driverPersistsEachPhaseEndTurnProseWithoutAToolCall`) — the exact mock-stability the prior four
attempts' tests lacked by construction. The CLI persistence tests (D7/D8/D9) were updated to drive
`END_TURN` prose through the production driver rather than scripting the tool_use the live model never
emits, closing the passed-but-broke gap at its root.

## Reuse / duplication
The new `PhaseArtifactWriter` seam was reuse-swept: no pre-existing artifact-writer abstraction exists
that the driver could have reused (`GreenfieldArtifactStore` is the concrete `tool/`-package store; the
driver lives in `workflow/`). The seam abstracts the store so the `workflow` package does not take a
hard dependency on the store's concrete `write`/`read` and stays unit-testable — consistent with the
driver's existing two functional seams, not a duplicate. The single anonymous impl in the composer
(`greenfieldArtifactWriter()`) and the identical in-memory/over-store test doubles
(`RecordingArtifactWriter`, `noopWriter()`, `writerOver(store)`) are per-test scaffolding for distinct
scopes (orchestration vs. prompt-on-wire vs. real-persistence), not production duplication.
`reuse_self_check: passed`.
