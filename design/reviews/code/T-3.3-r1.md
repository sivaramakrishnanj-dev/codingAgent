## Summary

T-3.3 adds the greenfield IMPLEMENT-phase loop: the orchestration the terminal phase of the
`GreenfieldPhase` machine drives once entered (reached only after the design + task breakdown are
approved, AC-2.3). It reads the approved task-breakdown artifact, implements the planned tasks one
at a time in breakdown order (AC-3.1), verifies each via the configured build/test command — reusing
the T-1.4 `VerifyLoop` (AC-3.2) — marks each verified task complete in the artifact before the next
(AC-3.3, reusing the T-3.2 `GreenfieldArtifactStore`), and stops + surfaces if a task fails
verification within `NFR-VERIFY-MAX-ITERATIONS` (AC-3.4). It is wired onto the live greenfield path:
the terminal-phase loop turn the `AgentLoopFactory` builds is now the implement loop, assembled in
the gate-covered `ToolRegistryComposer` (not the JaCoCo-excluded factory) and pinned reachable by a
composition test. A DRY refactor extracted the shared failure-feedback prompt
(`loop.RemedyPrompt`) so the brownfield and greenfield remedies share one builder.

### Nits

**[N1] [Readability] `workflow/GreenfieldImplementLoop.java` — parallel seam interfaces with `BrownfieldDriver`**
- Why: `GreenfieldImplementLoop.LoopTurn` / `Verifier` / `VerifierFactory` mirror the same-named
  nested interfaces in `BrownfieldDriver` (and `GreenfieldDriver.LoopTurn`). They are structurally
  identical functional interfaces. This is *consistent* with the established convention — each driver
  already owns its own `LoopTurn` type (`BrownfieldDriver.LoopTurn`, `GreenfieldDriver.LoopTurn`) — so
  it is not a new duplication so much as adherence to an existing one. Clean Code (DRY) would prefer a
  shared `loop`-package `AgentLoopTurn` seam someday.
- Suggestion: leave as-is for this task (matching the established per-driver-seam convention); a future
  consolidation of all three `LoopTurn` seams into one shared functional interface is a reasonable
  cross-cutting cleanup but is out of T-3.3's scope. Logged as a Nit, not acted on.

### Praise

**[P1] Verify-loop reuse honoured exactly.** `GreenfieldImplementLoop.overConfig` builds the verifier
as `remedy -> VerifyLoop.forConfig(executor, config, remedy)::verify` — byte-for-byte the same verify
composition `BrownfieldDriver.overConfig` uses — and the per-task remedy drives another agent-loop
turn fed the failure via the shared `RemedyPrompt`. Verification is reused, never reimplemented, per
the explicit T-3.3 directive.

**[P2] DRY refactor instead of absorbing a self-inflicted Major.** The first cut duplicated the
failure-feedback prompt across `BrownfieldDriver` and `GreenfieldImplementLoop` (~90% token overlap).
Phase C caught it; rather than ship the duplicate, the prompt was extracted to a shared, tested
`loop.RemedyPrompt` and both drivers now delegate to it. `BrownfieldDriver` returned to 100% line
coverage after losing its private copy.

## Design-doc alignment

- **AC-3.1** (one task at a time, breakdown order) — satisfied: `run` iterates
  `TaskTraceability.tasksInOrder` and drives one implementation turn per task, in file order.
- **AC-3.2** (verify via configured build/test command) — satisfied: per-task `VerifyLoop` reuse via
  `overConfig` over the configured test command.
- **AC-3.3** (mark complete in the artifact before the next) — satisfied: a VERIFIED task is appended
  a completion line to the tasks artifact before the loop advances (pinned by the
  `markCompleteHappensBeforeStartingNextTask` ordering test).
- **AC-3.4** (stop + surface on EXHAUSTED, not continue silently) — satisfied: the loop returns
  `VERIFY_EXHAUSTED` naming the stopped task and carrying the failing output, and does NOT run the
  next task's turn (pinned by `verifyExhaustedStopsWithoutAdvancing`).
- **AC-3.5** (single specific task requested → implement only that one and stop) — see Discussion D1:
  not implemented as a distinct request shape (the minimal shape implements the full approved
  breakdown). AC-3.5 is type "Op" (optional), so a Discussion note is the prescribed disposition per
  the task's own scope guidance.
- **AC-20.3/20.5/20.6** — satisfied: remedy feeds failure output back (shared `RemedyPrompt`);
  exhaustion surfaces with the last failure's output; no-test-command is reported, not papered over.
- **NFR-VERIFY-MAX-ITERATIONS** — satisfied: each task's fix-and-retry cycle is bounded by the verify
  loop's `config.verifyMaxIterations()` (pinned end-to-end through the real `VerifyLoop` by
  `overConfigBoundsRemedyTurnsByConfig`).
- **ADR-0012 implement clause** — satisfied: greenfield implement is orchestration over the same
  Agent Loop (the per-task turn is the IMPLEMENT-phase `AgentLoop`), not a separate engine; the
  terminal phase is reached only after approval gates pass (unchanged from T-3.1).
- **Live-path wiring** — the implement loop is reachable from the composition root: the terminal-phase
  loop turn the `AgentLoopFactory` builds is `composer.greenfieldImplementLoopTurn(loop::run)`,
  assembled in the gate-covered `ToolRegistryComposer` and pinned by `GreenfieldImplementCompositionTest`
  (both that the loop runs each task one at a time + marks complete, and that the per-task verify step
  is the *configured* command — a failing `false` surfaces a bounded stop).

## Discussion

**[D1] [Discussion] AC-3.5 (single-specific-task request) is not implemented as a distinct shape.**
- Spec: AC-3.5 (Op) — "Where the developer requested a single specific task, the agent shall implement
  only that task and then stop." The task row marks AC-3.5 optional and says "if the minimal shape
  naturally supports a single-task request, support it; if not, a Discussion note is fine."
- Observation: the minimal implement-loop shape implements the full approved breakdown in order; it
  does not take a "single task id" request parameter. Supporting AC-3.5 cleanly would mean threading a
  requested-task filter from the run path / CLI into the implement loop (and a way for the developer
  to express "just T-1.4"), which is a CLI-surface decision beyond this loop's orchestration. Per the
  task's own guidance, a Discussion note is the prescribed disposition; no spec is wrong or
  insufficient (AC-3.5 is optional), so this is NOT a design-change-needed.
- Suggested amendment kind: `ac-update` (only if the team later wants AC-3.5 promoted to mandatory and
  a request-shape pinned). For now: informational; does not block resolution.
