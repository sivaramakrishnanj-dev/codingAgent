## Summary

T-3.10 makes the greenfield IMPLEMENT loop's per-task completion marker (the durable
`- [x] <taskId> Implemented` line T-3.8 appends, written by `CompletionStamp.line`) a **write-and-read**
fact: before running, the loop reads the markers back from the same `02-tasks.md` artifact, computes the
already-completed task ids, and **skips** them — implementing only the remaining incomplete tasks in
breakdown order, resuming at the first incomplete one rather than restarting at `T-1` (AC-7.6 IMPLEMENT
facet, AC-3.3, ADR-0012, resolves D2). The end-of-phase verify (T-3.9) still gates the **phase**: it runs
once over the whole complete phase even on a re-entry that implements nothing because every planned task
was already marked complete (AC-3.2). The **double-count subtlety** (a `- [x] T-1 Implemented` marker is
itself recognized as task `T-1` by `TaskTraceability.TASK_LINE`) is handled by recognizing the marker
specifically via its whole shape (`CompletionStamp.isCompletionLine` — the checked `[x]` box **plus** the
trailing `Implemented` token) and stripping those marker lines before the planned-task enumeration;
`TaskTraceability.tasksInOrder()` / `check()` are left marker-unaware and unchanged (backward-compatible).

### Praise

**[P1] Spec-Adherence — the double-count subtlety is solved by shape, not by the `[x]` checkbox.** The
completion marker is recognized by its **whole** `CompletionStamp` shape (checked box + trailing
`Implemented` token), not by the `[x]` box alone — which is exactly correct, because a real planned task
may itself be a checked/unchecked checkbox line that `TaskTraceability` recognizes as a task. The
`completionStampDoesNotMatchPlannedTaskLines` test pins that a planned `- [x] T-1 Build the parser` (no
`Implemented` token) is **not** skipped, closing the silent-skip-of-an-unimplemented-task hazard.

**[P2] Reuse / one-source-of-truth — planned enumeration still goes through `TaskTraceability`.** Rather
than write a second task-line parser in the loop, the planned-task enumeration reuses
`TaskTraceability.tasksInOrder` over a marker-stripped copy of the breakdown. The marker-specific
recognition lives only in `CompletionStamp` (the marker's own writer) — so there is one parser for
"what is a planned task" (`TaskTraceability`) and one for "what is a completion marker" (`CompletionStamp`),
with no drift between writer and reader (round-trip test pins it).

**[P3] Spec-Adherence — CT-GF-6 verified at two layers.** The skip-resume behaviour is pinned at the
loop unit level **and** end-to-end over disk (`GreenfieldResumeContractTest` wires the real implement
loop behind a reconstruct→IMPLEMENT driver path with the production stamp + marker shapes), mirroring the
CT-GF-1/2 contract-test discipline — so the on-disk read-back is proven, not just the in-memory path.

## Stated assumptions

- **spec_ref: AC-7.6 / AC-3.3 (item 3 — distinguishing a completion-marker line from a planned-task
  line).** The markers (`- [x] <taskId> Implemented`) live in the SAME `02-tasks.md` artifact as the
  planned tasks, and `TaskTraceability.TASK_LINE` recognizes a `- [x] T-1 …` checkbox line as task `T-1`,
  so a naive re-read would double-count (planned line + marker line) and could re-run a completed task.
  **Reading:** a completion marker is recognized specifically by its **whole** `CompletionStamp` shape —
  the checked `- [x]` box PLUS the trailing `Implemented` `MARKER` token as the line's final content
  (`CompletionStamp.isCompletionLine`/`taskIdOf`) — NOT by the `[x]` checkbox alone. The completed-id set
  is computed from those marker lines; the planned-task enumeration runs over a copy of the breakdown with
  the marker lines stripped, so it is never polluted by the markers. `tasksInOrder()`/`check()` stay
  marker-unaware (the spec/directive require their contracts unchanged; the skip logic is the loop's job).
  **Alternative considered:** overloading `tasksInOrder()` with marker-awareness — rejected because the
  directive prefers the skip logic in the loop and requires `tasksInOrder`/`check` backward-compatible (a
  marker-aware `tasksInOrder` would change its meaning for callers passing a pre-completion breakdown).
- **spec_ref: AC-3.2 (item 5 — the end-of-phase verify on a fully-already-complete re-entry).** When every
  planned task already carries a completion marker on re-entry, nothing is implemented this run.
  **Reading:** the end-of-phase verify still runs **once** over the whole already-complete phase (AC-3.2
  gates the PHASE, not each task), reaching the same terminal disposition a single uninterrupted run
  would — verified work re-verified is consistent and safe, and matches the DCR-7 "verify gates the phase"
  model. The terminal outcome is `ALL_IMPLEMENTED` (reusing the existing disposition with the full
  completed set, per the directive's "prefer reusing ALL_IMPLEMENTED if it fits"), carrying the whole
  completed phase in breakdown order — not only the (empty) set of tasks this resumed run touched.
  **Alternative considered:** adding a distinct "nothing-to-implement / already-complete" disposition —
  rejected because it is not genuinely distinct from "every task complete + verify gates the phase," and
  reusing `ALL_IMPLEMENTED` keeps the disposition surface and the `asLoopTurn()`/report mapping unchanged.
