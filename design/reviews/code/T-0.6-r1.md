## Summary

### Minors

**[m1] [Testing] `tool/CommandExecutionException.java` (0% line) / `CommandExecutor` infra-failure arms uncovered**
- Why: the `start()` IOException arm, the `InterruptedException` arms in `waitFor`/`drain`, and `readStream`'s IOException arm have no test. These are infrastructure-failure guards (subprocess fails to start; the runner thread is interrupted), distinct from a command that runs and exits non-zero (which is a normal `CommandResult`, not an exception). Triggering them hermetically requires faking a `ProcessBuilder` start failure or interrupting the test thread mid-wait â€” both test the JVM/SDK more than the executor. The pinned "tests are not optional" tenet targets *business logic*; these are unreachable-by-normal-input guards, and the tool package is at 77.91% line with the bundle gate (0.80) met.
- Suggestion: optionally, a follow-on could inject a command-runner seam to simulate a start failure and assert `CommandExecutionException` wrapping, but it adds indirection for a 2-line guard. Non-blocking; no change applied (Phase C is review-only).

**[m2] [Testing] `tool/WriteFileToolTest.java:createsNewFile/replacesExistingFile` â€” asserts the impl-chosen words "created"/"modified"**
- Why: 04-apis Â§ 3 pins the `write_file` result as "ok/diff summary" but does **not** pin the exact summary vocabulary. The tests assert `.contains("created")` / `.contains("modified")`, which are words my Phase A `WriteFileTool` chose â€” a mild oracle-traceability soft spot (the *concept* created-vs-modified is spec-aligned; the literal token is impl-derived). The companion assertions (`startsWith("ok")` traces to the "ok" in the spec language; the file-content assertion traces to the spec'd write behavior) keep the test well-founded, so this is Minor, not a Major oracle-traceability failure.
- Suggestion: assert the semantic distinction via the filesystem precondition (file existed vs not) plus `startsWith("ok")`, leaving the exact word unasserted; or record the summary vocabulary as a pinned contract. Recorded as a stated assumption (diff-summary shape). Non-blocking; no change applied.

**[m3] [Observability] `tool/CommandExecutor.java:run` â€” `LOGGER.info("...timeout {}s...", timeout.toSeconds())` truncates sub-second timeouts to 0**
- Why: the timeout is enforced in milliseconds (`waitFor(timeout.toMillis(), ...)`), but the INFO log reports `timeout.toSeconds()`, which renders a 200 ms timeout as `0s`. The behavior is correct (the 200 ms test times out as expected); only the log line is misleading for sub-second timeouts. In production NFR-CMD-TIMEOUT is 300 s, so this is cosmetic â€” but it surfaced in the test logs (`timeout 0s: sleep 10`).
- Suggestion: log `timeout.toMillis()` (or `timeout` directly) so sub-second budgets are visible. Non-blocking; no change applied.

### Discussion

**[D1] [Discussion] `tool/ToolInputs` model-input validation vs `persistence.Payloads` field validation**
- Why: `ToolInputs.requireString` (and `Payloads.requireNonBlank` in persistence) both express "non-blank string or fail" â€” but they serve different layers and throw different types: `ToolInputs` validates *model-supplied* `toolUse.input` and throws `ToolInvocationException` (so the registry returns an `error` tool result the model reacts to), whereas `Payloads` validates *constructor* fields and throws `IllegalArgumentException` (a programming-error contract), and is package-private (not reusable cross-package). They are not duplicates by the reuse rubric (different intent, different failure semantics, different visibility), so this is **not** a `[Reuse]` Major. It is surfaced only so a future task that introduces a shared validation utility is aware both exist. No spec symbol pins a shared validator.
- Suggested amendment kind: none (no spec change needed). Surfaced for awareness.

### Praise

**[P1]** Reusing the existing `persistence.OperationClass {READ, SIDE_EFFECTING}` for the per-tool Class R/X marker (rather than minting a parallel `ToolClass` enum) keeps the gate's classification vocabulary single-sourced â€” the same enum the fixture's `PERMISSION_DECISION` line and the T-0.4 payload already use. The concurrent stream-drain + `ProcessHandle.descendants()` tree-kill is implemented carefully (drain futures are `.get()`-ed before the pool is `shutdownNow()`-ed, so no output is lost; the kill destroys the `sh` parent *and* its descendants so forked build tools are not orphaned), and the large-output (`seq 1 200000`) test would hang on a serial drain â€” proving the concurrency is load-bearing, not decorative.
