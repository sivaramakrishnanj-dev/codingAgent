## Summary

T-3.2 builds the greenfield **artifact-authoring layer** (component C3, ADR-0012 greenfield side;
RD-7) over the T-3.1 phase state machine: (a) the design-doc write path that persists the
requirements/design/tasks markdown into the **target** repo (AC-1.2/AC-2.1), (b) the approval gate
that records a timestamped approval into each phase's artifact when the developer confirms it
(AC-1.5, generalized per-phase per ADR-0012), and (c) the enforcement that every task in the
breakdown traces to тЙе 1 requirement (AC-2.5), with a stable id (AC-2.2).

Six new production classes:
- `tool.GreenfieldArtifactStore` тАФ confines artifact paths to the **target repo's** `design/`
  directory (reusing `WorkspacePaths`) and does the write/append/read IO. The seam that reconciles
  ADR-0012's allowed design-markdown write with AC-1.4's no-source-write rule: a write outside
  `design/` (a source file) is refused.
- `tool.WriteArtifactTool` (`write_artifact`, Class X) тАФ the design-doc write path the model uses in
  the pre-approval phases, distinct from the withheld general source-write `WriteFileTool`.
- `workflow.GreenfieldArtifact` тАФ the phase тЖТ target-repo-relative artifact path map
  (REQUIREMENTSтЖТ`design/00-requirements.md`, DESIGNтЖТ`design/01-design.md`, TASKSтЖТ`design/02-tasks.md`);
  IMPLEMENT authors none.
- `workflow.TaskTraceability` тАФ the AC-2.5/AC-2.2 validator: every recognized task (stable id) must
  reference a stated requirement (US/AC/NFR/RD/INV).
- `workflow.ArtifactApprovalGate` (implements `GreenfieldDriver.ApprovalGate`) тАФ the explicit T-3.2
  extension point: on approval it enforces traceability (tasks gate) and stamps the timestamped
  approval into the phase's artifact; declined тЖТ records nothing, does not advance.
- `workflow.ApprovalStamp` тАФ the single tested unit that formats the timestamped approval line.

Three modified CLI files: `ToolRegistryComposer` (adds `write_artifact` to the pre-approval registry
and the gate-covered `greenfieldApprovalGate(decision)` seam), `AgentLoopFactory` (the
JaCoCo-excluded `createGreenfieldDriver` that wires the driver from the gate-covered composer), and
`Main` (one-shot + REPL greenfield paths now build the driver via the timestamped gate, replacing the
T-3.1 bare `completedPhase -> false`).

**The design decision (recorded in `rationale_for_significant_choices`):** the artifacts are
**model-authored** (pattern a) тАФ the model writes the markdown body through the `write_artifact`
design-doc path the pre-approval registry now offers тАФ while the approval timestamp + traceability
enforcement are **driver-authored** (the `ArtifactApprovalGate` stamps deterministically at the phase
boundary). This is the minimal-viable shape over the existing seams: it uses the explicit
`ApprovalGate` extension point the driver's Javadoc names ("T-3.2 layers approval-timestamp recording
over this seam"), keeps AC-1.4 intact (design-doc write тЙа source write), and is reachable on the live
path (composer-built gate, wired through `Main`).

**Live-path reachability.** Both the design-doc write path (offered in the pre-approval registry) and
the timestamped/traceability-enforcing gate (built by `ToolRegistryComposer.greenfieldApprovalGate`)
are assembled in the gate-covered composer, NOT the JaCoCo-excluded `AgentLoopFactory`/`Main`, and are
reachable from the composition root (`createGreenfieldDriver`). `GreenfieldArtifactCompositionTest`
pins this тАФ the same gate-covered-seam discipline T-3.1 used and the front-run for the
built-but-not-wired defect class.

## Findings

### Majors

### Minors

**[m1] [Reuse] `tool/WriteArtifactTool.java:97` тАФ `requireContent` duplicates `WriteFileTool.requireContent`**
- Why: `WriteArtifactTool.requireContent` is a verbatim copy (~8 lines) of `WriteFileTool.requireContent` (`tool/WriteFileTool.java:99`). DRY (Fowler). `ToolInputs.requireString` can't be reused directly because it rejects blank, while artifact/file content may legitimately be empty (an empty file is valid).
- Suggestion: extract a shared `ToolInputs.requireContent` (or `requireStringAllowingEmpty`) and have both tools call it. Minor тАФ two small static methods in the same package; deferred-safe, does not block resolution. Not a `reuse_self_check` failure (that fails only on a Major-severity duplicate of business logic).

### Nits

**[N1] [Readability] `tool/GreenfieldArtifactStore.java:155` тАФ `writeString` IO idiom mirrors `WriteFileTool.write`**
- Why: the `createDirectories` + `writeString` + `ToolInvocationException`-wrap idiom (~6 lines) recurs in `WriteFileTool.write`. This is standard Java IO, not a domain utility; extracting it would couple two tools with different result semantics (diff summary vs. artifact path) for little gain.
- Suggestion: leave as-is; noted only so a future "extract a file-write primitive" refactor has a pointer. Author's call.

### Praise

**[P1] Structural AC-1.4 preservation while opening the design-doc write.** The tension the task
flagged тАФ "a design-markdown write SUCCEEDS in a pre-approval phase while a general source write does
NOT" тАФ is resolved by *path-scoping* (`GreenfieldArtifactStore` confines to `design/`) rather than by
re-opening `WriteFileTool` or relaxing the gate. `GreenfieldArtifactCompositionTest` pins exactly
`write_artifact` present / `write_file` absent in each pre-approval phase, so the safety property is a
toolset property the model ever sees, independent of permission mode.

**[P2] AC-2.5 enforced, not advised.** Traceability is checked at the tasks gate before the approval
is recorded; an untraceable breakdown refuses the advance into implementation. This makes ADR-0012's
"traceability mandatory" a property the gate guarantees, exercised end-to-end through the real driver
in `GreenfieldArtifactAuthoringTest`.
