## Summary

T-0.7 delivers the Permission Gate (C8, ADR-0004) — the security chokepoint — as a new `com.srk.codingagent.permission` package, eight production classes, no new dependency. `PermissionGate.evaluate(GateRequest)` is the single authorization point the loop (T-0.8) consults before `ToolRegistry.dispatch`; it returns a `GateDecision` the loop turns into the `PERMISSION_DECISION` event (`GateDecision.toPayload` builds the reused `persistence.PermissionDecisionPayload`) and the S3→S4 (approve) / S3→S2 (deny) transition. The 4-mode decision table (AC-9.1..9.6, the ADR table) is implemented exactly: Class R auto-approves in every mode before any other logic (AC-9.6); Class X is denied in READ_ONLY (AC-9.2/RD-6), prompted every time in ASK_EVERY_TIME (AC-9.4), prompted-then-remembered in ASK_ONCE_THEN_REMEMBER (AC-9.5), auto-approved in UNRESTRICTED (AC-9.3). The RD-2 destructive denylist runs **first** for a `run_command` (AC-10.4, INV-9): a denylisted command always prompts (even UNRESTRICTED), is denied outright in READ_ONLY, and never produces or matches a `Grant` — the `GateDecision` record enforces `denylisted ⇒ matchedGrant == null` at construction. The denylist is a conservative pattern test over a quote/escape-honoring `ShellTokenizer` (the load-bearing front end): it catches `rm -r/-f/-rf`, `rmdir`/`dd`/`truncate`/`sudo`/`killall`, two-operand `mv`/`cp`, `> ` overwrite, destructive `git` (push --force/-f, reset --hard, clean -fd(x), checkout/restore), `chmod`/`chown -R`, fork-bomb shapes, `curl|sh`/`wget|bash` pipe-to-shell, and `kill -9` — seeded from the ADR table, matched case-insensitively, on the basename of the executable (`/usr/bin/RM` → `rm`), and per chained segment (a destructive verb after `&&`/`;` is caught). RD-1 grant matching (`MatchKey`) keys commands at `run_command:<exe>[ <subcmd>]` with subcommand granularity for the known-subcommand set (git/mvn/npm/cargo/docker/gradle/aws) and executable granularity otherwise; file writes at `write:<subtree>` with descendant containment; other Class X at the coarse tool name. The `GrantStore` is in-memory, lineage-scoped (INV-10): a separate root session's store is empty, and `forSubAgent` mints a fresh empty child store (AC-10.6, RD-5). The approval seam is the injected `Approver` functional interface (T-1.1 supplies the UI; tests inject a stub). `mvn clean verify` is green: 353 unit tests (74 new across 6 permission test classes, including 15 adversarial tokenizer + 17 adversarial denylist tests), permission package **97.58% line** coverage, **BUNDLE 92.54% line**, the 0.80 JaCoCo gate met with **no new exclusion**. Two Minors (the literal "first non-flag token" subcommand parse for flags-with-arguments; the denylist's uncovered defensive arms), one Nit, two Discussion findings (the ADR's `rm`-of-non-empty-target and `mv`/`cp`-destination-exists rows need an FS probe the gate deliberately avoids — handled by a documented conservative simplification). No Blockers, no Majors.

### Minors

**[m1] [Spec-Adherence] `permission/MatchKey.java:forCommand` — "first non-flag token" is taken literally, so a flag's value can become the subcommand**
- Why: ADR-0004's grant algorithm step 3 says "take the first non-flag token as the subcommand", with no handling of flags that take a separate argument. Taken literally, `git -c color.ui=false status` keys on `color.ui=false` (the first non-`-`-prefixed token), not `status`. This is **conservative and safe** for the gate's only use of the key (auto-approval): a narrower/odd key re-prompts *more*, never less, and the destructive-denylist test runs independently of the match key, so no destructive command can ever be wrongly remembered. It is a Minor because the worked-example commands the ADR pins (`git status`, `mvn test`) all parse correctly; only the unpinned flag-with-argument shape diverges from a "smart" parse.
- Suggestion: leave as-is (matching the spec's literal algorithm avoids inventing unpinned spec content); or, if a future task wants tighter keys, the ADR could be amended to enumerate argument-taking flags. Recorded as a stated assumption. Tested by `MatchKeyTest.firstNonFlagTokenIsLiteral`.

**[m2] [Testing] `permission/DestructiveCommandDenylist.java` — defensive flag-variant arms partially branch-covered**
- Why: the class is 96.2% line (5 of 130 lines uncovered) but has 29 uncovered branches — individual flag-form arms in the per-executable matchers (e.g. the `--force-with-lease` long-form on `git push`, some `hasShortFlagLetter` paths). Every ADR *row* has a passing test (`rm -rf`, `git push --force`, `git reset --hard`, `git clean -fd(x)`, `chmod -R`, `kill -9`, `curl|sh`, fork-bomb, etc.); the missed branches are alternate spellings that route through already-tested code. The conservative bias means an untested variant errs toward flagging, never toward letting a destructive command through.
- Suggestion: optionally add a `@ParameterizedTest` enumerating each flag spelling per row to lift branch coverage; non-blocking, as the behavioral contract (each ADR row is denied) is fully covered and the bundle gate (0.80) is met at 92.54%.

### Nits

**[N1] [Readability] `permission/GateRequest.java` — fully-qualified `RunCommandTool` / `WriteFileTool` in the factory bodies**
- Why: `forCommand`/`forWrite` reference `com.srk.codingagent.tool.RunCommandTool.NAME` fully-qualified inside the method bodies while the class already imports those types in `PermissionGate`. Inside `GateRequest` they are not imported, so the FQN keeps the two `.NAME` references self-contained. Minor verbosity; author's choice.
- Suggestion: add the imports if the references recur. Non-blocking.

### Discussion

**[D1] [Discussion] ADR-0004 denylist row 1 ("rm of a non-empty target") and row 5 ("mv/cp whose destination exists") need filesystem state the gate cannot read at decision time**
- Why: the gate decides *before* execution and must not depend on mutable-FS state (a probe would be a TOCTOU race and would couple authorization to the filesystem). Two ADR rows are phrased in terms of FS state: "rm ... of a non-empty target" and "mv/cp whose destination exists (overwrite)". The implementation takes the conservative simplification the task note sanctions: any `rm` with a recursive/forced flag is flagged (a bare single-file `rm file` is *not* pattern-flagged — it still gates as Class X under the active mode), and any `mv`/`cp` with two-or-more non-flag operands (a real move/copy that *could* overwrite) is flagged. This over-flags (safe) rather than under-flags (unsafe), consistent with the ADR's "conservative pattern test; false positives prompt unnecessarily" guidance.
- Suggested amendment kind: optional `ac-update`/ADR clarification — the ADR table could state explicitly that these two rows are realized as pattern proxies (recursive/forced `rm`; two-operand `mv`/`cp`) because the gate has no FS access, so the design contract matches the code. No code change needed; surfaced so a future reader does not mistake the proxy for a gap. Recorded as a stated assumption.

**[D2] [Discussion] `permission/DestructiveCommandDenylist.java:containsForkBomb` matches a whitespace-stripped regex shape rather than a token sequence**
- Why: the fork-bomb row (ADR row 13, `:(){ :|:& };:`) has no clean token decomposition (`:` is both a function name and the no-op builtin; the shape is structural), so this one row is matched by stripping whitespace and testing the classic shape plus a generalized `name(){ name|name& };name` regex, rather than via the tokenizer like the other rows. It is conservative (it flags near-variants) and the two tested shapes pass, but it is the one matcher that does not consume `ShellTokenizer` output, which is worth noting for consistency.
- Suggested amendment kind: none (no spec change). Surfaced for awareness; a future hardening pass could broaden the fork-bomb shape set if real-world variants appear.

### Praise

**[P1]** The decision logic is single-sourced and reuses rather than re-declares: `OperationClass` (persistence, the T-0.6 Class R/X marker), `PermissionMode` (config, T-0.2), and `PermissionDecisionPayload`/`PermissionDecisionOutcome` (persistence, T-0.4) are all consumed from their existing homes — `GateDecision.toPayload` produces exactly the payload the loop logs, with no parallel enum or envelope. The INV-9 invariant ("a denylisted command never matches a Grant") is enforced *structurally* in the `GateDecision` canonical constructor (`denylisted && matchedGrant != null` throws), so a future caller cannot construct a decision that violates it — the invariant is a type-level guarantee, not just a runtime branch. The denylist's conservative bias is implemented faithfully (denylist test runs first; basename + case-fold + per-segment evaluation defeats `/usr/bin/RM`, `RM -RF`, quoted targets, and `mvn test && rm -rf target`), and the adversarial test suite exercises exactly the obfuscations the task flagged.

## Design-doc alignment

- **ADR-0004 (permission model):** classification (Class R auto in every mode; Class X gated; `run_command`'s command string additionally denylist-tested) → `PermissionGate.evaluate` orders Class-R-first, then denylist, then mode. The 4-mode table is implemented cell-for-cell. RD-1 grant matching (tokenize → basename → known-subcommand subcommand → match key) → `MatchKey`. RD-2 denylist seeded from the ADR table → `DestructiveCommandDenylist`. Grant lifecycle (lineage-scoped, sub-agents fresh) → `GrantStore`. Gate placement (between toolUse parse and execution; the loop consults it) → the gate returns a `GateDecision`; it does not dispatch (T-0.8) or render the prompt (T-1.1, the `Approver` seam).
- **AC-9.1..9.6 / AC-10.1..10.6:** AC-9.1 (exactly four modes — `gateHandlesAllFourModes`), AC-9.2 (READ_ONLY denies Class X), AC-9.3 (UNRESTRICTED auto except denylist prompt), AC-9.4 (ASK_EVERY_TIME prompts every time), AC-9.5 (ASK_ONCE prompts-then-remembers, excluding denylisted), AC-9.6 (Class R non-gated everywhere). AC-10.1 (exact-operation presentation — `GateRequest.presentation`), AC-10.2 (deny → no side effect, loop picks next step — the `GateDecision` DENY value), AC-10.3 (record grant on ASK_ONCE approve), AC-10.4 (denylist prompts regardless of mode, denied in READ_ONLY, never remembered), AC-10.6 (sub-agent independent, no inherited grants). AC-10.5 (record every approval/denial as a log event) is the loop's job (T-0.8); the gate supplies the `toPayload` data.
- **INV-8 / INV-9 / INV-10:** INV-8 (no Class X executes without a PERMISSION_DECISION) — satisfied at the gate level: the gate is the sole authorizer; a Class-X op's only `approved()` path is a gate decision (the loop, T-0.8, enforces the runtime ordering). INV-9 (a denylisted command is never auto-approved and produces/matches no Grant) — enforced both in `PermissionGate` (denylist branch records no grant) and structurally in the `GateDecision` constructor. INV-10 (a Grant is lineage-scoped; not read by a separate root session, not inherited by a sub-agent) — `GrantStore` per lineage; `forSubAgent` returns a fresh empty store.
- **CT-INV-7 / CT-INV-8 / CT-INV-9 / CT-SM-2:** all satisfied at the **gate-decision level** (the loop-level S3→S4/S3→S2 wiring + event emission is T-0.8, per the task's contract-test note):
  - **CT-INV-7 (INV-8, +):** `classXDecisionCarriesPermissionDecisionData` / `classRDecisionMapsToPayload` — every decision carries the operationClass + mode + outcome the loop logs; the gate is the only authorizer (no bypass path to `approved()`).
  - **CT-INV-8 (INV-9, −):** `denylistedNeverAutoApprovesViaGrant` — a denylisted command prompts every time and records/matches no grant even under ASK_ONCE; reinforced by the `GateDecision` construction guard (`GateDecisionTest.denylistedWithGrantRejected`).
  - **CT-INV-9 (INV-10, +):** `subAgentStoreStartsEmpty` / `grantCarriesItsLineage` — a sub-agent store is empty and a separate root session does not read another's grant.
  - **CT-SM-2 (A:T8, +):** `denyDecisionMapsToDenyPayload` / `deniedCallProducesDenyDecision` — a gate DENY is the value the loop turns into a TOOL_RESULT(denied) with no side effect (`approved()` is false).
- **Component C8 (02-architecture.md § 1.2):** classify every tool call, apply the active mode, enforce the denylist, collect approvals, remember grants; invariant "no Class X side effect without a gate decision; denylist always prompts" — all realized. The package lives at `com.srk.codingagent.permission` (§ 6 latitude), cohesive to C8.
