## Summary

T-2.2 implements compaction-with-derivation (component C6, ADR-0006, state-machine B). The new `Compactor` (`com.srk.codingagent.context`) runs the three-step flow ADR-0006 mandates: (1) a dedicated summary Converse call through the real `ModelClient` seam to the configured summarizer model with a fixed compaction system prompt that asks for outstanding task state, decisions, files touched, open work, and durable learnings (AC-18.4); (2) derivation of a NEW session (new sessionId, boundary-captured) seeded with the summary as an initial context block plus a configurable tail of recent verbatim turns, linked `edgeType = DERIVED_FROM` / `parentSessionId = original` on the child `.meta.json`, with a `COMPACTION(from,to,summaryRef)` marker appended to the child log (INV-4, AC-18.3, LT3); (3) preservation of the original unchanged — never edited, appended-to, or deleted (INV-5, AC-18.3, LT5, CT-INV-3). On the failure path (summary call fails, or returns no usable text) it appends an `ERROR` and returns a failed outcome carrying the context-exhausted exit code 5 (LT4 → LT7 → machine A T15, CT-SM-7).

The load-bearing INV-7 concern is handled: a `ContentBlock.Reasoning` variant (realizing the existing `content-block.schema.json` ReasoningBlock contract — schema unchanged) round-trips a reasoning `signature` verbatim through response → `ContentBlock.Reasoning` → persisted `MODEL_RESPONSE` → `SessionReplay` → `ConverseWireMapper.toRequest` (both directions wired). The recent-tail carryover re-appends the original's `USER_MESSAGE`/`MODEL_RESPONSE` payloads unchanged, so reasoning signatures (INV-7) and toolUse↔toolResult pairing (INV-6) survive into the derived seed. `mvn clean verify` is green (681 tests, +37 new; JaCoCo gate met). No Blockers, no Majors. One Minor (a small text-join helper near-duplicates a private `AgentLoop` helper across the loop↔context boundary).

Documented in the `Compactor` class Javadoc and `rationale_for_significant_choices`.
- **NOT the token-budget trigger arithmetic (T-2.1).** The `Compactor` consumes a `CompactionTrigger`; it does not re-implement the 0.85×window threshold (T-2.1's landed `TokenBudgetGuard`).
- **NOT the loop→compactor invocation (machine A T14 continue-in-derived).** Scope decision (recorded in `stated_assumptions`): the `AgentLoop`'s existing `surfaced(COMPACT)` is the correct LT2 hand-off from machine A to machine B; binding the loop to *invoke* the `Compactor` and restart in the derived conversation is run-driver orchestration (the REPL T-1.1 / brownfield driver T-1.6), not the loop's single-responsibility cycle. T-2.2's verify column is `CT-SM-6/7, CT-INV-3/4` — all four are machine-B / persistence assertions satisfied directly by the `Compactor` + persistence path. `AgentLoop` is left untouched (no regression to its heavily-tested cycle), all its existing tests stay green.
- **NOT prompt-cache placement (OQ-I, T-4.4)** and **NOT output disposal (US-19, T-1.5)** — untouched.
- **Schema not edited.** `content-block.schema.json` already defines the `reasoning` variant; T-2.2 realizes it in code (the test-resource schema copy is byte-identical to the design copy — confirmed by `diff`). No `design/` file edited other than this review.

### Minors

**[m1] [Reuse] `Compactor.java:295` `concatText` near-duplicates `AgentLoop.java:324` `finalText`.**
- Why: DRY (Fowler / rubric Maintainability). Both are ~9-line private static helpers that join a `List<ContentBlock>`'s `Text` blocks with `\n` into a `String`; the bodies are token-identical apart from the method name. They live in different components (C2 `loop` vs C6 `context`), and both are `private`.
- Suggestion: extract a package-shared helper (e.g. `ContentBlocks.joinText(List<ContentBlock>)` in `persistence`, beside `ContentBlock`) and have both call it. Deferred rather than done here because (a) it would require editing the heavily-tested `AgentLoop` and risking its coverage/tests for a trivial 9-line joiner, and (b) the two call sites carry distinct semantics (final-answer rendering vs summary-text extraction). Recorded as a Minor — does not block resolution; a clean candidate for a follow-up refactor when a third caller appears.

### Nits
None.

### Praise

**[P1] The reasoning-signature round-trip is asserted on the REAL contract, not a field-present check.** `CompactorTest.inv7_derivedSeedReSendsSignatureThroughWireMapper` takes a transcript with a reasoning block carrying a signature, runs compaction, reads the *derived* session back, replays it to `messages[]`, maps it through the real `ConverseWireMapper.toRequest`, and asserts the wire `reasoningContent.reasoningText.signature` is byte-identical to the original — exactly the directive's "assert the actual derived-session messages[] round-trips … byte-identical (not merely a signature field is present)." A dropped/mutated signature (the D2 class of live-only bug) fails this test.

**[P2] CT-INV-3 reads the raw parent bytes before/after.** `ctInv3_parentBytesUnchanged` and `ctInv3_parentUnchangedOnFailure` use `Files.readAllBytes` on the parent log before and after compaction (success AND failure paths) and `assertArrayEquals` — the literal byte-identity the contract pins, proving the COMPACTION/ERROR markers land on the child, never the parent.

**[P3] SUT-not-mocked discipline held end-to-end.** Every `CompactorTest` drives a real `Compactor` + real `SessionStore` (temp dir) + real `SessionReplay` + real `ModelClient` + real `ConverseWireMapper`; only the `BedrockRuntimeClient` is a hand-rolled scripted double (and a throwing double for the LT4 failure path). The summary call genuinely traverses the wire-mapping seam with a stubbed Bedrock response — no live AWS call, real contract exercised.
