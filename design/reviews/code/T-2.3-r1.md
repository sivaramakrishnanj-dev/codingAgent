## Summary

T-2.3 implements the sub-agent orchestrator (component C13, ADR-0010, in-process). `SubAgentOrchestrator.spawn(spec)` starts a **nested** child agent loop — reusing the existing C2 `AgentLoop` via an injected `ChildAgentLoopFactory`/`ChildAgentRun` seam, **not** a new loop — with: (1) its own isolated context (its own `messages[]` seeded from the scoped prompt inside the nested loop; AC-17.2); (2) its own session + a fresh `EventLog` (its own JSONL), sidestepping the shared-writer hazard at N=1 (ADR-0010 Notes); (3) a fresh `GrantStore` via `GrantStore.forSubAgent(childLineage)` so the child inherits none of the parent's remembered grants (AC-10.6, INV-10, CT-INV-9); (4) a concurrency bound of `NFR-SUBAGENT-MAX` (default 1) enforced by a `Semaphore` — the parent blocks on the single in-flight child at N=1, and the semaphore is the bounded-pool seam for N>1 (the parallelism stays config-gated, not built; AC-17.3, INV-12, CT-INV-10); (5) a wall-clock budget (600s default, overridable per spec) enforced via `Future.get(cap)` + `cancel(true)`, so an over-budget child is stopped and a **failure result** is returned to the parent — never a hang (AC-17.6); (6) lineage — each child is its own session linked `SPAWNED_BY` (on the child `.meta.json`), with `SUBAGENT_SPAWN`/`SUBAGENT_RESULT` events logged in the **parent's** log while the child's full transcript lives in the child's own JSONL (AC-17.5); and (7) **summary-only propagation** — the parent receives only a `SubAgentResult` summary block, never the child's transcript or per-turn events (AC-17.4, INV-11).

The `spawn_subagent` Class-X tool (`SpawnSubAgentTool`, `OperationClass.SIDE_EFFECTING`) routes through the permission gate like the other Class-X tools (ADR-0004) and returns the child's summary as a **plain `String`** — which the wire mapper sends on the Converse `text` member, never `json` (D2-safe). The directive-flagged wire concern is asserted on the real contract: a child tool-use cycle drives `tool_use → toolResult → end_turn` through the real `AgentLoop` + `ModelClient` + `ConverseWireMapper`, and the re-call request carries the child's plain-string toolResult on the `text` member with `json` null. `mvn clean verify` is green (718 tests, +34 new; JaCoCo gate met). No Blockers, no Majors, no Minors.

## Scope boundary held (noted explicitly per the task directive)

- **NOT N>1 parallel execution.** The `Semaphore(subAgentMax)` IS the bounded-pool seam; v1 ships `subAgentMax == 1` so the parent blocks on its single child. No multi-child parallel scheduling is built. A spawn that would exceed the bound is rejected with a failure result (CT-INV-10), never queued or run concurrently past the bound. The N>1 *shared-event-log synchronization* constraint (ADR-0010 Notes) is recorded as a future seam — the N=1 clean design gives the child its OWN `EventLog`, so no concurrent same-log append arises (documented in `ChildLoopContext`/orchestrator Javadoc + `rationale_for_significant_choices`).
- **NOT fork-based / separate-process sub-agents.** In-process only (OQ-C). The child shares the JVM (ADR-0010's accepted "no fault isolation" trade for N=1, documented in the orchestrator Javadoc).
- **NOT memory or compaction integration (T-2.4/T-2.5).** The child loop's `BudgetGuard` is wired by the factory; the orchestrator does not own compaction (a bounded-scope child typically goes L0→L5 without compaction, state-machine B Notes).
- **NOT the production composition-root wiring of `spawn_subagent` into the parent's registry.** That is pure wiring over a live Bedrock client (it belongs in the JaCoCo-excluded `AgentLoopFactory` / run-driver, per the task directive "keep new business logic OUT of the JaCoCo-excluded factory"). The tested deliverables — the orchestrator + budget/timeout/bound logic + the tool's input→spec→summary mapping — are exercised directly. Recorded in `rationale_for_significant_choices`.
- **Schema not edited.** `event.schema.json` enumerates `SUBAGENT_SPAWN`/`SUBAGENT_RESULT` in its `type` set but pins their `payload` only as a generic object (no dedicated `$defs`), exactly as it does for `ERROR`. A typed record that serializes to an object is schema-valid; no `design/06-formal` schema change is needed (no DCR). No `design/` file edited other than this review.

### Praise

**[P1] The D2 wire concern is asserted on the REAL contract, not by composition trust.** `SubAgentOrchestratorTest.childToolResultRoundTripsViaTextMember` drives a child `tool_use → toolResult → end_turn` cycle through a real nested `AgentLoop` + `ModelClient` + `ConverseWireMapper` (scripted Bedrock double), then inspects the captured re-call request and asserts the child's plain-string toolResult lands on `content.text()` with `content.json()` null — exactly the directive's "drive a child loop through a tool_use → toolResult → end_turn cycle with a stubbed Bedrock and assert the wire shape." Routing a String into `json` (the original D2 ValidationException) fails this test.

**[P2] INV-11 is asserted on both sides of the boundary.** `parentReceivesSummaryOnlyNotChildTranscript` runs a full child tool-use cycle and asserts the **parent** log carries ONLY `[SUBAGENT_SPAWN, SUBAGENT_RESULT]` — none of the child's `USER_MESSAGE`/`MODEL_RESPONSE`/`TOOL_USE`/`TOOL_RESULT` — while the **child's own** log carries the `TOOL_USE`/`TOOL_RESULT`, and the returned summary equals the child's final answer (not its transcript). The `SUBAGENT_RESULT` payload's serialized form is independently asserted to carry a `summary` but no `transcript`/`messages`/`events` field (`SubAgentPayloadCodecTest.subagentResult_carriesSummaryOnly`).

**[P3] The over-budget path is deterministic and never sleeps 600s.** `overBudgetChildYieldsFailureResult` injects a 150ms cap and a child run that blocks until interrupted, then asserts a failure result is returned in well under 5s — the AC-17.6 "rather than hanging" property, driven by the injected `Duration` budget seam (no real 600s sleep), exactly per the directive.

**[P4] The concurrency bound is asserted as a genuine race, not a counter peek.** `concurrencyBoundRejectsOverTheMaxSpawn` holds the single permit by blocking the first child inside its run on a latch, fires a second spawn from another thread, and asserts the second is rejected (`success==false`, names `NFR-SUBAGENT-MAX`) AND that the second child was never built (`childrenBuilt==1`) — proving the bound is enforced *before* a child runs, never "≤ max ran concurrently after the fact."

## Reuse / duplication

- **No pre-existing sub-agent / orchestrator to merge with.** A sweep for `Orchestrator`/`SubAgent`/`spawn`/`Semaphore` across `src/main` returns only (a) the new `subagent` package + payloads + tool, and (b) `GrantStore.forSubAgent` (the no-inherit factory the orchestrator *consumes*, whose Javadoc explicitly anticipated "The sub-agent orchestrator (T-2.3) gives a child a fresh GrantStore via GrantStore.forSubAgent"). The orchestrator is the first caller of that seam.
- **The loop / wire / persistence path is reused, not re-implemented.** The child runs the existing `AgentLoop` (over the existing `ModelClient` + `ConverseWireMapper`), opens its log via the existing `SessionStore.openLog`, and records its lineage via the existing `SessionStore.writeMeta` + `SessionMeta(...,parentSessionId,SPAWNED_BY,...)`. No parallel loop, wire, or persistence logic. The D2 text-member path is reused **by construction** (the child loop IS the parent's loop type) and additionally asserted.
- **The `Future.get(timeout)` + cancel idiom near `CommandExecutor`** is a different mechanism (subprocess tree-kill vs in-process loop interruption) operating on different resources; there is no extractable shared helper, and merging them would couple unrelated concerns. Not a finding.
- **`SubAgentOrchestrator.requireNonBlank` follows the established cross-package pattern**, not a gratuitous copy: `Payloads.requireNonBlank` is package-private to `persistence`, so `ResolvedConfig`, `GateRequest`, `CompactionRequest`, `OneShotRunner`, and others each carry an identical private `requireNonBlank`. The orchestrator (in `subagent`) follows that established convention. Not a finding.

## Design-doc alignment

- **AC-17.1** (spawn a well-scoped subtask, return a summarized result): `spawn(prompt)`/`spawn(spec)` run the nested child and return the `SubAgentResult` summary (`spawnReturnsCompletedSummary`, `spawnPromptConvenience`). ✓
- **AC-17.2** (own context window separate from the parent's; may run a different/cheaper model): the child is a nested loop with its own `messages[]`, its own `EventLog`, its own fresh gate; the resolved model is the parent's unless the spec overrides it (`modelOverrideForwarded`, `noOverrideInheritsParentModel`). ✓
- **AC-17.3 / INV-12** (concurrent sub-agents ≤ NFR-SUBAGENT-MAX, default 1): `Semaphore(subAgentMax)`; `concurrencyBoundRejectsOverTheMaxSpawn` (CT-INV-10) asserts at most 1 runs concurrently and the over-the-bound spawn is rejected without building a second child; `activeChildCountIdleAfterSpawn` asserts the permit is released after the child finishes. ✓
- **AC-17.4 / INV-11** (incorporate only the summarized result, not the transcript): `SubAgentResult` is the only thing crossing back; `parentReceivesSummaryOnlyNotChildTranscript` + `subagentResult_carriesSummaryOnly` assert the parent log/context never accrues the child's stream. ✓
- **AC-17.5** (record spawns + results as events, linked by lineage spawned-by): `spawnAndResultLoggedInParentLog` (parent log `[SUBAGENT_SPAWN, SUBAGENT_RESULT]`), `childMetaCarriesSpawnedByLineage` (child `.meta.json` `edgeType=SPAWNED_BY`, `parentSessionId=parent`), `SubAgentPayloadCodecTest` round-trips. ✓
- **AC-17.6** (fail/over-budget → failure result, decide a next step rather than hang): over-budget (`overBudgetChildYieldsFailureResult`, deterministic 150ms cap), child throws (`failingChildYieldsFailureResult`, `failingChildWithNullMessageYieldsFailureResult`), surfaced edge (`surfacedChildMapsToFailureResult`). The parent always gets a `SubAgentResult`, never an exception/hang. ✓
- **AC-10.6 / INV-10 / RD-5** (no inherited grants; `GrantStore.forSubAgent`): `childDoesNotInheritParentGrants` (CT-INV-9 — the child's fresh store is empty and does not contain the parent's remembered key) + `childGatePromptsForOpParentRemembered` (the child gate prompts for an op the parent had remembered, proving the grant did not carry over). ✓
- **ADR-0010** (in-process nested loop; isolation; bounded concurrency seam; budget; lineage; summary-only; no inherited grants; shared-writer N>1 constraint): all realized; the child gets its OWN `EventLog` (the N=1 clean design that sidesteps the shared-writer hazard), with the N>1 synchronization constraint recorded as a future seam in the orchestrator/context Javadoc + rationale. ✓
- **D2 (T-0.5-RD2)** (plain-string toolResult → Converse `text` member): reused by construction (the child IS the real loop) and asserted on the captured re-call request (`childToolResultRoundTripsViaTextMember`). The `spawn_subagent` tool's own result is a plain `String` (`returnsChildSummaryAsPlainString`). ✓
- **state-machine A T19 / S4** (`spawn_subagent` is a tool the loop dispatches at S4; its result is a TOOL_RESULT carrying the summary): `SpawnSubAgentTool` is Class X (gated at S4) and returns the plain-string summary the loop wraps as a TOOL_RESULT. ✓
- **Component ownership** (C13 sub-agent orchestrator, `com.srk.codingagent.subagent` per 02-architecture § 6): the orchestrator + spec/result/context/factory live in `subagent`; the persisted payloads live in `persistence` (beside the other `EventPayload`s); the tool lives in `tool` (beside the other `ToolHandler`s, so it can use `ToolInputs`/`ToolSchemas`). Recorded in `rationale_for_significant_choices`. ✓
