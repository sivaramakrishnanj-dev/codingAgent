## Summary

T-3.7 implements approved amendment DCR-6 in two halves, both pure `src/main` + `src/test` (no `design/`
edit, no contract change — the contract landed at e5e9b34).

**Half B — the gate (`TaskTraceability`, C3).** A miscounting-only hardening that changes recognition
COVERAGE, not STRICTNESS. The four real-breakdown shapes the original single-line parser miscounted are
now counted/flagged correctly: (i) a repeated id is de-duplicated in `untracedTasks` (a distinct
`LinkedHashSet`); (ii) an arrow/sequencing-diagram line (`T-1 -> T-2`, `T-1 → T-2`) is skipped, not read
as a task; (iii) a range heading `T-3 through T-8` expands so `T-3`..`T-8` are each individually
recognized and each flagged untraced if the heading line carries no on-line ref (not silently collapsed
to only `T-3`); (iv) a bold-wrapped table-cell id `| **T-1** |` is recognized as the bare `T-1`. The id
form (`T-<n>`/`T-<n>.<m>`, mandatory hyphen) and ref vocabulary (`US-`/`AC-`/`NFR-`/`RD-`/`INV-`) are
unchanged, the strict same-line-ref rule holds (a task whose own line lacks a ref is still flagged even
if a sibling `**Refs:**` line carries one), and no loose block scan is introduced (DCR-5 Option b stays
rejected). `check()` and `tasksInOrder()` share a single private `recognizeTaskIds(line)` — the
directive's "one source of truth for what is a task line" — so both agree on recognition.

**Half A — the prompt (`GreenfieldPlaybook` TASKS arm).** The tasks-phase prompt now forces a single
canonical single-line task row (`T-<n>: <description> (AC-x, US-y)`, one task per own line, ref inline)
and explicitly forbids the three shapes a strict line-by-line gate cannot count as the author intends:
range headings (`T-3 through T-8`), multi-line `**Refs:**` blocks (the symbol must be inline on the
task's own line), and arrow/sequencing-diagram-as-task-list (`T-1 -> T-2`). The existing
hyphen-mandatory + cite-a-symbol + DCR-5 vocabulary instructions are preserved; the `systemPrompt(phase)`
signature and the REQUIREMENTS/DESIGN/IMPLEMENT arms are untouched.

### Praise

**[P1] One source of truth for task-line recognition.** Both public methods (`check` and `tasksInOrder`)
delegate to a single private `recognizeTaskIds(line)` that owns all four DCR-6 coverage rules. This
directly honours the directive's "one source of truth for what is a task line (rather than a second,
drifting parser)" — a range heading expands, an arrow line is skipped, and a bold cell is recognized
identically for both callers, pinned by mirrored tests on each. A future shape change lands in one place.

**[P2] Coverage widened without touching strictness.** The id capture group `(T-\d+(?:\.\d+)*)` and the
`REQUIREMENT_REF` vocabulary are byte-for-byte unchanged; the new emphasis-wrapper prefix is `(?:\*\*|__)?`
outside the capture group, so the recognized id is still the bare `T-1`. The DCR-5 regression tests
(`hyphenLessTaskIdIsNotRecognized`, `priorLiveFailingBreakdownIsRefused`) stay green, demonstrating the
hyphen-mandatory strictness and the same-line-ref rule are preserved while only COVERAGE grows.

### Note on C1 — `NumberFormatException` robustness gap (Phase-C-caught, resolved in-session)

During the Phase C Reliability/Correctness lens I caught a defect in my own Phase A: the first cut of
`expandRange` did `Integer.parseInt` on the regex-captured (unbounded `\d+`) range bounds, so a heading
like `## T-1 through T-99999999999` would propagate an uncaught `NumberFormatException` out of `check()` /
`tasksInOrder()` rather than refuse the malformed range cleanly — a reliability gap on the model-generated
markdown the gate consumes. Because the fix is a self-contained guard and the round-trip would produce an
identical end state, I hardened it within this pass (rather than emit `needs-fix`): `exceedsMaxBound`
rejects any bound by digit-width before parsing (so an arbitrarily long capture cannot throw) and by the
numeric `MAX_RANGE_BOUND` cap, returning no task for an out-of-range heading — consistent with the
descending-range handling. Pinned by `implausiblyLargeRangeBoundIsRejectedCleanly` and
`parseableButOverBoundRangeIsRejected`. Re-ran `mvn clean verify` after the fix: 1118 tests green, coverage
gate met. This is disclosed here for audit traceability; the frozen Phase A reviewed below contains the
hardened form, so the final verdict carries zero open Majors.
