## Summary

T-4.3 completes the C5 capability layer (ADR-0002, OQ-J). `ModelCapabilityProfile` is extended from the
T-2.1/T-4.2 minimal form (window + the two multimodal flags) to the full ADR-0002 / 03-data-model § 2.6
shape: `providerFamily` (new `ProviderFamily` enum), `supportsExtendedThinking` + `thinkingBudgetConfigurable`,
`supportsToolUse`, `promptCache` (new nullable `PromptCacheCaps` value type with a `TimeToLive` enum whose
`@JsonValue` wire tokens are the schema's `5m`/`1h`), and `inferenceParamPassthrough`. The two
backward-compatible constructors (window-only; window + image + document) are preserved and delegate to the
canonical constructor with the conservative-default shape, so existing callers (`TokenBudgetGuard`/its test,
`AttachmentResolver`/its test, `ReplRunner`/its test, `AgentLoopFactory`) are unchanged. The prefix-keyed
resolution that was inline in the JaCoCo-excluded `AgentLoopFactory` is moved into a new, coverage-counted
`ModelCapabilityRegistry`: a known Claude id resolves to the populated ANTHROPIC profile (extended thinking,
tool use, image + document input, the 200K window, `top_k` passthrough, a prompt cache whose checkpoint
minimum is 4096 for Opus and 1024 for non-Opus per ADR-0002), and an unknown id degrades to the conservative
default (OTHER, no thinking, no prompt cache, no image/document input, tool-use assumed, the supplied
safe-minimum window). `ModelCapabilityProfile.forModelId` is retained as a thin delegate to the registry so
there is a single resolution path. `AgentLoopFactory` now calls `ModelCapabilityRegistry.resolve(...)` at both
sites. CT-SCH-15 is realized as a schema-oracle contract test (both a Claude profile and the conservative
default validate against the committed `model-capability-profile.schema.json`).

### Minors

**[m1] [Readability] `AgentLoopFactory.CONSERVATIVE_DEFAULT_CONTEXT_WINDOW_TOKENS` Javadoc now mis-states what
T-4.3 did.** `AgentLoopFactory.java:79-82` says "T-4.3 (the full capability registry + schema validation)
sources this from config `NFR-MODEL-CONTEXT-WINDOW` instead; until then it is a compiled-in default in this
composition root." T-4.3 built the registry + schema validation but deliberately kept the conservative-default
window as a compiled-in constant (option (a) — design/-clean; the config-key promotion is deferred, see D1).
The comment, written by T-2.1/T-4.2 as a forward-reference, now promises a behavior this task chose not to
implement, so it reads as a lie (Clean Code: comments must stay true). Not a correctness issue and in
JaCoCo-excluded wiring.
- Suggestion: on a future touch, reword to "T-4.3 kept this as a compiled-in safe-minimum; promoting it to a
  config key per ADR-0002 + closing T-2.1 D2 is deferred (see open-questions)." Left for the next round / a
  follow-on task rather than re-opening Phase A for a comment-only edit this round.

### Praise

**[P1] Schema-as-oracle for CT-SCH-15, with literal-JSON negative anchors.** The positive cases serialize the
profiles the registry actually resolves and validate them against the committed schema; the negative cases
(unknown `providerFamily`, missing required field, bad `ttls` token, extra property) validate *literal JSON*
against the schema branches, so the oracle is the spec artifact, not the code — the T-0.4/T-4.2 pattern the
task cited.

**[P2] Resolution consolidated, not duplicated.** The inline `forModelId` body moved out of the JaCoCo-excluded
`AgentLoopFactory` into the coverage-counted `ModelCapabilityRegistry`, and `forModelId` was kept as a thin
delegate — so there is exactly one resolution path and the logic is now unit-coverable (100% line) outside the
composition root, exactly as the task asked.

**[P3] Unpinned figure kept off the oracle surface.** `CLAUDE_MAX_CACHE_CHECKPOINTS = 4` is a verified
Anthropic figure that no spec symbol pins, so no test asserts its value (the tests assert only `promptCache != null`
and the ADR-0002-pinned 4096/1024 minima). This keeps every assertion traceable to a spec symbol.

### Discussion

**[D1] [Discussion] Conservative-default context window is still a compiled-in constant, not a config key
(NFR-MODEL-CONTEXT-WINDOW / T-2.1 D2).** ADR-0002 says the conservative-default window should be "read from
config `NFR-MODEL-CONTEXT-WINDOW`", and two open-questions entries (T-2.1 D1: per-family window figures not
pinned by a spec symbol; T-2.1 D2: conservative-default window read-from-config deferred to T-4.3) flagged the
config-key promotion for this task. T-4.3 chose **option (a)** from the task brief: keep the safe-minimum as a
documented compiled-in constant (`AgentLoopFactory.CONSERVATIVE_DEFAULT_CONTEXT_WINDOW_TOKENS = 100_000`, and
the Claude window `ModelCapabilityRegistry.CLAUDE_4_FAMILY_CONTEXT_WINDOW_TOKENS = 200_000`). Rationale: the
live default model resolves to a real Claude window through the registry, so the constant is off the production
happy path; and adding a `defaultModelContextWindowTokens` config key would require editing
`design/06-formal/resolved-config.schema.json` (which has `additionalProperties:false` and is the designer's
lane), so doing it here would breach the "no `design/` edits except the review file" rule. Promoting the
window to a config key remains a worthwhile follow-up.
- spec_refs: NFR-MODEL-CONTEXT-WINDOW, ADR-0002, (open-questions) T-2.1 D1 / T-2.1 D2
- suggested_amendment_kind: schema-update (add `defaultModelContextWindowTokens` to the resolved-config schema +
  `ConfigKeys`/`ConfigDefaults`/`ResolvedConfig`, then source the registry fallback from it)
- Non-blocking; the user decides post-task. Surfacing here is the expected non-blocking lane for the T-2.1 D2
  follow-on (per the task brief's config-key note, option (a)).

## Reuse / duplication
Reuse sweep clean. `ProviderFamily`, `PromptCacheCaps`, and `TimeToLive` are net-new types with no existing
twin. Capability resolution lives in exactly one place (`ModelCapabilityRegistry`); the inline `AgentLoopFactory`
path was *moved* there, not duplicated, and `forModelId` delegates rather than re-implementing. The
`@JsonValue` enum → wire-token idiom in `PromptCacheCaps.TimeToLive` follows the established house pattern
(`persistence/ToolResultStatus`, `CompactionTrigger`, `PermissionDecisionOutcome`) — same convention, distinct
enum + token set, so consistency rather than duplication.
