# ADR-0012 — Greenfield workflow formality: full spec-driven structure

## Status

accepted (2026-06-15); amended 2026-06-23 (DCR-1) — phase-deliverable persistence is now driver-authored in code rather than model-tool-dependent; amended 2026-06-23 (DCR-2) — each pre-approval phase is a **multi-turn conversation** that converges the deliverable, **approve = finalize**, and the greenfield Converse request sets an explicit output-token budget; amended 2026-06-23 (DCR-3) — greenfield phase-state is **resumable**: a fresh greenfield run re-derives which phases are approved from the target repo's on-disk approval-stamped artifacts and resumes at the first unstamped/absent phase; amended 2026-06-24 (DCR-5) — the greenfield playbook prompt **emits the strict traceability gate's vocabulary** on both the requirements and tasks phases so a greenfield project self-conforms to the gate, which stays strict (no `TaskTraceability` regex change); amended 2026-06-24 (DCR-6) — DCR-5's "the gate is unchanged / no regex relaxed" claim is **qualified to "unchanged in STRICTNESS"**: a miscounting-only recognition-COVERAGE hardening (dedup repeated ids, skip arrow/sequencing-diagram lines, expand range headings, recognize bold-wrapped ids in table rows) is now permitted while the strict same-line-ref guarantee and the rejection of a loose block-scan (DCR-5 Option b) both stand; DCR-6 also tightens `write_artifact` containment to the known design-doc artifacts (rejecting source paths under `design/`, closing the AC-1.4 pre-approval source-write hole); amended 2026-06-25 (DCR-7) — the greenfield IMPLEMENT phase adopts a **verify-at-end / testable-only** verification model: each task is marked complete **on implementation** (durable on-disk marker, read back on resume), verification runs **once at the end of the phase** (not per task; tasks not independently testable are implemented without per-task verification), a failing end verify retries bounded by `NFR-VERIFY-MAX-ITERATIONS` then stop-and-surface, **no configured test command → complete-with-warning** (terminal success, exit 0, no re-prompt loop), and **intra-IMPLEMENT resume skips completed tasks** (resume at the first incomplete task, not `T-1`). A new **AC-3.6** carries the no-test-command behavior that three code sites previously **mis-cited as AC-20.6** (see Decision + Notes).

## Context

Greenfield mode (US-1/2/3, C3) takes a project from idea → requirements → design → tasks → implementation. RD-7 fixed that it persists requirements/design/tasks as markdown with approval gates between stages. **OQ-B** asks *how formal* that structure is: a lightweight scaffold or the full spec-driven ceremony (EARS acceptance criteria, numeric NFRs, ADRs, formal contracts, milestone/task breakdown with traceability). **User chose full spec-driven structure.**

There is a pleasing reflexivity here: greenfield mode is essentially the workflow that produced *this* design. The agent runs a designer playbook.

## Decision

**Greenfield mode implements the full spec-driven structure — the same phase-gated methodology used to design codingAgent itself.**

- **Phases + artifacts** (persisted as markdown in the *target* project, per RD-7):
  1. **Requirements** — personas + user stories → **EARS** acceptance criteria → numeric **NFRs**.
  2. **Design** — overview → architecture (+ **ADRs**) → data model → APIs → operations.
  3. **Formal contracts** — schemas, state machine, exit codes, contract tests, fixtures (when the project shape warrants).
  4. **Tasks** — milestones + task breakdown with **traceability** (each task → AC/NFR).
- **Multi-turn phase dialogue, approve = finalize (AC-1.1, AC-2.3, AC-1.5, AC-2.4 — amended DCR-2, 2026-06-23).** Each pre-approval phase (requirements, design, tasks) is a **multi-turn conversation**, not a single model turn. The developer converses with the agent across several REPL turns to shape the phase's deliverable; the model refines the deliverable each round and may ask clarifying questions (AC-1.1 — it now has room to, given a terse idea). **The phase transcript carries across turns *within* the phase** — the model sees its own prior turns within the phase, fixing the fresh-conversation-per-phase in-phase discontinuity that made the single-turn shape capture *questions* rather than a converged deliverable. **Finalize = approve, one gesture** (reuse the existing `ArtifactApprovalGate` / `InteractiveGreenfieldApproval`): each round the developer is offered the approval prompt; a **non-approve** answer is *not* persist-and-stop — it keeps the phase conversation going (another refining turn, which also satisfies AC-2.4's revise-and-re-request). When the developer **approves**, that approval IS the finalize signal: the driver captures the model's latest substantive deliverable text in the phase conversation, persists it (the driver-authored path below), records the AC-1.5 approval timestamp, and advances to the next phase (AC-2.3 — implementation begins only after the design + task breakdown are approved).
- **Driver-authored phase deliverables (AC-1.2, AC-2.1 — amended DCR-1, 2026-06-23; persistence trigger refined by DCR-2).** The greenfield workflow driver (C3) **authors each pre-approval phase deliverable deterministically in code**, not via a model-emitted tool call. On **phase approval** (the finalize signal above; DCR-1 originally triggered this on each phase's first `END_TURN`, DCR-2 moves the capture-and-persist trigger to approval so the *converged* deliverable is what is written), the driver captures the model's latest substantive deliverable text in the phase conversation and writes it to the phase artifact (the target project's `design/00-requirements.md` / `design/01-design.md` / `design/02-tasks.md`) via `GreenfieldArtifactStore.write()` — a truncating write of the composed content; the `ArtifactApprovalGate` then appends the AC-1.5 approval stamp. Persistence is therefore **driver-guaranteed** — it does not depend on the live model reliably emitting a `write_artifact` `toolUse`, which empirically it does not (the model answers in prose and stops; see Context). The `write_artifact` design-doc tool stays **registered/available** in the pre-approval registry (C7) but is **optional — no longer the persistence mechanism**.
  - **Cross-phase transcript continuity (DCR-1, kept).** Later phases **inject the approved earlier-phase artifact content into their phase's conversation context** (requirements → design → tasks), so design and tasks are authored against the actual approved upstream content rather than a discontinuous fresh start. DCR-2 adds *in-phase* continuity (the phase's own turns carry within the phase); DCR-1's *cross-phase* injection is retained unchanged.
  - **AC-1.4 preserved — write_artifact containment tightened to the known design-doc artifacts (DCR-6, T-3.6).** The driver's in-code artifact write is confined to the target project's `design/` markdown; **source-write Class X tools (`write_file` / `edit_file`) stay withheld** from the pre-approval phase registry across *every* turn of the multi-turn dialogue, until the breakdown is approved. The change is to *who persists the deliverable* (driver, in code) and *how the phase converges* (multi-turn) — not to *what may be written before approval* (still only `design/` markdown). **DCR-6 closes a residual hole in that confinement:** `GreenfieldArtifactStore`/`WriteArtifactTool` previously confined a write only by `resolved.startsWith(<workspaceRoot>/design)` — with no allowlist — so a model-supplied source path *under* `design/` (e.g. `design/impl/pom.xml`, `design/impl/src/main/java/.../Calculator.java`) resolved under `design/` and passed the check, reaching `store.write()` even though the class Javadocs already promise the tool "cannot write source files" and AC-1.4 forbids any Class-X op against source files pre-approval. DCR-6 restricts `GreenfieldArtifactStore`/`WriteArtifactTool` to **only the three known design-doc artifacts** (`design/00-requirements.md`, `design/01-design.md`, `design/02-tasks.md`) and **rejects any other path under `design/`** (including a source path under `design/impl/**`), so the code now matches the Javadoc/AC-1.4 contract. No contract change — this tightens an existing confinement that was looser than its own promise. Implemented by `07-tasks.md` task **T-3.6** (M3, deps T-3.2).
- **Greenfield-phase output-token budget (DCR-2, 2026-06-23 — D1 follow-on folded in).** The greenfield phase Converse request sets an explicit `inferenceConfig.maxTokens` so a full requirements / design / tasks deliverable is not truncated at the backend's default 4096 output-token cap (a live run observed `stopReason=MAX_TOKENS` mid-deliverable; the Converse request previously set no `inferenceConfig`, so the backend default applied — see `02-architecture.md` § 2.1 + C4). **Chosen value: 16384 tokens (16K), configurable.** Rationale: a full phase deliverable is realistically a few thousand tokens of markdown (occasionally up to ~8K for a rich design doc), so 16K is ~2–4× headroom; it stays well within the Claude Opus 4.x output ceiling across the whole family (Opus 4.8/4.7/4.6 = 128K max output, Opus 4.5 = 64K, legacy 4.1 = 32K — verified 2026-06-23 from platform.claude.com), and it is small enough not to invite runaway generation. **This is the model's output-token cap for the generation, distinct from `NFR-OUTPUT-MAX-INLINE`** (16 KB), which governs how much *tool/command output* enters context (a disposal threshold, ADR-0006) — different axis, coincidentally similar magnitude.
- **Traceability mandatory (AC-2.5).** Every task traces to ≥ 1 requirement; the methodology's chain (US → AC → NFR/ADR → task) is preserved. Traceability is verified against the **driver-written** task-breakdown artifact.
- **The playbook prompt emits the gate's vocabulary; the gate stays strict (AC-2.2, AC-2.5 — amended DCR-5, 2026-06-24).** The traceability gate (`TaskTraceability`, the `ArtifactApprovalGate`'s tasks-phase check) is **strict by design**: it recognizes a task only when a line carries a stable id of the form `T-<n>` / `T-<n>.<m>` (the hyphen is mandatory), and it counts a line as traced only when it cites at least one requirement symbol from the fixed vocabulary `US-<n>` / `AC-<n>(.<m>)` / `NFR-<NAME>` / `RD-<n>` / `INV-<n>`. This strictness is the formal traceability guarantee this ADR deliberately chose over the lightweight scaffold (the rejected alternative below) — it pins the full-rigor US → AC → NFR/ADR → task chain and keeps greenfield reflexively consistent with the methodology that built codingAgent itself. **The burden of conformance therefore sits on the playbook prompt, not on a relaxed gate.** The greenfield playbook (the per-phase system prompt, C3 `GreenfieldPlaybook`) **emits** the gate's vocabulary on both pre-approval authoring phases:
  - **Requirements phase:** the prompt directs the model to author acceptance criteria as **numbered `AC-<n>.<m>` symbols**, user stories as **`US-<n>`**, and non-functional requirements as **`NFR-<NAME>`** — the gate-recognizable requirement-symbol shapes. A greenfield project's own model-authored requirement symbols thus become the traceability catalog its tasks trace to.
  - **Tasks phase:** the prompt directs the model to give each task a stable id of the form **`T-<n>` / `T-<n>.<m>`** (hyphen mandatory) and to make **each task line cite ≥ 1 requirement symbol authored in the requirements phase**.
  - **The gate is unchanged in STRICTNESS (qualified by DCR-6).** No `TaskTraceability` regex is relaxed in the sense that matters for the formal guarantee: which lines count as tasks (`T-<n>`/`T-<n>.<m>`, hyphen mandatory) and which refs count as traces (`US-`/`AC-`/`NFR-`/`RD-`/`INV-`) is not loosened, and the same-line-ref rule is **not** loosened into a block scan (the rejected alternative — broaden `TASK_LINE` to accept hyphen-less `T1`/`T10` and `REQUIREMENT_REF` to accept bare `R<n>`, i.e. DCR-5 Option b — would discard the strict guarantee and is recorded as rejected below; it stays rejected). The DCR-5 fix is a production-prompt change plus regression tests pinning the exact live-failing cases (`T1`/`T2`/`T10` ids citing `R1`–`R6` refs, which the strict gate correctly refuses), so the defect can never silently regress; there is no schema / contract-test / data-model blast radius. This closes the defect where a live greenfield model emitted hyphen-less ids citing `R<n>` refs — neither in the gate vocabulary — so the strict gate correctly refused `0-tasks`/untraceable and greenfield never reached implement (the existing `TaskTraceabilityTest` only exercised gate-conformant forms, so the unit suite was green while live model output failed — a defect class those tests could not catch by construction).
  - **A miscounting-only recognition-COVERAGE hardening is permitted (DCR-6, T-3.7).** DCR-5 said "the gate is unchanged"; DCR-6 qualifies that to **unchanged in STRICTNESS**. The strict gate is correct on *which* lines are tasks and *which* refs are traces, but it **miscounts the shapes a real (Sonnet-style) breakdown contains**: (i) a repeated id is double-counted in the untraced set (no dedup); (ii) an arrow/sequencing-diagram line `T-1 -> T-2` is read as a single task (only the first id captured); (iii) a range heading `T-3 through T-8` recognizes only `T-3`, silently dropping `T-4..T-8`; (iv) a bold-wrapped id in a table row `| **T-1** |` is not recognized at all (the `**` defeats the list/heading/table prefix). DCR-6 permits a hardening that fixes exactly these: **dedup repeated ids; skip arrow/sequencing-diagram lines; expand range headings so each task in the range is individually recognized — and correctly flagged if untraced, not silently collapsed; recognize bold-wrapped ids in table rows.** This changes recognition **COVERAGE, not strictness** — every shape it adds is still held to the same strict same-line-ref rule (a recognized task is traced only if its own line carries a valid `US-`/`AC-`/`NFR-`/`RD-`/`INV-` ref). The **strict same-line-ref guarantee and the rejection of a loose block-scan (DCR-5 Option b) both stand.** Single-line task rows are guaranteed by the **prompt** (the greenfield TASKS prompt is extended to force one canonical single-line task row per task and to forbid range headings / multi-line `**Refs:**` blocks / arrow-diagram-as-task-list); the gate hardening only stops the parser MISCOUNTING the shapes a real breakdown contains. Implemented by `07-tasks.md` task **T-3.7** (M3, deps T-3.5) with regression tests pinning all four failing shapes plus a full Sonnet-style breakdown that now passes.
- **Implementation — verify at end / testable-only (US-3, AC-3.x — amended DCR-7, 2026-06-25).** Tasks are implemented one at a time in breakdown order, and **each task is marked complete ON IMPLEMENTATION** (a durable on-disk completion marker in the task-breakdown artifact, AC-3.3) — not on passing an individual verify. **Verification runs ONCE AT THE END OF THE PHASE** (AC-3.2), not after each task: the greenfield implement phase is a **flat task list with no milestone substructure**, so the verify boundary is end-of-phase (a single configured build/test run after the last task). The end verify is **testable-only** — tasks that are not independently testable (e.g. an early scaffold, or a `pom.xml` not buildable until later tasks land) are implemented **without** per-task verification; only the end-of-phase verify gates the phase. A failing end verify **retries bounded by `NFR-VERIFY-MAX-ITERATIONS`** then **stop-and-surface** (AC-3.4, AC-20.5). **No configured test command → the end verify is skipped with a single warning, every task already implemented + marked complete, and the phase terminates deterministically (no re-prompt loop)** — a **complete-with-warning** terminal success (exit 0), consistent with the brownfield no-verify precedent; it is **not** a hard-stop and **not** a re-loop into a fresh implement attempt (AC-3.6 — this is the correct AC that three code sites previously mis-cited as AC-20.6; AC-20.6 is "prefer configured named commands over ad-hoc command strings", a different requirement). **Intra-IMPLEMENT resume skips completed tasks** — on a greenfield re-entry whose reconstructed phase is IMPLEMENT, the driver reads back the per-task completion markers and resumes at the **first incomplete task**, terminating rather than restarting at `T-1` (AC-7.6). This verify-at-end / testable-only model is what a scaffold-first breakdown actually needs: scaffold → `pom.xml` → testable code, verified once at the end, rather than hard-stopping at the not-yet-buildable scaffold under a per-task verify.
- **Gating before source writes (AC-1.3, AC-1.4).** Across every turn of the requirements/design multi-turn dialogue, the agent does not write source — only design markdown — until the breakdown is approved. (The "only design markdown" write is the driver-authored persistence above, performed at phase approval; source-write tools stay withheld for the whole pre-approval dialogue.)
- **Greenfield mid-flow resume — re-derive phase-state from stamped artifacts (AC-7.6, US-7 — amended DCR-3, 2026-06-23).** Greenfield phase-state is **resumable**, and the durable resume state is the **on-disk approval-stamped artifacts themselves** — no separate phase-state persistence is introduced. The contract:
  - **The AC-1.5 approval stamp is the phase-approved marker.** A phase whose target-repo artifact (`design/00-requirements.md` / `design/01-design.md` / `design/02-tasks.md`) bears the AC-1.5 approval stamp is an **approved** phase; the **current** phase is the first one whose artifact is **unstamped or absent**.
  - **A fresh `codingagent --mode greenfield` reconstructs its phase-state on session start** by reading the target repo's `design/` artifacts and applying the stamp rule above, then **resumes at the first unstamped/absent phase** rather than restarting at requirements. A greenfield run over a project with an approved requirements artifact resumes at the design phase; a project with approved requirements + design resumes at tasks; and so on.
  - **Transient mid-phase failure is retryable in place.** A model-backend timeout (or any non-approval interruption) mid-phase leaves that phase's artifact **unstamped** (approval is the only thing that stamps — see the multi-turn / approve-to-finalize bullet), so the *next* greenfield prompt re-derives the same phase as current and **re-enters it** — there is no separate retry contract to invent. Retry-in-place and resume-a-prior-project fall out of the **same** stamp-driven mechanism.
  - **Keyed to the target project (AC-7.3, brought forward).** Resumable greenfield sessions are scoped to the target repository by the **real AC-7.3 repo key** (git remote URL when present, else normalized absolute path; see ADR-0005), replacing the fixed `Main.ONE_SHOT_LINEAGE` M0 placeholder that made distinct runs over different target projects collide. Bringing the real repo-keying forward is part of this decision (it is the root cause of the run-collision symptom); see `02-architecture.md` § 1.2 (C15).
  - **The AC-1.5 stamp serves two roles from one durable fact.** It is simultaneously (a) the **resume signal** (this phase is approved → skip it on resume) and (b) the **clobber-protection signal**: `GreenfieldArtifactStore.write()` refuses to truncate an already-approval-stamped artifact (raising `ApprovedArtifactProtectedException`), so a fresh run can no longer silently overwrite a prior run's approved deliverable. The clobber guard is a safety stopgap derivable from AC-1.2/AC-1.5 (one durable on-disk fact — the stamp — protects the approved deliverable and marks the phase resumable).
  - **Accepted tradeoff — in-phase transcript is not preserved across an interruption.** Resume re-derives only **phase boundaries** from the artifacts, not the multi-turn in-phase conversation. An interrupted mid-phase dialogue **loses its in-phase turns**; resume re-enters at the phase boundary and the developer re-converses (the prior phases' approved artifacts are still injected as upstream context per the cross-phase-continuity bullet, so only the *current* phase's in-flight turns are lost). This is the user-accepted cost of reusing the on-disk artifacts as the resume state rather than persisting the full phase transcript (the rejected Option B in the DCR persisted phase events instead — heavier surface).
- **Reuse the engine.** This is a **workflow driver** (C3) over the same Agent Loop (ADR-0001), tools (file/exec/memory), and permission gate (ADR-0004) — not a separate engine. The "phases + gates" are orchestration on top of the loop.

## Consequences

**Positive**
- Maximally rigorous, reviewable output — the project gets a real design baseline, not just code.
- Reflexive consistency: greenfield ≈ the designer methodology, so the agent's own design process is the reference implementation (dogfooding).
- Traceability + per-phase gates give the developer tight control and auditable artifacts (pairs with the event log, ADR-0005).

**Negative / costs**
- **Heavy**: many phases, many gates, many model turns — slow for a user who "just wants to start a project." This is the explicit tradeoff the user accepted over the lightweight scaffold.
- More surface to build + test in v1 (the full phase machinery) than a scaffold.
- Risk of ceremony fatigue → mitigated by the gates being approvals (fast yes/no), and the agent doing the drafting.

**Neutral / follow-ons**
- A lighter or configurable greenfield rigor (the rejected alternatives) remains a future option if the full ceremony proves too heavy in practice — the workflow-driver seam allows it.
- Phase 4 task breakdown will size the greenfield driver as a substantial milestone (it's effectively a methodology engine).
- Greenfield artifacts in the *target* project are separate from codingAgent's own `design/` — the agent writes them into the user's repo.
- **Driver-authored persistence (DCR-1) shifts artifact composition from model to driver:** the model still *produces* the deliverable prose, but the driver *captures and writes* it. This makes persistence a real, mock-stable contract a test can assert deterministically (the prior model-tool-dependent contract was only live-observable and empirically unmet after four fixes — T-3.2-RD-D6 through D9). The `write_artifact` tool becomes vestigial/optional rather than the documented mechanism; it is left registered so a future model that reliably calls it still functions, and so the existing tool/store/dispatch plumbing (proven correct by the D9 probe) is not removed.
- **Multi-turn phase dialogue (DCR-2) matches how a phase actually converges.** The single-turn-per-phase shape was the wrong contract for a deliverable that needs conversational back-and-forth: given a terse idea, the requirements model correctly does AC-1.1 clarification and the single-turn driver captured the *questions* as the artifact (live G3 after DCR-1: requirements/design/tasks wrote 3432/1720/2365 bytes of questions, and AC-2.5 correctly rejected 0 tasks — every component behaving correctly, the interaction shape wrong). Multi-turn gives the model room to converge; approve-to-finalize captures the converged deliverable. **Cost:** a larger interaction surface to build + test (multi-turn loop state, in-phase transcript carry); approve-to-finalize overloads the approval gesture — accepted because approval *is* the natural finalize point and a non-approve turn already means "keep going" (AC-2.4).
- **Output-token budget (DCR-2) is a small, orthogonal Converse-wiring fix** ridden along on the same greenfield request path: setting `inferenceConfig.maxTokens` (16K, configurable) prevents the default-4096-cap truncation of a large deliverable that even driver-authored capture would otherwise persist in truncated form.

## Alternatives considered

| Option | Summary | Why not chosen |
|--------|---------|----------------|
| **Lightweight scaffold** | requirements/design/tasks as plain markdown + approval gates, no EARS/NFR/ADR/contract ceremony | User chose full rigor; scaffold loses the traceability + formal-contract value. Kept as a possible future "light mode" |
| **Configurable (default light)** | Flag to opt into full structure | Doubles the greenfield surface to build/test in v1; user chose a single full path |
| **No greenfield in v1** (brownfield only) | Smallest scope | US-1/2/3 are accepted core-value requirements |
| **Single-turn-per-phase + harder prompt** (DCR-2 Option B) | Keep one model turn per phase; lead the prompt harder, forbid clarifying questions, demand a full doc in one shot | Fights AC-1.1 (the model *should* ask clarifying questions on a terse idea); a single turn cannot converge a deliverable that genuinely needs back-and-forth — live G3 (after DCR-1) proved the single turn captures *questions*, not a deliverable. Rejected in favour of multi-turn dialogue (DCR-2 Option A, the Decision above). |
| **First-class greenfield phase events + replay** (DCR-3 Option B) | Add `GREENFIELD_PHASE_APPROVED`/`GREENFIELD_PHASE_ADVANCED` EventType(s), append them at each gate, and reconstruct the phase machine from the session log via a greenfield analogue of `SessionReplay` (AC-7.2-shaped, but for phase-state); unify greenfield resume under the brownfield list-and-replay UX (AC-7.1/7.2/7.4) | More new surface than re-deriving from artifacts: a new `EventType` (touches `event.schema.json` + `CT-SCH-2` + the EventType taxonomy in `03-data-model.md`), a new replay projection, and gate/driver wiring to emit the events. Robust to artifact edits and carries richer in-phase state, but a larger blast radius — and it *still* needs the same AC-7.3 repo-keying. Rejected: the approval-stamped artifacts already on disk (the DCR-1/D13 stamp) are sufficient resume state for the phase boundary, which is the level the user accepted resuming at. Kept as a future option if richer in-phase resume is later wanted. |
| **Relax the traceability gate** (DCR-5 Option b) | Broaden `TaskTraceability`'s `TASK_LINE` to also accept hyphen-less ids (`T1`/`T10`) and broaden `REQUIREMENT_REF` to accept bare `R<n>` refs, so a live model's natural output passes without a prompt change | Rejected by the user. It discards the strict formal traceability guarantee this ADR deliberately chose over the lightweight scaffold: `T1`/`R5` are ambiguous and collide with unrelated tokens, and the gate would no longer pin the US → AC → NFR/ADR → task chain. It also breaks the reflexive-consistency value — a greenfield project would no longer match the methodology that built codingAgent itself. Chosen instead (DCR-5 Option a, the "playbook prompt emits the gate's vocabulary" Decision bullet above): fix the prompt, keep the gate strict. |

## Notes

- Resolves OQ-B: full spec-driven structure.
- The methodology mirrors the skill that built codingAgent — a useful reference and a source of prompt/playbook content for the driver.
- Because it's a workflow driver over the shared engine, greenfield inherits persistence, permission, sub-agents, and memory for free; its specific work is phase orchestration + the per-phase prompts/templates.
- **Amended 2026-06-23 (DCR-1, T-3.2-RD-D10).** Phase-deliverable persistence is now driver-authored in code (`GreenfieldArtifactStore.write()`) rather than dependent on a model-emitted `write_artifact` `toolUse`; later phases inject approved earlier-phase artifacts into their prompt for transcript continuity; AC-1.4 design/-confinement is preserved (source-write tools stay withheld pre-approval). See the Decision section's "Driver-authored phase deliverables" bullet and `02-architecture.md` § 1.2 (C3, C7). The model-tool-dependent persistence (the prior mechanism) was kept as the rejected Option B in the DCR — empirically unproven after four fixes.
- **Amended 2026-06-23 (DCR-2, T-3.2).** DCR-1 landed correctly but the live G3 run after it proved a deeper defect: each pre-approval phase ran as a *single* model turn, so the driver captured the model's AC-1.1 clarifying *questions* rather than a converged deliverable.
